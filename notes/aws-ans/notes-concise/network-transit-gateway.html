AWS Certified Advanced Networking - Specialty (ANS-C01)

Transit Gateway

AWS Transit Gateway - Core Concepts

Architecture Overview

  • Network Hub Architecture:
    • Regional resource that acts as a highly available and scalable router
    • Supports hub-and-spoke and more complex network topologies
    • Enables transitive routing between VPCs and on-premises networks
    • Processes packets at line rate with consistent latency
  • Global Connectivity:
    • Inter-Region peering over AWS backbone with automatic encryption
    • Support for cross-account attachments via AWS RAM
    • Centralized management for complex network architectures

Attachment Types

  • VPC Attachments:
    • Requires one subnet per AZ for high availability
    • Maximum bandwidth: 50 Gbps per VPC attachment (adjustable to 100 Gbps)
    • Appliance mode option for stateful inspection
    • Security group referencing across VPCs
  • Direct Connect Gateway:
    • Global resource connecting Direct Connect to Transit Gateway
    • Supports private VIF association
    • Maximum 20 Transit Gateway associations per Direct Connect Gateway
    • Allowed prefixes control traffic from on-premises to AWS
  • VPN Attachments:
    • Supports IPSec VPN connections
    • Maximum throughput: 1.25 Gbps per tunnel
    • ECMP support for load balancing across tunnels
    • Requires BGP for dynamic routing and ECMP
  • Connect Attachments:
    • GRE tunnel-based connectivity for SD-WAN integration
    • Maximum throughput: 5 Gbps per Connect peer
    • Requires BGP for dynamic routing
    • Maximum 4 Connect peers per attachment
  • Peering Attachments:
    • Connects Transit Gateways across regions
    • Requires static routes (no automatic propagation)
    • Cannot peer Transit Gateways in same region
    • Maximum 50 peering attachments per Transit Gateway

AWS Transit Gateway - Route Tables and Routing

Route Table Structure

  • Route Table Components:
    • Default route table created automatically during gateway setup
    • Custom route tables enable network segmentation (similar to VRF)
    • Each attachment associates with exactly one route table
    • Route propagation can be enabled/disabled per attachment
  • Route Types:
    • Static routes: Manually created, highest priority in routing decisions
    • Propagated routes: Automatically learned from attachments
    • Blackhole routes: Explicitly drop traffic to specified destinations
    • Prefix list references: Group common CIDR blocks for easier management

Route Processing and Propagation

  • Route Evaluation Hierarchy:
    • Static routes take precedence over propagated routes
    • Most specific route match (longest prefix) wins
    • Equal cost routes enable ECMP routing (requires BGP)
    • Blackhole routes override other routes
  • Route Propagation Rules:
    • VPC routes: No automatic propagation, requires explicit enablement
    • VPN with BGP: Routes automatically propagate when enabled
    • Direct Connect: Follows BGP attributes and allowed prefixes
    • Peering connections: Require static routes (no propagation)

Key Limitations

  • Route Table Constraints:
    • Maximum 20 route tables per Transit Gateway (adjustable)
    • Maximum 10,000 routes across all tables per gateway (adjustable)
    • No support for identical or overlapping CIDR ranges across attached VPCs
    • Routes not propagated for newly attached VPCs with overlapping CIDRs
  • Routing Behavior Limitations:
    • No transitive routing between peered Transit Gateways
    • Maximum of 3 levels of peering (TGW to TGW to TGW)
    • Transit Gateway must have at least one VPC attachment before route table creation
    • Point-to-point /30 ranges don't propagate from Direct Connect

AWS Transit Gateway - Deployment Patterns

Centralized Router Pattern

  • Architecture:
    • All attachments associate with and propagate to default route table
    • Functions as a simple layer 3 IP router
    • Full mesh connectivity between all attachments
    • Single routing domain for all networks
  • Use Cases:
    • Shared services architecture
    • Multi-VPC applications
    • Consolidated on-premises connectivity
    • Simplified routing management

Isolated VPCs Pattern

  • Architecture:
    • Multiple route tables with selective associations
    • Each VPC can connect only to on-premises network
    • No VPC-to-VPC communication allowed
    • VPN/Direct Connect attachment has routes to all VPCs
  • Use Cases:
    • Multi-tenant environments
    • Regulatory compliance requirements
    • Security isolation needs
    • Environments with strict network boundaries

Isolated VPCs with Shared Services Pattern

  • Architecture:
    • Multiple route tables with selective routing
    • VPCs remain isolated from each other
    • All VPCs can access shared services VPC
    • On-premises network can access all VPCs
  • Use Cases:
    • Enterprise shared services deployment
    • Centralized security services
    • DNS and Directory Services
    • Compliance with isolation requirements

Centralized Outbound Routing Pattern

  • Architecture:
    • VPCs without internet gateways route through central VPC
    • Central VPC contains NAT gateway for internet access
    • Transit Gateway routes 0.0.0.0/0 to central VPC
    • Centralized control over internet access
  • Key Requirements:
    • TGW attachment must be in private subnet of central VPC
    • NAT Gateway must be in public subnet
    • IGW required only in central VPC
    • Static route (0.0.0.0/0) in TGW points to central VPC

Appliance in Shared Services VPC Pattern

  • Architecture:
    • Security appliance deployed in dedicated VPC
    • All traffic between attachments inspected by appliance
    • Requires specific route table configuration
    • Appliance mode ensures symmetric routing
  • Critical Limitations:
    • Must connect exactly one transit gateway to appliance VPC
    • Multiple transit gateways cannot guarantee flow stickiness
    • Traffic must originate from TGW attachment for correct routing
    • Appliance mode required for stateful inspection

AWS Transit Gateway - Appliance Mode and Traffic Flow

Appliance Mode Purpose

  • Key Functions:
    • Enables centralized inspection of traffic between attachments
    • Ensures symmetric routing through security appliances
    • Maintains flow stickiness to specific network interfaces
    • Preserves stateful inspection capabilities
  • Flow Management:
    • Uses flow hash algorithm to select network interface
    • Maintains same interface for return traffic
    • Ensures bidirectional traffic through same AZ
    • Critical for stateful firewalls and security appliances

Behavior Comparison

  • Without Appliance Mode:
    • Transit gateway keeps traffic in originating AZ
    • Traffic crosses AZs only during failures or missing subnets
    • Can lead to asymmetric routing and dropped connections
    • Example flow:
      • Request: AZ1 VPC → TGW → Appliance1 → TGW → VPC B
      • Response: AZ2 VPC B → TGW → Appliance2 → TGW → VPC A
      • Result: Connection fails - different appliances see different flows
  • With Appliance Mode:
    • Transit gateway maintains flow symmetry
    • All traffic for a flow uses same AZ
    • Both directions processed by same appliance
    • Example flow:
      • Request: AZ1 VPC A → TGW → Appliance1 → TGW → VPC B
      • Response: AZ1 VPC B → TGW → Appliance1 → TGW → VPC A
      • Result: Connection succeeds - same appliance sees both flows

Critical Limitations

  • Single Transit Gateway Requirement:
    • Must use exactly one transit gateway per appliance VPC
    • Multiple transit gateways do not share flow state
    • Cannot guarantee flow stickiness with multiple gateways
  • Traffic Flow Restrictions:
    • Source and destination must use same transit gateway attachment
    • Traffic may drop if using different gateway types (e.g., Internet Gateway)
    • Enabling on existing attachment may affect current routes
    • Traffic must originate from TGW attachment for correct routing

Implementation Considerations

  • High Availability Design:
    • Deploy appliances in multiple AZs
    • Use separate subnets for transit gateway attachments
    • Configure appropriate routing tables per subnet
    • Ensure appliances can handle failover scenarios
  • Route Management:
    • Separate route tables for appliance VPC
    • Different routing needs for transit gateway attachment subnets
    • Specific routes for appliance communication
    • Careful planning of return traffic paths

AWS Transit Gateway - Direct Connect Integration

Architecture Components

  • Required Components:
    • Transit Gateway: Central hub for network traffic
    • Direct Connect Gateway: Global resource that connects Direct Connect to Transit Gateway
    • Transit Virtual Interface (Transit VIF): Connects DX connection to DX Gateway
  • Integration Flow:
    • 1. Create Direct Connect Gateway
    • 2. Create Transit Virtual Interface (Transit VIF)
    • 3. Associate Transit Gateway with Direct Connect Gateway
    • 4. Configure route tables and propagation

Allowed Prefixes

  • Configuration:
    • Provisioned on Direct Connect Gateway
    • Controls traffic from on-premises to AWS
    • Works even without VPC assigned CIDRs
    • Prefixes originate from DX Gateway
  • Behavior:
    • Advertised to on-premises network
    • Controls routing domain boundaries
    • Enables granular route control
    • Affects both Transit and Virtual Private Gateways

Cross-Account Associations

  • Association Process:
    • Transit Gateway owner creates association proposal
    • DX Gateway owner must accept proposal
    • Can include allowed prefix specifications
    • DX Gateway owner can override requested prefixes

Key Limitations

  • Direct Connect Gateway Restrictions:
    • Cannot attach to Transit Gateway if already associated with a Virtual Private Gateway
    • Cannot attach if already attached to a private virtual interface
    • Maximum 20 Transit Gateway associations per Direct Connect Gateway
    • Maximum 6 Transit Gateways per Direct Connect Gateway
  • Routing Limitations:
    • Point-to-point /30 ranges don't propagate to Transit Gateway
    • Must use unique ASNs for multi-region Transit Gateways
    • BGP path attributes determine route selection

Connection Options

  • Dedicated Connection:
    • Direct physical connection to AWS
    • Available in 1Gbps, 10Gbps, and 100Gbps ports
    • Supports up to 50 Transit VIFs per connection
    • Can be combined in LAG for higher bandwidth (up to 400Gbps)
  • Hosted Connection:
    • Through AWS Direct Connect Partners
    • Sub-1Gbps speeds available
    • Limited to one Transit VIF per connection

AWS Transit Gateway - VPN Integration

Core Functionality

  • Purpose:
    • Connects on-premises networks to AWS VPCs
    • Supports both dynamic (BGP) and static routing
    • Enables IPv4 and IPv6 connectivity
    • Centralizes VPN management through Transit Gateway
  • Tunnel Redundancy:
    • Two VPN tunnels per connection
    • Each tunnel terminates in different AZ
    • Automatic failover between tunnels
    • Active/Active configuration possible with BGP

Routing Options

  • Dynamic Routing (BGP):
    • Automatic route propagation
    • Support for route failover
    • ECMP capability for load balancing
    • Dynamic path selection
  • Static Routing:
    • Manual route addition to Transit Gateway route table
    • Static routes must be configured before VPN creation
    • No automatic route filtering
    • Manual failover configuration
  • ECMP Support:
    • Equal Cost Multi-Path routing across tunnels
    • Requires matching BGP attributes
    • Only available for dynamic routing VPNs
    • Improves bandwidth utilization

Important Limitations

  • Technical Constraints:
    • Maximum throughput per tunnel: 1.25 Gbps
    • MTU limitation: 1500 bytes (after encryption)
    • NAT traversal (NAT-T) required for NAT devices
    • Cannot use accelerated VPN with Transit Gateway
  • Routing Limitations:
    • BGP ASN must be unique per VPN connection
    • Static routes don't support automatic failover
    • Route propagation delay during failover
    • Maximum 100 BGP routes per VPN connection

Customer Gateway Requirements

  • Device Configuration:
    • Must meet AWS customer gateway specifications
    • Requires proper IKE and IPsec configuration
    • Needs support for:
      • IKEv1 or IKEv2
      • IPsec with perfect forward secrecy
      • BGP routing (for dynamic VPNs)

AWS Transit Gateway - Connect Attachments

Core Concepts

  • Purpose:
    • Connects Transit Gateway to third-party virtual appliances
    • Enables SD-WAN appliance integration
    • Supports virtual networking appliances
    • Facilitates custom routing solutions
  • Key Components:
    • Transport Attachment: Existing VPC or Direct Connect attachment
    • Connect Peers: GRE tunnels between Transit Gateway and appliance
    • BGP sessions for dynamic routing

Connect Peer Configuration

  • Inside CIDR Blocks (BGP Addresses):
    • IPv4: Must specify /29 CIDR from 169.254.0.0/16
    • First address used for appliance BGP IP
    • Reserved ranges: 169.254.0.0/29 through 169.254.5.0/29
    • IPv6 (optional): /125 CIDR from fd00::/8
  • Peer Addressing:
    • Peer IP Address: GRE outer IP on appliance side
    • Transit Gateway Address: Must be from Transit Gateway CIDR block
    • Must be unique across Connect attachments

BGP Configuration Requirements

  • Supported BGP Types:
    • eBGP: For different autonomous systems
      • Requires ebgp-multihop with TTL=2
    • iBGP: For same autonomous system
      • Requires next-hop-self configuration
    • MP-BGP: Required for IPv6 support
      • IPv4 BGP peering required for IPv6
  • ECMP Requirements:
    • Same prefixes must be advertised
    • Same BGP AS-PATH attribute required
    • Same ASN required
    • No ECMP between redundant BGP peerings

Technical Limitations

  • BGP Configuration:
    • Default keep-alive: 10 seconds
    • Default hold timer: 30 seconds
    • No IPv6 BGP peering support
    • No BFD support
    • No BGP graceful restart
  • Performance Constraints:
    • Maximum throughput: 5 Gbps per Connect peer
    • Maximum packet processing: 300,000 pps per Connect peer
    • Maximum 4 Connect peers per attachment
    • Maximum 5000 routes per Connect attachment

High Availability Design

  • BGP Session Redundancy:
    • Recommended: Two BGP peering sessions per Connect peer
    • Provides routing plane redundancy
    • Protects against AWS infrastructure operations
    • Single session may experience connectivity loss during maintenance

AWS Transit Gateway - Peering Attachments

Core Functionality

  • Peering Capabilities:
    • Inter-Region peering between Transit Gateways
    • Cannot peer Transit Gateways in same region
    • Supports both IPv4 and IPv6 traffic
    • Cross-account peering supported
  • Routing:
    • Requires static routes to peer Transit Gateway
    • No automatic route propagation
    • Route table entries point to peering attachment
    • Recommended to use unique ASNs for each peered gateway

Security and Encryption

  • Inter-Region Traffic:
    • Double encryption:
      • AES-256 encryption at virtual network layer
      • Additional AES-256 encryption at physical layer
    • Uses same infrastructure as VPC peering
  • Intra-Region Traffic:
    • Single encryption at physical layer only
    • Internal AWS network traffic protected by infrastructure

Key Limitations

  • DNS Resolution:
    • No Cross-Region DNS Resolution:
      • Cannot resolve public IPv4 DNS hostnames to private IPv4
      • Cannot resolve private IPv4 DNS hostnames to private IPv4
      • Route 53 Resolver limitation across regions
  • Routing Limitations:
    • No transitive routing through peering connections
    • Maximum 50 peering attachments per transit gateway
    • Maximum of one peering attachment between two transit gateways
    • Maximum of 3 levels of peering (TGW to TGW to TGW)
  • Security Group Limitations:
    • Security group referencing not supported across peering connections

Implementation Process

  • Setup Steps:
    • Create peering attachment on requester Transit Gateway
    • Specify peer Transit Gateway ID
    • Peer Transit Gateway owner accepts request
    • Configure static routes in both Transit Gateways
  • Route Configuration:
    • Add static routes pointing to peering attachment
    • Configure return routes on peer Transit Gateway
    • Ensure unique ASNs for future compatibility
    • Verify route propagation

AWS Transit Gateway - DNS Integration

DNS Resolution Challenges

  • Complex DNS Requirements:
    • Multiple VPCs need to resolve each other's private DNS names
    • On-premises networks need to resolve AWS private DNS names
    • AWS resources need to resolve on-premises DNS names
    • Each VPC has its own private DNS namespace

Route 53 Resolver Components

  • Resolver Endpoints:
    • Inbound Endpoints:
      • Allows on-premises DNS servers to resolve AWS private DNS names
      • Provides IP addresses for on-premises DNS to forward queries to
      • Must be configured in VPCs attached to Transit Gateway
    • Outbound Endpoints:
      • Enables AWS resources to resolve on-premises DNS names
      • Forwards DNS queries from VPCs to on-premises DNS servers
      • Configured with rules to determine which domains to forward
  • Resolver Rules:
    • Conditional Forwarding Rules:
      • Define which DNS queries should be forwarded to on-premises
      • Can be shared across accounts using AWS RAM
    • System Rules:
      • Handle DNS resolution for VPC domains automatically
      • Manage resolution between VPCs connected via Transit Gateway
      • Cannot be modified but can be overridden

Private Hosted Zones Integration

  • Cross-VPC DNS Resolution:
    • Enable DNS resolution across VPCs connected by Transit Gateway
    • Requires enablement of DNS support on Transit Gateway
    • Implementation Steps:
      • 1. Enable DNS support on Transit Gateway
      • 2. Associate Private Hosted Zones with required VPCs
      • 3. Configure route tables to allow DNS traffic (port 53)
      • 4. Verify DNS resolution across VPCs

Centralized DNS Architecture

  • Common Patterns:
    • Central DNS VPC:
      • Dedicated VPC for DNS services
      • Houses Resolver endpoints
      • Manages DNS traffic for all connected VPCs
      • Simplifies DNS management in large environments
    • Hybrid DNS Setup:
      • Deploy Resolver endpoints in DNS VPC
      • Configure Transit Gateway route tables for DNS traffic
      • Set up conditional forwarding rules
      • Configure on-premises DNS servers to forward to Resolver inbound endpoints

DNS Security Integration

  • Route 53 DNS Firewall:
    • Filter and control DNS queries in Transit Gateway environment
    • Prevent DNS data exfiltration
    • Block malicious domain resolution
    • Implementation with Transit Gateway:
      • Deploy DNS Firewall rules in central DNS VPC
      • Associate rules with Resolver endpoints
      • Monitor DNS query logs for security analysis
      • Share rules across accounts using RAM

Key Limitations

  • DNS Resolution Constraints:
    • No direct DNS resolution for custom DNS names
    • Private hosted zones require special configuration
    • No inter-Region DNS resolution across peering connections
    • Requires centralized DNS management solution for complex environments

AWS Transit Gateway - Design Best Practices

Subnet Design

  • Transit Gateway Attachment Subnets:
    • Use separate subnet for each TGW attachment
    • Implement small CIDR blocks (e.g., /28)
    • Preserves IP addresses for EC2 resources
    • Place in dedicated subnets separate from workloads
  • Network ACL Configuration:
    • Keep TGW subnet ACLs open (inbound/outbound)
    • Apply restrictive ACLs to workload subnets instead
    • Use single ACL for all TGW-associated subnets
    • Simplifies troubleshooting and management

Route Table Management

  • Best Practices:
    • Use same route table for all TGW-associated subnets
    • Exception: Middle-box VPCs requiring multiple NAT gateways
    • Limit number of transit gateway route tables
    • Enable route propagation for:
      • Direct Connect gateway attachments
      • BGP Site-to-Site VPN attachments
  • Route Table Strategy:
    • Centralized Router: Single route table for full connectivity
    • Isolated VPCs: Separate route tables for network segmentation
    • Shared Services: Selective route propagation for controlled access
    • Consider future growth when designing route tables

Connectivity Design

  • VPN Connections:
    • Prefer BGP Site-to-Site VPN connections
    • Enable multipath if customer gateway supports it
    • Deploy VPN endpoints in multiple locations for redundancy
    • Consider Direct Connect with VPN backup for critical workloads
  • Migration Considerations:
    • VPC peering to TGW migration:
      • Be aware of MTU size mismatches
      • Update both VPCs simultaneously
      • Prevent jumbo packet drops
      • Test thoroughly before production migration

High Availability and Redundancy

  • Key Design Points:
    • TGW inherently highly available
    • Additional TGWs not needed for HA
    • Use single TGW per Region for disaster recovery
    • Deploy resources across multiple AZs
  • Multi-Gateway Deployments:
    • Use unique ASN for each transit gateway
    • Consider inter-Region peering
    • Enables global network architecture
    • Plan for cross-region failover scenarios

Security Design

  • Security Architecture:
    • Implement inspection VPC for centralized security
    • Use appliance mode for stateful inspection
    • Consider network segmentation through route tables
    • Implement least privilege access model
  • DNS Security:
    • Centralize DNS management in dedicated VPC
    • Implement Route 53 DNS Firewall
    • Monitor DNS query logs
    • Control cross-VPC DNS resolution

Monitoring Strategy

  • Best Practices:
    • Enable Transit Gateway Flow Logs
    • Create CloudWatch dashboards for key metrics
    • Set up alerts for critical thresholds
    • Implement regular route table audits
    • Use Network Manager for topology visualization

AWS Transit Gateway - Quotas and Limitations

Resource Quotas

  • Transit Gateway Resources:
    • Transit gateways per account: 5 (adjustable)
    • CIDR blocks per transit gateway: 5 (not adjustable)
    • Transit gateways per VPC: 5 (not adjustable)
    • Route tables per transit gateway: 20 (adjustable)
  • Attachment Limitations:
    • Attachments per transit gateway: 5,000 (not adjustable)
    • Maximum of one VPC attachment per VPC
    • Peering attachments per transit gateway: 50 (adjustable)
    • Pending peering attachments: 10 (adjustable)
    • Connect peers per Connect attachment: 4 (not adjustable)

Routing Limitations

  • Route Table Constraints:
    • Total routes across all tables per gateway: 10,000 (adjustable)
    • Dynamic routes from virtual router to Connect peer: 1,000 (adjustable)
    • Routes from Connect peer to virtual router: 5,000 (not adjustable)
    • Static routes per prefix to single attachment: 1 (not adjustable)
    • BGP routes per VPN connection: 100 (not adjustable)
  • Routing Behavior Limitations:
    • No support for identical CIDR ranges across attached VPCs
    • No support for overlapping CIDR ranges
    • Routes not propagated for newly attached VPCs with overlapping CIDRs
    • No transitive routing between peered Transit Gateways
    • Maximum of 3 levels of peering (TGW to TGW to TGW)

Bandwidth and Performance Limitations

  • Bandwidth Caps:
    • VPC attachment per AZ: Up to 100 Gbps (adjustable with AWS assistance)
    • Direct Connect/peering connection per AZ: Up to 100 Gbps (adjustable with AWS assistance)
    • VPN tunnel: Up to 1.25 Gbps (not adjustable)
    • Connect peer (GRE tunnel): Up to 5 Gbps (not adjustable)
  • Packet Processing Limits:
    • VPC attachment per AZ: Up to 7,500,000 pps (adjustable with AWS assistance)
    • Direct Connect/peering attachment per AZ: Up to 7,500,000 pps (adjustable with AWS assistance)
    • VPN tunnel: Up to 140,000 pps (not adjustable)
    • Connect peer: Up to 300,000 pps (not adjustable)

Direct Connect Integration Limitations

  • Gateway Constraints:
    • Direct Connect gateways per transit gateway: 20 (not adjustable)
    • Transit gateways per Direct Connect gateway: 6 (not adjustable)
    • Point-to-point /30 ranges don't propagate to Transit Gateway

MTU Considerations

  • Maximum Transmission Unit Limits:
    • VPC, Direct Connect, Connect, and peering attachments: 8500 bytes
    • VPN connections: 1500 bytes
  • MTU Behavior:
    • Transit gateway enforces MSS clamping for all packets
    • Supports Path MTU Discovery (PMTUD) for VPC and Connect attachments
    • No PMTUD support for Site-to-Site VPN, Direct Connect, and Peering attachments
    • Migration from VPC peering may cause MTU mismatch issues

Critical Architectural Limitations

  • Connectivity Constraints:
    • Resources in AZs without TGW attachment cannot reach TGW
    • Local Zone subnets cannot connect directly to transit gateway
    • IPv6-only subnets not supported for attachments
    • Maximum of one peering attachment between two transit gateways
    • Cannot peer Transit Gateways in same region
  • Feature Limitations:
    • No DNS resolution for custom DNS names in private hosted zones
    • No inter-Region DNS resolution across peering connections
    • ECMP not supported for static VPN routing
    • No ECMP between BGP peerings of same Connect peer
    • Security group referencing not supported across peering connections

AWS Transit Gateway - Network Services Integration

Network Firewall Integration

  • Architecture Overview:
    • Dedicated inspection VPC for AWS Network Firewall deployment
    • All traffic routes through firewall endpoints before reaching destination
    • Centralized security inspection for multiple VPCs
    • Requires specific route table configuration in Transit Gateway
  • Implementation Steps:
    • Create inspection VPC with multiple AZs
    • Deploy Network Firewall endpoints in each AZ
    • Configure Transit Gateway route tables to direct traffic to inspection VPC
    • Set up return traffic routes from firewall back through Transit Gateway

AWS Transit Gateway - Advanced Networking Concepts

BGP Routing with Transit Gateway

  • BGP Fundamentals:
    • Autonomous System Numbers (ASNs):
      • Transit Gateway uses ASN for BGP sessions
      • 16-bit ASN range: 64512 to 65534
      • 32-bit ASN range: 4200000000 to 4294967294
      • Must be unique per Transit Gateway for proper routing
    • BGP Path Selection:
      • Highest local preference
      • Shortest AS path
      • Lowest MED value
      • eBGP over iBGP
      • Lowest IGP metric to BGP next hop
  • BGP Attributes and Traffic Engineering:
    • AS_PATH Manipulation:
      • Path prepending to influence route selection
      • Longer paths less preferred
      • Used for active/standby configurations
    • MED (Multi-Exit Discriminator):
      • Influences inbound traffic from AWS to on-premises
      • Lower values preferred
      • Used when multiple connection points exist
    • Communities:
      • Tag routes with specific attributes
      • Filter routes based on community values
      • Control route propagation

Equal Cost Multi-Path (ECMP) Routing

  • ECMP Implementation:
    • Load balancing across multiple paths
    • Requires BGP routing (not available with static routes)
    • Requirements for ECMP:
      • Same destination CIDR blocks
      • Same BGP AS_PATH attribute
      • Same route preference/weight
    • Flow-based hashing algorithm for path selection
  • ECMP Use Cases:
    • VPN Tunnel Redundancy:
      • Multiple VPN tunnels for higher throughput
      • Aggregate bandwidth across tunnels
      • Automatic failover if a tunnel fails
    • Direct Connect Redundancy:
      • Multiple Direct Connect connections
      • Load balancing across connections
      • Improved resilience against failures

MTU and Packet Handling

  • MTU Considerations:
    • Different MTU sizes across attachment types:
      • VPC, Direct Connect, Connect, Peering: 8500 bytes (jumbo frames)
      • VPN connections: 1500 bytes
    • Path MTU Discovery (PMTUD):
      • Supported for VPC and Connect attachments
      • Not supported for VPN, Direct Connect, and Peering
      • Critical for preventing fragmentation issues
  • Packet Handling:
    • MSS Clamping:
      • Transit Gateway enforces MSS clamping for all packets
      • Prevents fragmentation issues
      • Adjusts TCP segment size based on path MTU
    • Fragmentation Behavior:
      • Packets exceeding MTU may be fragmented
      • DF (Don't Fragment) bit honored
      • ICMP "fragmentation needed" messages generated

Advanced Routing Scenarios

  • Route Aggregation:
    • Summarizing routes for efficient routing tables
    • Reduces route table size and processing overhead
    • Important for large-scale deployments
    • Example: Advertising 10.0.0.0/16 instead of multiple /24 subnets
  • Asymmetric Routing:
    • Traffic takes different paths in each direction
    • Can cause issues with stateful firewalls
    • Appliance mode prevents asymmetric routing
    • BGP path manipulation can control routing symmetry
  • Blackhole Routing:
    • Explicitly drop traffic to specific destinations
    • Used for security isolation
    • Overrides other routing entries
    • Useful for blocking unwanted traffic flows

Advanced Security Concepts

  • Traffic Inspection Architectures:
    • Inline Inspection:
      • All traffic passes through security appliances
      • Requires appliance mode for stateful inspection
      • Can introduce latency but provides comprehensive security
    • Out-of-band Inspection:
      • Traffic mirroring to security appliances
      • No impact on production traffic
      • Detection without prevention capabilities
  • Defense in Depth:
    • Multiple security layers:
      • Network segmentation with route tables
      • Security groups for instance-level control
      • Network ACLs for subnet protection
      • Centralized inspection with Network Firewall
      • DNS filtering with Route 53 Resolver DNS Firewall

AWS Transit Gateway - Practice Questions

Question 1

A company has implemented a Transit Gateway with multiple VPCs and a Site-to-Site VPN connection to their on-premises data center. They need to ensure that traffic between two specific VPCs passes through a security appliance for inspection. Which configuration is required?

  • A. Deploy the security appliance in a separate VPC and configure Transit Gateway route tables to direct traffic through this VPC.
  • B. Enable VPC peering between the two VPCs and deploy the security appliance in the peering connection.
  • C. Deploy the security appliance in a separate VPC, enable appliance mode on the Transit Gateway attachment, and configure appropriate route tables.
  • D. Configure Network ACLs in both VPCs to redirect traffic to a centralized security VPC.

Answer: C

Explanation: To ensure traffic inspection between VPCs, you need to deploy the security appliance in a separate VPC and enable appliance mode on the Transit Gateway attachment. Appliance mode ensures symmetric routing through the security appliance, which is critical for stateful inspection. You also need to configure Transit Gateway route tables to direct traffic through the security VPC. Option B is incorrect because VPC peering doesn't support inserting a security appliance in the path. Option A is incomplete because without appliance mode, you may encounter asymmetric routing issues. Option D is incorrect because Network ACLs cannot redirect traffic.

Question 2

A network architect is designing a multi-account AWS environment with Transit Gateway. The design requires that development VPCs can communicate with shared services but not with each other, while production VPCs need full connectivity to each other and to shared services. Which Transit Gateway configuration meets these requirements?

  • A. Create a single route table with blackhole routes between development VPCs.
