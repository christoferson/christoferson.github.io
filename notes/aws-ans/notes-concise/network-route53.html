AWS Certified Advanced Networking - Specialty (ANS-C01)

Route 53

AWS Route 53 - Core Concepts

Service Overview

  • Definition and Architecture:
    • Highly available and scalable Domain Name System (DNS) web service
    • Control plane in us-east-1 (with failover to us-west-2)
    • Data plane distributed globally across 200+ Points of Presence
    • Performs three main functions: domain registration, DNS routing, health checking
  • DNS Resolution Process:
    • User enters domain name in browser
    • Request goes to DNS resolver (typically ISP)
    • Resolver queries root name servers
    • Root servers direct to TLD servers
    • TLD servers provide Route 53 name servers
    • Route 53 provides IP address
    • Browser connects to web server

Record Types

  • Standard DNS Records:
    • A Record: Maps domain to IPv4 address
    • AAAA Record: Maps domain to IPv6 address
    • CNAME: Maps domain to another domain (cannot be used at zone apex)
    • MX: Defines mail servers
    • TXT: Text records for various purposes (verification, SPF, etc.)
  • Route 53 Specific:
    • Alias Records:
      • AWS-specific feature
      • Maps to AWS resources (S3, CloudFront, ALB, etc.)
      • No additional charge for Alias queries
      • Automatically updates if target IP changes
      • Can be used at zone apex (example.com)

Routing Policies

  • Simple Routing:
    • Basic routing to single resource
    • Multiple values return random selection
    • Cannot be associated with health checks
  • Weighted Routing:
    • Route traffic based on assigned weights
    • Percentage = (Weight for record / Sum of all weights)
    • Use cases: A/B testing, gradual migrations
  • Latency-based Routing:
    • Routes based on lowest network latency for end user
    • Uses AWS Region latency measurements
    • Requires creating records in multiple regions
  • Failover Routing:
    • Active-passive failover configuration
    • Requires health check on primary resource
    • Automatically routes to secondary if primary fails
  • Geolocation Routing:
    • Routes based on user location
    • Can specify continents, countries, or US states
    • Requires default record for non-matched locations
  • Geoproximity Routing:
    • Routes based on geographic location of resources
    • Can shift traffic with bias values (-99 to +99)
    • Requires Route 53 Traffic Flow
  • IP-based Routing:
    • Routes based on client IP addresses
    • Uses CIDR blocks to define IP ranges
    • Requires CIDR location mapping
  • Multivalue Answer:
    • Returns multiple healthy records randomly
    • Can be associated with health checks
    • Returns up to 8 healthy records

AWS Route 53 - Health Checks and DNS Failover

Health Check Fundamentals

  • Types of Health Checks:
    • Endpoint Monitoring:
      • Monitor specific IP address or domain
      • Supports HTTP, HTTPS, TCP protocols
      • Can monitor external endpoints
    • Status of Other Health Checks:
      • Calculated health checks
      • Monitor multiple endpoints
      • Useful for complex failover scenarios
    • State of CloudWatch Alarm:
      • Monitor CloudWatch metrics
      • Broader monitoring capabilities
      • Custom metric integration
  • Key Parameters:
    • Request Interval:
      • Standard: 30 seconds
      • Fast: 10 seconds (higher cost)
    • Failure Threshold:
      • Number of consecutive failures before marking unhealthy
      • Default: 3 consecutive failures
    • Global Perspective:
      • Health checks from multiple regions
      • Minimum 3 regions must report healthy
      • Global status determination

DNS Failover Configuration

  • Active-Passive Failover:
    • Primary and backup resources
    • Automatic failover when primary is unhealthy
    • Uses failover routing policy
    • Health check required on primary resource
  • Active-Active Failover:
    • All resources active simultaneously
    • Uses multiple routing policies (weighted, latency, etc.)
    • Resources can be weighted
    • Health checks on all resources
  • Private Endpoint Monitoring:
    • Requires CloudWatch metric
    • Use CloudWatch alarm health check
    • Useful for internal resources
    • Monitors resources not accessible from internet

Integration with Other Services

  • CloudWatch Integration:
    • Health check metrics published to CloudWatch
    • Custom alarms possible
    • Automated actions via CloudWatch
    • Metrics include health check status, time to complete check
  • SNS Integration:
    • Notifications for status changes
    • Multiple notification endpoints
    • Custom message formats
    • Automated remediation workflows
  • Routing Policy Integration:
    • Failover routing requires health checks
    • Multivalue answer supports health checks
    • Weighted, latency, geolocation policies can use health checks
    • Simple routing cannot use health checks

Health Check Process

  • Operation Flow:
    • Health checkers send requests at configured interval
    • Response evaluated against configured criteria
    • Status updated based on failure threshold
    • DNS responses updated based on health status
    • Health checkers distributed globally
  • String Matching:
    • Search for specific string in response
    • First 5120 bytes checked
    • Case-sensitive matching
    • Useful for validating response content

AWS Route 53 - Resolver and DNS Firewall

Route 53 Resolver Overview

  • Core Functionality:
    • Recursive DNS service for VPCs
    • Handles hybrid DNS resolution
    • Supports both inbound and outbound DNS queries
    • Regional service - runs independently in each AWS Region
  • Components:
    • Resolver Endpoints:
      • Inbound: Allows on-premises to resolve AWS DNS names
      • Outbound: Allows VPCs to resolve on-premises DNS names
      • Each endpoint requires 2+ IP addresses in different AZs
      • Highly available by design
    • Resolver Rules:
      • Forward rules: Direct queries to custom DNS servers
      • System rules: Default AWS resolution behavior
      • Recursive lookups: Handle standard DNS resolution
      • Can be shared across accounts using AWS RAM

Route 53 Resolver DNS Firewall

  • Key Features:
    • Domain List Management:
      • Allow lists: Permitted domains
      • Block lists: Denied domains
      • Custom domain lists
      • AWS managed domain lists
    • Rule Groups:
      • Collections of firewall rules
      • Priority-based processing
      • Association with VPCs
      • Shareable across accounts
  • Response Actions:
    • ALLOW: Permit the DNS query
    • BLOCK: Deny the DNS query
    • ALERT: Allow but log the query
    • Custom response: Return specific DNS response

Route 53 Resolver Query Logging

  • Logging Features:
    • Log DNS queries from VPCs
    • Integration with CloudWatch Logs
    • Supports multiple destinations (S3, CloudWatch Logs, Kinesis)
    • Includes query metadata
  • Log Contents:
    • Query timestamp
    • Domain name requested
    • Query type
    • Response code
    • Source IP address
    • Transport protocol (UDP/TCP)

Hybrid DNS Architecture

  • On-premises to AWS Resolution:
    • Inbound endpoints receive queries from on-premises
    • Resolves AWS-specific domains (.amazonaws.com)
    • Resolves private hosted zone domains
    • Requires VPC security group configuration
  • AWS to On-premises Resolution:
    • Outbound endpoints forward queries to on-premises DNS
    • Conditional forwarding based on domain names
    • Forward rules define which domains go where
    • Supports multiple target DNS servers
  • Resolver on AWS Outposts:
    • Local DNS resolution on Outposts racks
    • Integration with on-premises DNS
    • Reduced latency for local queries
    • Extends hybrid DNS architecture

AWS Route 53 - Traffic Flow and Advanced Routing

Traffic Flow Overview

  • Purpose:
    • Visual traffic management tool
    • Complex routing configuration management
    • Global traffic optimization
    • Policy version control
  • Key Components:
    • Traffic Policy:
      • Collection of routing rules
      • Visual editor for configuration
      • Support for multiple versions
      • Reusable across multiple domains
    • Policy Records:
      • DNS records created from traffic policies
      • Associates policy with domain/subdomain
      • Supports multiple endpoints
      • Enables complex routing logic

Geoproximity Routing

  • Key Features:
    • Routes based on geographic distance between users and resources
    • Requires Traffic Flow (not available directly in console)
    • Resources can be AWS regions or custom coordinates
    • Bias adjustments (-99 to +99) modify effective distance
  • Bias Configuration:
    • Positive bias: Expands resource region (attracts more traffic)
    • Negative bias: Shrinks resource region (reduces traffic)
    • Zero bias: Geographic distance only
    • Allows fine-tuning of regional traffic distribution

Multi-Region Application Architecture

  • Global Applications:
    • Deploy resources across multiple AWS regions
    • Use Route 53 for intelligent traffic routing
    • Health checks for automatic failover
    • Optimize for latency, availability, and cost
  • Routing Strategy Combinations:
    • Latency + Health Checks: Route to closest healthy endpoint
    • Geolocation + Failover: Regional routing with backup
    • Weighted + Latency: Controlled distribution with performance
    • Geoproximity + Health Checks: Distance-based with availability

Private DNS for VPCs

  • Private Hosted Zones:
    • DNS resolution within VPCs only
    • Not accessible from public internet
    • Can associate with multiple VPCs
    • Requires enableDnsHostnames and enableDnsSupport VPC settings
  • Split-View DNS:
    • Different responses for same domain based on query source
    • Public hosted zone for internet users
    • Private hosted zone for VPC resources
    • Enables internal/external DNS separation

Route 53 Profiles

  • Core Features:
    • Centralized DNS Management:
      • Multi-VPC configurations
      • Cross-account management
      • Standardized DNS settings
      • Consistent DNS configuration
    • Profile Components:
      • DNS settings templates
      • VPC associations
      • Rule configurations
      • Version control

AWS Route 53 - DNSSEC Implementation

DNSSEC Fundamentals

  • Security Problem Addressed:
    • DNS spoofing (man-in-the-middle attacks)
    • Traffic hijacking to fake websites
    • Interception of DNS queries
    • Unauthorized response substitution
  • DNSSEC Solution:
    • Protocol for securing DNS traffic
    • Creates chain of trust for DNS responses
    • Uses cryptographic signatures to verify authenticity
    • Prevents acceptance of spoofed DNS records

Route 53 DNSSEC Support

  • Dual Functionality:
    • DNSSEC signing (for DNS service)
    • DNSSEC for domain registration
    • Can be used together or separately
  • Implementation Options:
    • Route 53 as both registrar and DNS service
    • Route 53 as registrar with external DNS service
    • External registrar with Route 53 DNS service

DNSSEC Implementation Sequence

  • Critical Order:
    • Enabling DNSSEC:
      1. Configure DNSSEC at DNS provider FIRST
      2. THEN add public keys to domain registration
      3. Never reverse this order
    • Disabling DNSSEC:
      1. Delete public keys from domain registration FIRST
      2. Wait 3 days
      3. THEN disable DNSSEC at DNS provider
      4. Reversing causes domain outage

Key Management

  • Key Types:
    • KSK (Key-Signing Key)
    • ZSK (Zone-Signing Key)
  • Key Limits:
    • .com and .net domains: Maximum 13 public keys
    • All other TLDs: Maximum 4 public keys
  • Key Rotation:
    • Add new public keys
    • Wait 3 days
    • Delete old public keys
  • Format Requirements:
    • Base64 format only
    • Public key, not digest
    • Must match algorithm used at DNS provider

Critical Failure Scenario

  • Domain Outage Cause:
    • Disabling DNSSEC at DNS provider while keys remain at registry
    • Results in SERVFAIL errors
    • Makes domain completely inaccessible
    • Requires proper decommissioning sequence

Prerequisites and Limitations

  • Requirements:
    • TLD registry must support DNSSEC
    • DNS service provider must support DNSSEC
    • Not all TLDs support DNSSEC
    • One key operation at a time
    • Must wait for confirmation email between operations

AWS Route 53 - Domain Registration and Management

Domain Registration Fundamentals

  • Registration Components:
    • Domain Name: Human-readable website address (e.g., example.com)
    • Domain Registrar: ICANN-accredited company processing registrations
    • Domain Registry: Company owning rights to specific TLDs
    • Top-level Domain (TLD): Last part of domain (.com, .org, etc.)
  • Route 53 Registrar Options:
    • Amazon Registrar (owned by AWS)
    • Gandi (registrar associate)
    • Automatic selection based on TLD
    • Gradual migration from Gandi to Amazon Registrar

Registration Process

  • Required Information:
    • Domain name and TLD selection
    • Registration period (1-10 years)
    • Contact information:
      • Registrant (domain owner)
      • Administrative contact
      • Technical contact
    • Privacy protection preferences
  • Verification Requirements:
    • Email verification for new registrant contacts
    • Verification emails from:
      • noreply@registrar.amazon (Amazon Registrar)
      • noreply@domainnameverification.net (Gandi)
    • Domain suspension if not verified
    • 3-15 day response window

Domain Security Features

  • Transfer Lock:
    • Prevents unauthorized domain transfers
    • Shows as 'clientTransferProhibited' in WHOIS
    • Must be disabled before domain transfer
    • Recommended to keep enabled during normal operations
  • Privacy Protection:
    • Must be same setting for ALL contacts
    • Enabled by default on new registrations
    • Some TLDs (.uk, .co.uk) require support case for changes
    • Different behavior between Amazon Registrar and Gandi

Domain Lifecycle Management

  • Renewal Timeline:
    • Standard domains: Renewed 35 days before expiration
    • Special TLDs (.com.ar, .com.br, .jp): 30 days before expiration
    • 45-day advance notification email
    • AWS credits cannot be used for renewals
  • Restoration Process:
    • Requires AWS Support case
    • Significantly higher fees than standard renewal
    • Processing during Paris business hours (UTC+1)
    • Expired domains: New date is 1-2 years after original expiration

Domain Transfers

  • External Transfer Prerequisites:
    • 60-day minimum registration before transfer
    • Transfer lock must be disabled (clientTransferProhibited)
    • Authorization code required
    • Privacy protection should be disabled
  • AWS Account Transfers:
    • Requires domain:TransferDomains permission for initiator
    • Requires domain:AcceptDomainTransferFromAnotherAwsAccount for recipient
    • Free and typically completes within minutes
    • Hosted zone must be transferred separately

AWS Route 53 - Hosted Zones and DNS Configuration

Hosted Zone Fundamentals

  • Definition and Types:
    • Container for DNS records with same name as domain
    • Public hosted zone: Routes internet traffic
    • Private hosted zone: Routes VPC traffic
    • Each zone gets four unique name servers
  • Critical Components:
    • NS and SOA records created automatically
    • Never modify or delete these automatic records
    • Maximum one active hosted zone per domain (though multiple can exist)
    • DNS metrics available only in us-east-1 region

NS and SOA Records

  • Name Server (NS) Records:
    • Four name servers assigned to each hosted zone
    • Format: ns-XXXX.awsdns-XX.TLD (e.g., ns-2048.awsdns-64.com)
    • All four must be configured at registrar for DNS to work
    • Default TTL: 172800 seconds (2 days)
  • Start of Authority (SOA) Records:
    • Format: [nameserver] [admin-email] [serial] [refresh] [retry] [expiry] [minimum-ttl]
    • Minimum TTL controls negative caching (NXDOMAIN/NODATA responses)
    • Negative caching = lesser of SOA minimum TTL and SOA record TTL
    • Default minimum TTL: 86400 seconds (1 day)

Hosted Zone Management

  • Creation Methods:
    • Automatic creation when registering domain with Route 53
    • Manual creation when transferring DNS service to Route 53
    • Service-created zones (AWS Cloud Map, ECS Service Discovery)
    • Programmatic creation via API/CLI
  • Deletion Requirements:
    • Can only delete zones with no records except NS and SOA
    • Must disable DNSSEC signing before deletion
    • Service-created zones require special deletion process
    • For subdomain delegation: delete parent NS record first, wait for TTL, then delete child zone

Subdomain Delegation

  • Standard Approach:
    • Create records in parent domain's hosted zone
    • Use subdomain name in record name field
    • Simplest configuration
    • All records managed in one place
  • Separate Hosted Zone Approach:
    • Create new hosted zone for subdomain
    • Create NS record in parent domain pointing to subdomain's name servers
    • Called "delegating responsibility" or "subdomain delegation"
    • Enables administrative separation
  • Delegation Chain:
    • DNS resolver queries parent domain for subdomain
    • Parent domain returns subdomain's name servers
    • Resolver queries subdomain's name servers
    • Subdomain name servers return requested record

Advanced Configurations

  • White-Label Name Servers:
    • Custom domain names for Route 53 name servers
    • Requires reusable delegation set (API/CLI only)
    • Cannot change name servers for existing zones - must recreate zones
    • Requires both A and AAAA records for each name server
    • Requires glue records at registrar
  • Reusable Delegation Sets:
    • Same four name servers across multiple hosted zones
    • Created programmatically only (API, CLI, SDK)
    • Must be specified when creating hosted zone
    • Cannot be changed for existing zones
    • Simplifies DNS migration

AWS Route 53 - DNS Migration and Implementation

DNS Migration Strategies

  • Active Domain Migration:
    • For domains with existing traffic
    • Step-by-step process:
      1. Get current DNS configuration
      2. Create Route 53 hosted zone
      3. Create DNS records
      4. Lower TTL settings
      5. Remove DNSSEC if configured
      6. Wait for TTL expiration
      7. Update name servers
      8. Monitor traffic
      9. Restore TTL values
  • Inactive Domain Migration:
    • For domains with minimal traffic
    • Simplified process:
      1. Get current DNS configuration
      2. Create Route 53 hosted zone
      3. Create DNS records
      4. Update name servers

TTL Optimization

  • TTL Considerations:
    • Lower TTLs: Faster propagation, higher query volume
    • Higher TTLs: Better caching, lower query volume
    • NS record TTL: Typically 172800 seconds (2 days)
    • Strategic TTL adjustment during migrations
  • Migration TTL Strategy:
    • Reduce TTL at current DNS provider
    • Also reduce TTL in Route 53 hosted zone
    • Recommended value: 60-900 seconds during migration
    • Wait full original TTL period before name server change
    • Restore to normal TTL after successful migration

DNS Resolution Flow

  • Standard Resolution Process:
    • DNS resolver queries root servers
    • Root servers direct to TLD servers
    • TLD servers direct to authoritative name servers
    • Authoritative name servers provide record data
    • Resolvers cache responses according to TTL
  • Subdomain Resolution:
    • Resolver queries parent domain for subdomain
    • Parent domain returns subdomain's name servers
    • Resolver queries subdomain's name servers
    • Subdomain name servers return requested record
    • Small performance impact on first query

Hybrid DNS Architecture

  • Route 53 Resolver Configuration:
    • Inbound endpoints: On-premises → AWS resolution
    • Outbound endpoints: AWS → On-premises resolution
    • Minimum 2 IP addresses in different AZs per endpoint
    • Forward rules define conditional forwarding
  • Conditional Forwarding:
    • Domain-specific forwarding rules
    • Forward specific domains to on-premises DNS
    • Keep AWS resolution for other domains
    • Shareable across accounts with AWS RAM

Implementation Best Practices

  • DNS Testing and Verification:
    • Use Route 53 DNS response checking tool
    • Test from multiple locations
    • Verify all critical record types
    • Monitor resolution during migration
  • Security Considerations:
    • Implement DNS Firewall for VPC protection
    • Consider DNSSEC for critical domains
    • Follow proper DNSSEC implementation sequence
    • Maintain transfer lock when not actively transferring
  • Cost Optimization:
    • Use Alias records for AWS resources (free queries)
    • Optimize TTL values for balance of performance and cost
    • Consider query logging costs
    • Monitor query volume with CloudWatch metrics

AWS Route 53 - Critical Exam Concepts

Key Architectural Concepts

  • Service Structure:
    • Control Plane: us-east-1 (failover to us-west-2)
    • Data Plane: 200+ global Points of Presence
    • Regional Service: Resolver components
    • Highly available by design
  • DNS Resolution Process:
    • Recursive resolver queries root servers
    • Root servers direct to TLD servers
    • TLD servers direct to authoritative name servers
    • Authoritative name servers provide record data
    • Resolvers cache responses according to TTL

Critical Numerical Values

  • Health Check Parameters:
    • Health Check Intervals: 30s standard, 10s fast
    • Health Check Threshold: Minimum 3 regions must report healthy
    • String Matching: First 5120 bytes checked
  • DNS Configuration Limits:
    • Resolver Endpoints: Minimum 2 IP addresses in different AZs
    • Geoproximity Bias: -99 to +99
    • Multivalue Answer: Returns up to 8 healthy records
    • Name Servers: 4 per hosted zone, maximum 6 with white-label configuration
  • TTL Values:
    • Default NS TTL: 172800 seconds (2 days)
    • Default SOA minimum TTL: 86400 seconds (1 day)
    • Recommended migration TTL: 60-900 seconds
    • DNS propagation: Up to 48 hours for name server changes
  • DNSSEC Limits:
    • .com and .net domains: Maximum 13 public keys
    • All other TLDs: Maximum 4 public keys
    • DNSSEC waiting period: 3 days between key operations

Domain Management Critical Points

  • Domain Registration:
    • 60-day minimum registration before transfer
    • Email verification required (3-15 day window)
    • Verification emails from specific sources (noreply@registrar.amazon, etc.)
    • Domain suspended if verification fails
  • Domain Security:
    • Transfer lock (clientTransferProhibited) prevents unauthorized transfers
    • Privacy protection must be same for ALL contacts
    • Ownership changes require specific verification
    • First/Last name changes for Person type = ownership change
  • Domain Lifecycle:
    • Standard domains: Renewed 35 days before expiration
    • Special TLDs: Renewed 30 days before expiration
    • AWS credits cannot be used for domain fees
    • Restoration requires support case and higher fees

Critical Implementation Sequences

  • DNSSEC Implementation:
    • Enabling: Configure at DNS provider FIRST, then add keys to registration
    • Disabling: Delete keys from registration FIRST, wait 3 days, then disable at DNS provider
    • Key Rotation: Add new keys, wait 3 days, delete old keys
    • Improper sequence causes domain outage (SERVFAIL errors)
  • Subdomain Delegation Removal:
    • Delete NS record in parent zone first
    • Wait for TTL duration
    • Then delete child hosted zone
    • Prevents subdomain hijacking
  • DNS Migration:
    • Active domains: Lower TTL, wait for expiration, then update name servers
    • Inactive domains: Direct name server update
    • Name server changes take up to 48 hours to fully propagate

Routing Policy Selection Guide

  • Simple Routing: Single resource, no health checks
  • Weighted Routing: Specified percentages (weights/sum of weights)
  • Latency Routing: Based on AWS Region measurements
  • Failover Routing: Requires health checks for primary
  • Geolocation Routing: Based on user location (continent/country/state)
  • Geoproximity Routing: Based on resource location (requires Traffic Flow)
  • IP-based Routing: Uses CIDR blocks
  • Multivalue Answer: Returns up to 8 healthy records

  • Load Balancer Types:

AWS Route 53 - Advanced Routing Strategies

Complex Routing Combinations

  • Nested Routing Policies:
    • Combining multiple routing types for sophisticated traffic management
    • Implemented using record sets with same name but different routing policies
    • Traffic Flow visual editor simplifies complex configurations
    • Enables multi-criteria routing decisions
  • Common Combinations:
    • Geolocation + Failover: Regional routing with backup
    • Weighted + Latency: Controlled distribution with performance
    • Geoproximity + Health Checks: Distance-based with availability
    • IP-based + Weighted: Network-specific traffic distribution
    • Latency + Failover: Performance-based with disaster recovery

Geolocation Routing Strategies

  • Precision Hierarchy:
    • Most specific match takes precedence
    • Location hierarchy: US state > country > continent > default
    • Default record required for non-matched locations
    • Enables compliance with regional requirements
  • Use Cases:
    • Content localization (language-specific content)
    • Distribution rights restrictions
    • Load balancing across regional endpoints
    • Regulatory compliance (GDPR, data sovereignty)
    • Disaster recovery with regional isolation

Geoproximity Advanced Configuration

  • Bias Optimization:
    • Fine-tuning traffic distribution with bias values (-99 to +99)
    • Positive bias: Expands resource region (attracts more traffic)
    • Negative bias: Shrinks resource region (reduces traffic)
    • Gradual adjustments for optimal distribution
    • Monitoring and iterative refinement
  • Resource Types:
    • AWS Region resources: Automatically uses region coordinates
    • Custom endpoints: Requires latitude/longitude coordinates
    • Mixed deployment: Combination of AWS and non-AWS resources
    • Global coverage with strategic resource placement
    • Traffic Flow required for implementation

IP-based Routing Implementation

  • CIDR Block Configuration:
    • Define IP ranges using CIDR notation
    • Map CIDR blocks to specific endpoints
    • Supports both IPv4 and IPv6 addresses
    • Location mapping file defines CIDR associations
    • Up to 100 CIDR blocks per record
  • Use Cases:
    • ISP-specific routing
    • Enterprise network optimization
    • Content delivery to specific networks
    • Gradual migration between providers
    • Testing with specific user segments

Traffic Flow Advanced Features

  • Visual Editor Capabilities:
    • Drag-and-drop interface for complex routing
    • Nested routing policy configuration
    • Endpoint health check integration
    • Policy versioning and rollback
    • Reusable policies across multiple domains
  • Version Management:
    • Multiple versions of each traffic policy
    • Easy rollback to previous configurations
    • Staged deployment of routing changes
    • Policy comparison between versions
    • Audit trail of routing configuration changes

Multi-Region Application Architecture

    • Applications deployed in multiple regions
