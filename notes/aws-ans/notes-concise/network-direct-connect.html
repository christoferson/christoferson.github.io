AWS Certified Advanced Networking - Specialty (ANS-C01)

Direct Connect

AWS Direct Connect - Core Concepts

What is AWS Direct Connect?

  • Definition and Purpose:
    • Dedicated network connection from on-premises to AWS
    • Bypasses internet service providers in your network path
    • Uses standard Ethernet fiber-optic cable
    • Provides consistent network performance with reduced bandwidth costs
  • Connection Types:
    • Dedicated Connections: 1, 10, 100, or 400 Gbps ports dedicated to a single customer
    • Hosted Connections: 50 Mbps to 10 Gbps connections provided by AWS Direct Connect Partners
    • All connections require single-mode fiber with specific transceiver types

Virtual Interface Types

  • Private Virtual Interface:
    • Connects to Amazon VPC using private IP addresses
    • Requires virtual private gateway or Direct Connect gateway
    • Supports jumbo frames up to 9001 bytes MTU
  • Public Virtual Interface:
    • Connects to all AWS public services using public IP addresses
    • Cannot use Elastic IPs or BYOIP addresses from Amazon pool
    • Limited to 1500 bytes MTU
  • Transit Virtual Interface:
    • Connects to Transit Gateway for multi-VPC and multi-account access
    • Requires Direct Connect gateway
    • Supports jumbo frames up to 8500 bytes MTU

Direct Connect Gateway

  • Purpose and Function:
    • Globally available resource for connecting to multiple VPCs across regions
    • Acts as distributed BGP route reflectors
    • Operates outside the data traffic path
    • Enables access to VPCs in any AWS Region (except China)
  • Association Types:
    • Virtual Private Gateway associations (up to 20 VPCs)
    • Transit Gateway associations (up to 3 Transit Gateways)
    • Cannot mix association types on same Direct Connect gateway

AWS Direct Connect - Resiliency and High Availability

Resiliency Models

  • Maximum Resiliency (99.99% SLA):
    • Separate connections terminating on separate devices at multiple locations
    • Protects against device failures, connectivity failures, and complete location failures
    • Recommended for mission-critical enterprise workloads
    • Most comprehensive protection against all failure scenarios
  • High Resiliency (99.9% SLA):
    • Two single connections to multiple locations
    • Protects against fiber cuts, device failures, and location failures
    • Suitable for production workloads requiring high availability
    • Less complex than maximum resiliency model
  • Development and Test (No specific SLA):
    • Separate connections on separate devices at a single location
    • Protects against device failures only
    • Suitable for non-critical workloads and development environments
    • No protection against location failures

Failover Testing

  • AWS Direct Connect Failover Test:
    • Tests connection resiliency without physical disconnection
    • Disables BGP peering session between on-premises and AWS
    • Duration: 1-180 minutes (configurable)
    • Test history preserved for 365 days
  • Test Capabilities:
    • Works with all virtual interface types
    • Tests both IPv4 and IPv6 BGP sessions
    • Only virtual interface owner can initiate tests
    • Can be canceled while running

Link Aggregation Groups (LAGs)

  • Purpose and Function:
    • Logical interface using LACP (Link Aggregation Control Protocol)
    • Aggregates multiple connections at a single Direct Connect endpoint
    • Simplifies management of multiple connections
    • Operates in Active/Active mode only
  • Key Limitations:
    • MLAG (Multi-chassis LAG) is NOT supported
    • Maximum 4 connections for speeds < 100 Gbps
    • Maximum 2 connections for 100/400 Gbps
    • All connections must be same bandwidth and terminate at same endpoint
  • Operational Features:
    • Configurable minimum operational connections threshold
    • LAG becomes non-operational if connections fall below threshold
    • Single MACsec configuration applies to all member connections

AWS Direct Connect - Security and Encryption

Data Protection Options

  • Default Behavior:
    • Traffic is not encrypted by default
    • Requires explicit encryption configuration
    • Multiple encryption options available based on requirements
  • Encryption Methods:
    • AWS Site-to-Site VPN over Direct Connect
    • MAC Security (MACsec)
    • Transit Gateway with VPN attachments
    • Application-level encryption

MACsec (MAC Security)

  • Overview:
    • IEEE 802.1AE standard for Layer 2 encryption
    • Provides data confidentiality, integrity, and origin authenticity
    • Point-to-point encryption between customer router and AWS Direct Connect router
    • No additional charge for MACsec capability
  • Connection Support:
    • Available on 10, 100, and 400 Gbps dedicated connections only
    • Not available on 1 Gbps connections or hosted connections
    • Requires MACsec-capable interface on customer device
    • Connection must be Layer 2 transparent
  • Technical Requirements:
    • 10 Gbps: Supports GCM-AES-256 and GCM-AES-XPN-256
    • 100/400 Gbps: Requires GCM-AES-XPN-256 only
    • 256-bit keys only
    • SCI (Secure Channel Identifier) must be enabled
    • No support for dot1q-in-clear (VLAN tag outside encrypted payload)

Identity and Access Management

  • Authentication Methods:
    • IAM users, roles, and federated identities
    • BGP MD5 authentication (mandatory)
    • Service-linked roles for MACsec integration
  • Key Managed Policies:
    • AWSDirectConnectFullAccess
    • AWSDirectConnectReadOnlyAccess
    • AWSDirectConnectServiceRolePolicy (for service-linked role)
  • Resource-Level Permissions:
    • Support for resource ARNs (connections, LAGs, virtual interfaces, gateways)
    • Tag-based access control
    • Cross-account access capabilities

AWS Direct Connect - Routing and BGP Configuration

BGP Requirements and Configuration

  • Basic Requirements:
    • BGP (Border Gateway Protocol) required for all Direct Connect connections
    • BGP MD5 authentication mandatory
    • One BGP session per IP family per virtual interface
    • Maximum 100 routes per BGP session for private/transit VIFs
    • Maximum 1000 routes for public VIFs
  • ASN Configuration:
    • 16-bit ASN range: 64512-65534 (private)
    • 32-bit ASN range: 1-2147483647
    • Cannot use same ASN for customer and AWS gateways
    • Must use unique ASNs for multi-region Transit Gateways
  • BGP Timers:
    • Hold timer: 90 seconds default, 3 seconds minimum
    • Keepalive timer: 30 seconds default, 1 second minimum
    • Graceful restart timer: 120 seconds
    • BFD minimum interval: 300ms, multiplier: 3

Traffic Engineering with BGP

  • Path Selection Methods (Priority Order):
    • 1. Longest Prefix Match
    • 2. Local Preference
    • 3. AS_PATH Length
    • 4. MED (Multi-Exit Discriminator)
  • Local Preference BGP Communities:
    • 7224:7100 - Low Preference (backup/passive paths)
    • 7224:7200 - Medium Preference (default paths)
    • 7224:7300 - High Preference (primary/active paths)
    • Communities are mutually exclusive
  • Public VIF BGP Communities:
    • 7224:9100 - Local AWS Regions
    • 7224:9200 - Continental Regions (North America, Asia Pacific, EMEA)
    • 7224:9300 - Global (all public AWS Regions)
    • 7224:8100 - Same Region Routes
    • 7224:8200 - Same Continent Routes

Allowed Prefixes Behavior

  • Virtual Private Gateway Associations:
    • Acts as prefix filter only
    • Only advertises actual VPC CIDR
    • Must be same or wider than VPC CIDR
    • Example: For VPC CIDR 10.0.0.0/16, allowed prefix 10.0.0.0/15 will advertise 10.0.0.0/16
  • Transit Gateway Associations:
    • Directly advertises allowed prefixes
    • Independent of attached VPC CIDRs
    • Originates from Direct Connect gateway ASN
    • Example: For VPC CIDR 10.0.0.0/16, allowed prefix 22.0.0.0/24 will advertise 22.0.0.0/24
  • Important Limitations:
    • No prefix overlaps allowed with multiple transit gateways
    • Cannot use 0.0.0.0/0 with multiple transit gateways
    • Each prefix must be unique across associations

AWS Direct Connect - Gateway Integrations

Direct Connect Gateway with Virtual Private Gateway

  • Key Capabilities:
    • Connects DX connection to VPCs across accounts and regions
    • Uses private virtual interface
    • Enables multi-VPC connectivity through single connection
    • Global resource - create in any region
  • Critical Rules:
    • One VGW per Direct Connect gateway
    • VGW must be attached to a VPC
    • Cannot attach if already associated with Transit Gateway
    • No overlapping CIDR blocks between associated VPCs
    • Do not enable route propagation before gateway association
  • Association Process:
    • Association proposals expire after 7 days if not accepted
    • Accepted/deleted proposals visible for 3 days
    • VGW detachment from VPC auto-disassociates from DX gateway

Direct Connect Gateway with Transit Gateway

  • Key Capabilities:
    • Connect DX connection to multiple VPCs/VPNs through Transit Gateway
    • Uses transit virtual interface
    • Enables hub-and-spoke networking model
    • Maximum 3 Transit Gateway associations per DX gateway
  • Critical Rules:
    • Cannot attach if already associated with VGW
    • Cannot attach if connected to private virtual interface
    • Must use unique ASNs for multi-region Transit Gateways
    • Point-to-point /30 ranges don't propagate to Transit Gateway
    • Association can take up to 40 minutes to establish
  • Cross-Account Associations:
    • Transit Gateway owner creates association proposal
    • DX Gateway owner must accept proposal
    • Can include allowed prefix specifications
    • DX Gateway owner can override requested prefixes

Direct Connect Gateway with Cloud WAN

  • Key Capabilities:
    • Routes traffic between core network edge locations and DX connections
    • Uses shortest available path
    • Enables central policy-based management
    • Supports automated routing through BGP
  • Critical Limitations:
    • One core network per DX gateway
    • Single segment association only
    • Cannot associate with other AWS region resources simultaneously
    • No allowed prefixes list support
    • All segment prefixes advertised to DX gateway
    • ASN must be outside core network ASN range
  • Management Scope:
    • Created and managed from Cloud WAN Console
    • Uses Network Manager for administration
    • Direct Connect console shows read-only information

AWS Direct Connect - SiteLink and Advanced Features

AWS Direct Connect SiteLink

  • Purpose and Function:
    • Enables direct PoP-to-PoP connectivity between Direct Connect locations
    • Uses shortest available AWS network path
    • Bypasses regional routing requirements
    • Reduces latency between on-premises sites
  • Configuration Requirements:
    • Enable on VIFs at two or more Direct Connect locations
    • All locations must attach to same Direct Connect gateway
    • Works with existing Direct Connect connections
    • No new connections required
  • Support Matrix:
    • Supported: Transit virtual interfaces, Private VIF with Direct Connect gateway
    • Not Supported: Private VIF attached to virtual gateway, Public virtual interfaces
    • Not available in AWS GovCloud (US) or China Regions
  • Technical Specifications:
    • BGP required
    • IPv6 supported
    • MACsec supported (if port/PoP supports it)
    • No managed QoS functionality, but DSCP markings preserved
    • BGP Communities for path preference:
      • 7224:7100 - Low preference
      • 7224:7200 - Medium preference
      • 7224:7300 - High preference

AWS Local Zones Integration

  • Connectivity Options:
    • Direct Connect can connect to Local Zones
    • Data travels directly to/from Local Zones
    • Bypasses AWS Region
    • Improves performance and reduces latency
  • Configuration Process:
    • Extend VPC from parent Region
    • Create new subnet and assign to Local Zone
    • Private Connection: Associate VGW to Direct Connect private VIF or DX Gateway
    • Public Connection: Use Direct Connect Public VIF with IGW
  • Key Limitations:
    • Transit Gateway not supported in Local Zones
    • MTU size: 1468 (vs 9001 in Regions)*
    • Single flow limit: ~2.5 Gbps (vs 5 Gbps in Regions)
    • Ingress routing goes through parent Region first
    • *Exception: Los Angeles Local Zone not affected by MTU and flow limits

Remote Region Access

  • Global Access Capabilities:
    • Access to all public Regions from any DX location
    • Includes AWS GovCloud (US)
    • Excludes China Regions (Beijing and Ningxia)
    • Uses AWS global network backbone
  • Remote Public Service Access:
    • Requires public virtual interface
    • Established BGP session
    • Router learns routes for all public AWS Regions
    • Uses AWS IP Address Ranges (published in documentation)
  • Remote VPC Access:
    • Primary Method: Direct Connect Gateway
    • Alternative Method: VPN over Public VIF
    • Outbound data from remote Region charged at that Region's rate

AWS Direct Connect - Maintenance and Operations

Maintenance Events

  • Types of Maintenance:
    • Planned Maintenance:
      • Scheduled in advance
      • Improves availability
      • Delivers new features
    • Emergency Maintenance:
      • Critical service-impacting failures
      • Requires immediate AWS action
      • Not planned in advance
  • Notification Timeline:
    • Planned Maintenance:
      • 14 calendar days notice
      • 7 calendar days notice
      • 1 calendar day notice
      • Calendar days include non-business days and holidays
    • Emergency Maintenance:
      • Up to 60 minutes prior notification
      • Cannot be canceled
  • Notification Methods:
    • AWS Personal Health Dashboard
    • Email notifications
    • AWS Direct Connect Console subscriptions

Monitoring and Metrics

  • CloudWatch Key Metrics:
    • ConnectionState: 1 = up, 0 = down
    • ConnectionBpsEgress/Ingress: Bandwidth utilization
    • ConnectionPpsEgress/Ingress: Packet rates
    • ConnectionErrorCount: Includes CRC errors
    • ConnectionLightLevelTx/Rx: Fiber connection health (dBm)
    • VirtualInterfaceBpsEgress/Ingress: VIF bandwidth
  • Metric Collection:
    • Default interval: 5 minutes
    • Minimum interval: 1 minute
    • Error counts reset after each reporting interval
    • Light level normal range: -10 to -3 dBm
  • Monitoring Tools:
    • CloudWatch Alarms for threshold-based notifications
    • CloudTrail for API activity logging
    • Direct Connect Console for connection status
    • CloudWatch Dashboard for metrics visualization

Troubleshooting Common Issues

  • Connection Issues:
    • Light Levels: Below -10 dBm indicates potential problems
    • Auto-Negotiation: Must be disabled for speeds > 1 Gbps
    • Physical connectivity: Check cross-connect and fiber issues
  • BGP Session Problems:
    • ASN mismatch
    • MD5 authentication failure
    • Route limit exceeded (>100 for private/transit VIF, >1000 for public VIF)
    • BGP timer misconfigurations
  • Recovery Times:
    • BGP reconvergence: 3-5 minutes
    • LAG member failure: Under 1 minute
    • Full connection failure: Based on failover design

AWS Direct Connect - Design Patterns and Best Practices

Network Design Patterns

  • Single Region Access:
    • Direct private virtual interface to VGW
    • No Direct Connect gateway required
    • Simplest configuration for single VPC access
    • Limited to resources in one region
  • Multi-Region Access:
    • Direct Connect gateway with private virtual interfaces
    • Enables access to VPCs across regions
    • No overlapping CIDR blocks allowed
    • Traffic takes shortest path to destination region
  • Hub-and-Spoke Design:
    • Transit Gateway with transit virtual interfaces
    • Centralized routing and policy control
    • Supports VPC-to-VPC communication
    • Enables complex network segmentation
  • Global Network Architecture:
    • Cloud WAN with Direct Connect gateway
    • Policy-based global network management
    • Automated routing through BGP
    • Centralized control with distributed connectivity

High Availability Design Patterns

  • Active/Passive Configuration:
    • Primary and backup connections
    • Use BGP path attributes for traffic steering:
      • AS_PATH prepending
      • Local preference communities
      • MED values
    • Automatic failover based on BGP path selection
  • Active/Active Configuration:
    • Load balancing across multiple connections
    • Equal cost multi-path routing
    • Requires careful prefix advertisement
    • Maximizes available bandwidth
  • Hybrid Connectivity:
    • Direct Connect with VPN backup
    • Direct Connect path always preferred when available
    • Automatic failover to VPN
    • Ensure VPN can handle failover traffic volume

Performance Optimization

  • Bandwidth Optimization:
    • Use LAGs to aggregate multiple connections
    • Configure jumbo frames where supported
    • Consider traffic patterns when selecting connection speeds
    • Monitor bandwidth utilization for capacity planning
  • Latency Optimization:
    • Use Direct Connect locations closest to your network
    • Implement SiteLink for direct PoP-to-PoP connectivity
    • Use Local Zones for latency-sensitive applications
    • Consider Global Accelerator for improved performance
  • Route Optimization:
    • Implement route summarization
    • Use BGP communities for traffic engineering
    • Configure BFD for faster failure detection
    • Optimize prefix advertisements

AWS Direct Connect - Implementation and Troubleshooting

Implementation Process

  • Connection Setup:
    • Request connection through AWS Management Console, CLI, or API
    • Receive LOA-CFA (Letter of Authorization and Connecting Facility Assignment)
    • Complete cross-connect with colocation provider within 90 days
    • Configure customer router with appropriate settings
  • Virtual Interface Creation:
    • Create appropriate virtual interface type (private, public, transit)
    • Configure VLAN tag (1-4094, unique per connection)
    • Set up BGP peering with MD5 authentication
    • Configure IP addressing (AWS-generated or customer-specified)
  • Gateway Configuration:
    • Create Direct Connect gateway if needed
    • Associate with virtual private gateways or transit gateways
    • Configure allowed prefixes for routing control
    • Wait for association to complete before enabling route propagation

Common Implementation Challenges

  • Physical Layer Issues:
    • Incorrect fiber type (single-mode required)
    • Unsupported transceivers
    • Auto-negotiation misconfiguration (must be disabled for speeds > 1 Gbps)
    • Light level problems (normal range: -10 to -3 dBm)
  • BGP Configuration Issues:
    • ASN mismatch between customer and AWS
    • MD5 authentication key mismatch
    • Exceeding route advertisement limits
    • Incorrect BGP timer settings
  • Gateway Association Issues:
    • Overlapping CIDR blocks
    • Mixing association types on same gateway
    • Enabling route propagation before association completes
    • Incorrect allowed prefix configuration

Troubleshooting Methodology

  • Physical Connectivity:
    • Check connection state in AWS console
    • Verify light levels (ConnectionLightLevelTx/Rx metrics)
    • Confirm cross-connect completion with facility provider
    • Validate physical interface settings on customer router
  • BGP Session Issues:
    • Verify BGP configuration on both sides
    • Check for route advertisement limits
    • Confirm MD5 authentication key matches
    • Examine BGP timer settings
  • Routing Problems:
    • Verify route propagation settings
    • Check allowed prefixes configuration
    • Examine route tables in VPCs and on-premises routers
    • Use Reachability Analyzer for connectivity testing
  • Performance Issues:
    • Monitor bandwidth utilization (ConnectionBpsEgress/Ingress)
    • Check for packet errors (ConnectionErrorCount)
    • Verify MTU settings match end-to-end
    • Test with different traffic patterns

AWS Direct Connect - Critical Numbers and Limitations Summary

Connection Specifications

  • Dedicated Connection Speeds:
    • 1 Gbps: 1000BASE-LX (1310 nm)
    • 10 Gbps: 10GBASE-LR (1310 nm)
    • 100 Gbps: 100GBASE-LR4
    • 400 Gbps: 400GBASE-LR4
  • Hosted Connection Speeds:
    • 50 Mbps to 10 Gbps
    • Specific options: 50M, 100M, 200M, 300M, 400M, 500M, 1G, 2G, 5G, 10G, 25G
    • 25G only available where 100G ports exist
  • MTU Sizes:
    • Private VIF: Up to 9001 bytes
    • Transit VIF: Up to 8500 bytes
    • Public VIF: 1500 bytes only
    • Local Zones: 1468 bytes (except Los Angeles)

Critical Quotas

  • Virtual Interface Limits:
    • Dedicated Connection: 51 total VIFs (50 private/public + 4 transit)
    • Hosted Connection: 1 VIF only
  • BGP Limits:
    • 100 routes per private/transit VIF
    • 1000 routes per public VIF
    • Hold timer: 90s default, 3s minimum
    • Keepalive timer: 30s default, 1s minimum
    • BFD minimum interval: 300ms
  • Gateway Limits:
    • 20 VGWs per Direct Connect gateway
    • 3 Transit Gateways per Direct Connect gateway
    • 30 VIFs per Direct Connect gateway
    • 20 Direct Connect gateways per Transit Gateway

Key Timeframes

  • Implementation Timelines:
    • LOA-CFA validity: 90 days
    • VGW association proposal expiration: 7 days
    • Accepted/deleted proposal visibility: 3 days
    • Transit Gateway association: Up to 40 minutes
  • Maintenance Notifications:
    • Planned maintenance: 14, 7, and 1 calendar day notices
    • Emergency maintenance: Up to 60 minutes prior
    • Calendar days include non-business days and holidays
  • Failover Test Duration:
    • Minimum: 1 minute
    • Maximum: 180 minutes (3 hours)
    • Test history preserved for 365 days

SLA and Resiliency

  • Maximum Resiliency:
    • 99.99% SLA
    • Multiple locations, separate devices
    • Protects against device, connectivity, AND location failures
  • High Resiliency:
    • 99.9% SLA
    • Multiple locations, single connections
    • Protects against fiber cuts and location failures
  • Development and Test:
    • No specific SLA
    • Single location, separate devices
    • Only protects against device failures

AWS Direct Connect - Comparison with Other Connectivity Options

Direct Connect vs. Site-to-Site VPN

  • Direct Connect Advantages:
    • Consistent, predictable network performance
    • Reduced data transfer costs for high-volume traffic
    • Private network connection that bypasses internet
    • Higher bandwidth options (up to 400 Gbps)
    • Support for jumbo frames
  • Site-to-Site VPN Advantages:
    • Quick to set up (minutes vs. weeks)
    • Lower initial cost (no dedicated circuit required)
    • Built-in encryption
    • Greater geographic flexibility
    • Easier to implement redundancy
  • When to Use Together:
    • Direct Connect as primary path
    • VPN as backup for failover
    • Direct Connect path always preferred when available
    • Ensure VPN can handle failover traffic volume

Direct Connect vs. VPC Peering

  • Direct Connect Use Cases:
    • Connecting on-premises networks to AWS
    • Hybrid cloud architectures
    • High-bandwidth, consistent connections
    • Access to multiple VPCs across regions
  • VPC Peering Use Cases:
    • Connecting VPCs within AWS
    • Low-latency communication between VPCs
    • No bandwidth constraints
    • No additional gateway or connection charges
  • Key Differences:
    • Direct Connect connects on-premises to AWS; VPC Peering connects VPC to VPC
    • Direct Connect requires physical infrastructure; VPC Peering is purely virtual
    • Direct Connect has bandwidth limitations; VPC Peering has no bandwidth constraints
    • Direct Connect supports global connectivity; VPC Peering is region-specific (except inter-region peering)

Direct Connect vs. Transit Gateway

  • Direct Connect Focus:
    • Physical connectivity between on-premises and AWS
    • Layer 2 and Layer 3 connectivity
    • Dedicated bandwidth
    • Bypassing the public internet
  • Transit Gateway Focus:
    • Network transit hub for connecting VPCs and on-premises networks
    • Simplified network architecture
    • Centralized routing control
    • Regional resource with cross-region peering
  • Integration Benefits:
    • Direct Connect provides physical connectivity
    • Transit Gateway provides network routing and segmentation
    • Transit virtual interface connects Direct Connect to Transit Gateway
    • Enables complex hub-and-spoke architectures

Direct Connect vs. AWS PrivateLink

  • Direct Connect Purpose:
    • Dedicated network connection to AWS
    • Access to multiple AWS services and VPCs
    • Layer 2 connectivity option
    • Requires physical infrastructure
  • PrivateLink Purpose:
    • Private connectivity to services
    • No traversal of public internet
    • Service endpoint access
    • No physical infrastructure required
  • When to Use Each:
    • Direct Connect: For dedicated connection from on-premises to AWS
    • PrivateLink: For private access to services without exposing to internet
    • Can be used together: Direct Connect for connectivity, PrivateLink for service access

AWS Direct Connect - Advanced Scenarios and Solutions

Multi-Account Architectures

  • Cross-Account Access:
    • Direct Connect owner creates hosted virtual interface
    • Allocates to another AWS account
    • Recipient accepts and associates with their resources
    • Enables shared connection scenarios
  • Gateway Association Proposals:
    • Direct Connect gateway owner creates association proposal
    • VGW or Transit Gateway owner accepts proposal
    • Enables cross-account gateway associations
    • Proposals expire after 7 days if not accepted
  • AWS RAM Integration:
    • Share Direct Connect gateways across accounts
    • Centralized management of Direct Connect resources
    • Consistent network policies across organizational units
    • Simplified billing and cost allocation

Complex Routing Scenarios

  • Overlapping IP Address Spaces:
    • Cannot connect overlapping CIDRs to same Direct Connect gateway
    • Solutions:
      • Use separate Direct Connect gateways
      • Implement NAT at customer gateway
      • Redesign IP addressing scheme
      • Use Transit Gateway with route tables for segmentation
  • Asymmetric Routing:
    • Occurs when traffic takes different paths in each direction
    • Solutions:
      • Consistent BGP path attributes

Scenario 1: Designing for High Availability

  • Problem Statement:
    • A financial services company requires 99.99% SLA for their AWS connectivity
    • They have mission-critical applications that cannot tolerate downtime
    • They need to design a Direct Connect solution that meets these requirements
  • Solution Analysis:
    • Implement Maximum Resiliency model:
      • Separate connections terminating on separate devices
      • Multiple Direct Connect locations
      • Protects against device failures, connectivity failures, and location failures
    • Additional considerations:
      • Configure BGP for automatic failover
      • Implement BFD for faster failure detection
      • Consider VPN backup for additional redundancy
      • Regularly test failover with Direct Connect Failover Test feature

Scenario 2: Multi-Region Access

  • Problem Statement:
    • A global enterprise has VPCs in us-east-1, eu-west-1, and ap-southeast-2
    • They need to connect their on-premises data center to all three regions
    • They want to minimize the number of Direct Connect connections
    • They need to ensure no overlapping IP addresses between VPCs
  • Solution Analysis:
    • Implement Direct Connect gateway with private virtual interfaces:
      • Create Direct Connect gateway
      • Associate virtual private gateways from each region
      • Create private virtual interface to Direct Connect gateway
      • Traffic takes shortest path to each region
    • Address overlapping IP concerns:
      • Ensure VPC CIDRs don't overlap
      • If overlap exists, use separate Direct Connect gateways
      • Consider NAT or IP address redesign if necessary

Scenario 3: Transit Gateway Integration

  • Problem Statement:
    • A company has 50 VPCs across multiple AWS accounts
    • They need to connect all VPCs to their on-premises network
    • They want centralized routing control and network segmentation
    • They need to minimize the number of BGP sessions
  • Solution Analysis:
    • Implement Transit Gateway with transit virtual interface:
      • Create Transit Gateway and attach VPCs
      • Create Direct Connect gateway
      • Create transit virtual interface
      • Associate Transit Gateway with Direct Connect gateway
      • Configure allowed prefixes for routing control
    • Benefits of this approach:
      • Single BGP session for multiple VPCs
      • Centralized routing control with Transit Gateway route tables
      • Network segmentation with Transit Gateway route tables
      • Simplified management of multiple VPCs

Scenario 4: Security Requirements

  • Problem Statement:
    • A healthcare organization needs to connect to AWS
    • They have strict data protection requirements
    • All data in transit must be encrypted
    • They need high bandwidth (10 Gbps) connectivity
  • Solution Analysis:
    • Options for encrypted connectivity:
      • MACsec encryption on 10 Gbps Direct Connect:
        • Layer 2 encryption
        • Provides data confidentiality, integrity, and origin authenticity
        • Requires MACsec-capable customer device
        • Supports GCM-AES-256 or GCM-AES-XPN-256
      • VPN over Direct Connect:
        • IPsec encryption
        • End-to-end encryption
        • May have bandwidth limitations compared to pure Direct Connect
    • Recommended approach:
      • MACsec for physical link security
      • Application-level encryption for sensitive data
      • Regular security audits and compliance checks

Scenario 5: Bandwidth Optimization

  • Problem Statement:
    • A media company transfers large video files between on-premises and AWS
    • They currently have a 10 Gbps Direct Connect connection that's reaching capacity
    • They need to increase bandwidth without disrupting existing connectivity
    • They want to simplify management of multiple connections

Scenario 6: Troubleshooting BGP Issues

  • Problem Statement:
    • A company has established a Direct Connect connection with a private virtual interface
    • The connection shows as "available" but the BGP session is down
    • They need to identify and resolve the issue
  • Solution Analysis:
    • Systematic troubleshooting approach:
      • Check BGP configuration on both sides:
        • Verify ASN matches configuration
        • Confirm MD5 authentication key matches
        • Check IP addressing configuration
        • Verify BGP timers are compatible
      • Check for route advertisement limits:
        • Private VIF limit: 100 routes
        • Exceeding limit causes BGP session to go idle
      • Verify physical connectivity:
        • Check light levels (normal range: -10 to -3 dBm)
        • Confirm auto-negotiation settings (disabled for >1 Gbps)
    • Common resolution steps:
      • Correct MD5 authentication key
      • Reduce advertised routes to below limit
      • Fix ASN configuration
      • Address physical layer issues

Scenario 7: Cross-Account Access

  • Problem Statement:
    • A large enterprise has a centralized networking account with Direct Connect
    • They need to provide connectivity to VPCs in multiple application accounts
    • They want to maintain centralized control of Direct Connect resources
    • They need to minimize administrative overhead
  • Solution Analysis:
    • Options for cross-account connectivity:
      • Direct Connect gateway with VGW associations:
        • Create Direct Connect gateway in networking account
        • Create association proposals for VGWs in application accounts
        • Application account owners accept proposals
        • Limited to 20 VPCs per Direct Connect gateway
      • Transit Gateway with Direct Connect integration:
        • Create Transit Gateway in networking account
        • Share Transit Gateway with RAM to application accounts
        • Application accounts create Transit Gateway attachments
        • Connect Transit Gateway to Direct Connect gateway
        • Supports more VPCs and additional routing control
    • Recommended approach for large enterprise:
      • Transit Gateway with Direct Connect integration
      • AWS Resource Access Manager for sharing
      • Centralized monitoring and management
      • Network segmentation with Transit Gateway route tables
