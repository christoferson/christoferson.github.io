AWS Certified Advanced Networking - Specialty (ANS-C01)

AWS Route 53 Resolver - Core Concepts

Introduction

  • What is Route 53 Resolver:
    • Recursive DNS service available by default in all VPCs
    • Responds to DNS queries for:
      • Public DNS records
      • VPC-specific DNS names
      • Route 53 private hosted zones

Architecture Components

  • Core Components:
    • Nitro Resolver Service:
      • Runs on Nitro card or Dom0
      • Handles local DNS resolution
      • Maintains local cache
    • Zonal Resolver Fleet:
      • Highly available resolver fleet
      • Located in same AZ as instance
      • Handles non-cached queries

Key Features

  • Resolver Capabilities:
    • Automatic DNS resolution for:
      • Local VPC domain names
      • Private hosted zones
      • Public domain recursive lookups
    • Hybrid DNS Resolution:
      • Inbound endpoints for on-premises to AWS
      • Outbound endpoints for AWS to on-premises
      • Conditional forwarding rules

Exam Tips

  • Remember:
    • VPC+2 IP address connects to Resolver
    • Regional service with AZ-level components
    • Automatic caching and failover capabilities
    • Requires VPN/Direct Connect for hybrid scenarios
  • Key Concepts:
    • Transparent recursive DNS service
    • Hybrid DNS resolution capabilities
    • Automatic integration with AWS services
    • Built-in high availability

AWS Route 53 Resolver - Endpoints and Rules

Resolver Endpoints

  • Inbound Endpoints:
    • Purpose: Allow DNS queries from on-premises to VPC
    • Key Requirements:
      • Minimum 2 IP addresses in different AZs
      • VPN or Direct Connect connection required
      • Security group must allow port 53 (TCP/UDP)
      • For DoH, port 443 must be allowed
  • Outbound Endpoints:
    • Purpose: Forward DNS queries from VPC to on-premises
    • Requirements:
      • Minimum 2 IP addresses in different AZs
      • VPN, Direct Connect, or NAT gateway needed
      • Must be associated with forwarding rules

Resolver Rules

  • Rule Types:
    • Forwarding Rules:
      • Forward queries to specified resolvers
      • Domain-specific forwarding
      • Requires outbound endpoint
    • System Rules:
      • Override forwarding behavior
      • Used for specific subdomains
      • Takes precedence over forwarding rules
    • Recursive Rules:
      • Default "Internet Resolver" rule
      • Handles non-matched domains
      • Automatic recursive resolution

Rule Processing

  • Rule Matching:
    • Most specific domain match wins
    • Example:
      • Rule for example.com
      • Rule for dev.example.com
      • Query for dev.example.com uses more specific rule
  • Rule Sharing:
    • Can be shared across AWS accounts
    • Regional service - rules are region-specific
    • Outbound endpoints shared with rules

Important Limitations

  • Endpoint Constraints:
    • Cannot create in dedicated-tenancy VPCs
    • IP addresses must be available in subnet
    • Cannot use shared subnet IP addresses
    • Maximum query limits per IP address
  • Rule Constraints:
    • One rule per domain name
    • Rules are region-specific
    • Must be associated with VPCs
    • Cannot forward some AWS internal domains

Exam Tips

  • Key Concepts:
    • Minimum 2 IP addresses per endpoint
    • Rule precedence based on specificity
    • Hybrid connectivity requirements
    • Regional service limitations
  • Common Scenarios:
    • Hybrid DNS resolution setup
    • Rule configuration and precedence
    • Endpoint requirements
    • Cross-account sharing

AWS Route 53 Resolver - Availability and Scaling

Architecture Components

  • Two-Tier Architecture:

      • Maintains local cache
    • Zonal Resolver Fleet:
      • Highly available resolver infrastructure
      • Operates within same AZ
      • Handles non-cached queries
      • Manages external DNS resolution

Scaling Characteristics

  • Link-Local Services:
    • Part of instance network interface capacity
    • 1024 packets per second (PPS) limit per interface
    • Includes:
      • Route 53 Resolver
      • Amazon Time Service (NTP)
      • Instance Metadata Service (IMDS)
      • Windows Licensing Service
  • Scaling Mechanisms:
    • Scales with network interfaces
    • No per-VPC aggregate query limit

High Availability Features

  • Failure Handling:
    • Transparent failure handling
    • Cache-based resilience
    • Extended TTL during upstream failures
    • Automatic query rerouting
  • Cache Behavior:
    • May exceed TTL during failures
    • Handles timeouts gracefully
    • Manages refused connections
    • Handles SERVFAIL responses

Security and Network Design

  • Network Isolation:
    • Resolver traffic outside customer VPC
    • Controlled network access
    • Rigorous security controls
    • No customer traffic interception
  • External Resolution:
    • Uses public IP addresses
    • No EDNS0-Client Subnet support
    • Client IP information not forwarded

Monitoring and Troubleshooting

  • Performance Monitoring:
    • linklocal_allowance_exceeded metric
    • Available via ethtool
    • CloudWatch integration possible
    • Per-interface monitoring

Exam Tips

  • Key Points:
    • 1024 PPS limit per interface
    • Scales with network interfaces
    • No explicit VPC query limits
    • Built-in failure handling
  • Architecture Understanding:
    • Two-tier resolver design
    • Local caching benefits
    • Automatic failover capabilities
    • Network isolation principles

AWS Route 53 Resolver - DNSSEC Validation

Introduction to DNSSEC

  • Purpose:
    • Cryptographically verify DNS responses
    • Prevent DNS tampering
    • Ensure response integrity
    • VPC-level security feature

Key Characteristics

  • Validation Scope:
    • Applies to public signed names only
    • Works during recursive DNS resolution
    • VPC-specific configuration
  • Important Limitations:
    • Route 53 Resolver:
      • Ignores DO (DNSSEC OK) EDNS header bit
      • Ignores CD (Checking Disabled) bit
      • Doesn't return DNSSEC records
      • Doesn't set AD bit in responses

Implementation Considerations

  • Critical Warnings:
    • Can impact DNS resolution for public records
    • Potential outage risk during enablement
    • Changes take several minutes to apply
    • Affects all DNS queries in VPC
  • Forwarding Scenarios:
    • When forwarding to another resolver:
      • Target resolver must perform validation
      • Route 53 Resolver delegates validation
      • Validation responsibility transfers

Custom Validation Requirements

  • Self-Validation Needs:
    • Not supported by Route 53 Resolver
    • Requires own recursive resolution
    • Must implement outside Route 53

Exam Tips

  • Key Points:
    • VPC-level configuration
    • Applies only to public signed names
    • Cannot perform custom validation
    • Changes require several minutes
  • Remember:
    • Potential service impact during enablement
    • Forwarding affects validation responsibility
    • No support for DNSSEC record return
    • No custom validation support
  • Common Scenarios:
    • Enabling DNSSEC for VPC
    • Understanding validation scope
    • Handling forwarded queries
    • Managing implementation risks

AWS Route 53 - Routing Traffic to AWS Resources

Introduction to Resource Routing

  • Core Concept:
    • Route 53 can direct traffic to various AWS services
    • Uses DNS records to map domain names to AWS resources
    • Supports both root domains and subdomains
    • Provides health checking and failover capabilities

Alias Records vs. CNAME Records

  • Alias Records (Route 53 Specific):
    • Can be used for root domain (example.com) and subdomains
    • No charge for alias queries to AWS resources
    • Automatically updates if target IP changes
    • Supports health checking of targets
    • Recommended for AWS service integration
  • CNAME Records (Standard DNS):
    • Can only be used for subdomains (www.example.com)
    • Each query incurs standard Route 53 charges
    • Cannot be used at zone apex (root domain)
    • Standard DNS feature supported by all DNS providers

Common AWS Resource Integrations

  • Compute Services:
    • EC2 Instances: Direct IP mapping
    • Elastic Beanstalk: Environment URLs
    • App Runner: Service endpoints
  • Content Delivery:
    • CloudFront: Global distribution endpoints
    • S3 Website Buckets: Static website hosting
    • Global Accelerator: Static IP addresses
  • Load Balancing:
    • Application Load Balancers
    • Network Load Balancers
    • Classic Load Balancers
  • API and Networking:
    • API Gateway: Custom domain names
    • VPC Interface Endpoints: PrivateLink connections
    • WorkMail: Email routing

Key Prerequisites

  • For All Resource Types:
    • Registered domain name
    • Route 53 as DNS service for domain
    • Properly configured target AWS resource
  • Resource-Specific Requirements:
    • S3 Buckets: Must match domain name exactly
    • CloudFront: Requires alternate domain name configuration
    • API Gateway: Custom domain name setup
    • EC2: Elastic IP recommended for stability

Record Configuration Patterns

  • Common Settings:
    • Record Type: Usually A (IPv4) or AAAA (IPv6)
    • Alias Target: AWS resource-specific endpoint
    • Evaluate Target Health: Usually Yes
    • TTL: Not applicable for alias records
  • Special Cases:
    • EC2 Instances: Non-alias A record with direct IP
    • Older Elastic Beanstalk: CNAME record required
    • WorkMail: Requires MX, TXT, and CNAME records

Exam Tips

  • Key Concepts:
    • Alias records are Route 53-specific extensions
    • No charges for alias queries to AWS resources
    • Root domains require alias records (not CNAME)
    • S3 website bucket names must match domain name
  • Common Scenarios:
    • Choosing between alias and CNAME records
    • Setting up custom domains for AWS services
    • Configuring health checks for failover
    • Understanding resource-specific requirements

AWS Route 53 - DNSSEC Study Guide

Introduction to DNSSEC

  • Core Purpose:
    • Protects domains from DNS spoofing/man-in-the-middle attacks
    • Secures DNS traffic through cryptographic signatures
    • Establishes a chain of trust for DNS responses
    • Verifies authenticity of DNS records
  • Security Problem Solved:
    • Prevents attackers from intercepting DNS queries
    • Blocks substitution of fake IP addresses in responses
    • Protects users from being redirected to malicious sites
    • Ensures DNS data integrity

DNSSEC Components and Concepts

  • Key Terminology:
    • Key Pairs: Public and private cryptographic keys
    • DS Record (Delegation Signer): Public key stored at parent zone
    • RRSIG: Resource Record Signature that validates DNS data
    • KSK (Key Signing Key): Signs the ZSK
    • ZSK (Zone Signing Key): Signs the actual DNS records
  • Chain of Trust:
    • Begins at TLD registry (parent zone)
    • Extends to authoritative name servers
    • Validates each link in DNS resolution chain
    • Requires DNSSEC-aware resolvers

Route 53 DNSSEC Support

  • Two Distinct Features:
    • DNSSEC signing: For DNS service (hosted zones)
    • DNSSEC for domain registration: Managing keys with registrar
  • Implementation Options:
    • Route 53 as both registrar and DNS service
    • Route 53 as registrar with external DNS service
    • Route 53 as DNS service with external registrar

DNSSEC Resolution Process

  • How It Works:
    • Client submits DNS query
    • Resolver checks if answer is cached
    • If not cached, resolver requests DS record from TLD registry
    • Resolver follows chain to authoritative name servers
    • Name server returns signed DNS record
    • Resolver verifies signature using public key
    • If verified, returns answer to client; if not, returns error

Prerequisites and Limitations

  • Requirements:
    • TLD must support DNSSEC (check Route 53 documentation)
    • DNS service provider must support DNSSEC
    • DNSSEC must be configured at DNS provider before adding keys to registrar
  • Key Limitations:
    • .com and .net domains: Up to 13 public keys
    • All other domains: Up to 4 public keys
    • Keys must be in base64 format

Key Management Best Practices

  • Key Rotation:
    • Add new keys before deleting old ones
    • Wait approximately 3 days between adding and deleting
    • Rotate keys periodically for security
  • Disabling DNSSEC:
    • Delete public keys from registrar first
    • Wait approximately 3 days
    • Then disable DNSSEC at DNS service
    • Warning: Improper disabling causes SERVFAIL errors

Critical Warnings

  • Implementation Risks:
    • Incorrect configuration can make domain inaccessible
    • Mismatched keys between DNS and registrar break resolution
    • Improper disabling sequence causes SERVFAIL errors
    • DNSSEC-aware resolvers will reject unverifiable responses

Exam Tips

  • Key Concepts:
    • DNSSEC creates a chain of trust from TLD to authoritative servers
    • Public keys are stored with registrar; private keys with DNS provider
    • Key types: KSK (Key Signing Key) and ZSK (Zone Signing Key)
    • Implementation order matters: DNS provider first, then registrar
  • Common Scenarios:
    • Enabling DNSSEC for a domain
    • Key rotation procedures
    • Troubleshooting DNSSEC validation failures
    • Disabling DNSSEC safely
  • Remember:
    • Not all TLDs support DNSSEC
    • Not all resolvers validate DNSSEC
    • Key limits vary by TLD (.com/.net: 13 keys, others: 4 keys)
    • Improper configuration can break domain resolution

DNSSEC Key Concepts and Components

DNSSEC Key Types and Their Functions

  • Zone Signing Key (ZSK):
    • Primary purpose: Signs individual DNS resource records within a zone
    • Typically rotated more frequently (every 1-3 months)
    • Usually smaller in size (1024-2048 bits) for operational efficiency
    • Creates RRSIG records for each DNS record set
    • Managed by the DNS operator/service provider
  • Key Signing Key (KSK):
    • Primary purpose: Signs the DNSKEY record set (which includes the ZSK)
    • Rotated less frequently (every 1-2 years)
    • Usually larger in size (2048-4096 bits) for stronger security
    • Used to establish the trust chain with the parent zone
    • Public portion is shared with the parent zone as a DS record
  • Combined Signing Key (CSK):
    • Alternative approach that combines KSK and ZSK functions
    • Single key that signs both the zone records and the DNSKEY record set
    • Simplifies key management but offers less flexibility
    • Typically used in smaller or less complex deployments

DNSSEC Record Types

  • DNSKEY Record:
    • Contains the public key portion of the signing keys (ZSK and KSK)
    • Published in the DNS zone
    • Used by validators to verify signatures
    • Includes flags to distinguish between KSK and ZSK
  • RRSIG (Resource Record Signature):
    • Digital signature for a specific DNS record set
    • Contains signature, expiration time, inception time, key tag
    • Created using the private portion of ZSK
    • Allows validators to verify record authenticity
  • DS (Delegation Signer) Record:
    • Cryptographic hash of the KSK's DNSKEY record
    • Published in the parent zone (e.g., .com for example.com)
    • Creates the secure delegation and trust chain
    • Contains hash algorithm, digest type, key tag, and digest
  • NSEC/NSEC3 Records:
    • Provide authenticated denial of existence
    • Prove that requested DNS records don't exist
    • NSEC: Lists next secure record name in zone
    • NSEC3: Hash-based alternative that mitigates zone enumeration

Cryptographic Algorithms

  • Common Signing Algorithms:
    • RSA/SHA-256 (Algorithm 8): Widely supported, common choice
    • ECDSA with P-256 (Algorithm 13): More efficient, smaller signatures
    • Ed25519 (Algorithm 15): Modern, high-security elliptic curve
  • Digest Algorithms (for DS records):
    • SHA-256 (Digest Type 2): Current recommended standard
    • SHA-384 (Digest Type 4): Higher security option

Trust Anchors

  • DNS Root Zone Trust Anchor:
    • Starting point for the DNSSEC validation chain
    • Managed by ICANN and distributed to DNS resolver software
    • Public KSK for the DNS root zone
    • Pre-configured in DNSSEC-validating resolvers
  • Local Trust Anchors:
    • Can be configured manually for specific zones
    • Used when parent zone is not signed
    • Creates "islands of security" in DNS

DNSSEC Key Interoperation: Example with example.com

Initial DNSSEC Setup for example.com

  • Step 1: Key Generation
    • DNS operator for example.com (e.g., Route 53) generates two key pairs:
      • KSK pair: 2048-bit RSA/SHA-256 key pair
      • ZSK pair: 1024-bit RSA/SHA-256 key pair
    • Each pair consists of:
      • Private key: Kept secure by DNS operator
      • Public key: Published in the zone's DNSKEY records
    • Example DNSKEY records in example.com zone:
      • KSK: example.com. IN DNSKEY 257 3 8 [base64-encoded public key]
      • ZSK: example.com. IN DNSKEY 256 3 8 [base64-encoded public key]

Zone Signing Process

  • Step 2: Record Signing
    • DNS operator uses the private ZSK to sign all records in example.com zone
    • For each record set (e.g., A, MX, CNAME), an RRSIG record is created
    • Example for A record:
      • Original: www.example.com. IN A 192.0.2.1
      • RRSIG: www.example.com. IN RRSIG A 8 3 3600 20230615000000 20230515000000 12345 example.com. [signature]
  • Step 3: DNSKEY Signing
    • DNS operator uses the private KSK to sign the DNSKEY record set
    • This creates an RRSIG for the DNSKEY records
    • Example:
      • example.com. IN RRSIG DNSKEY 8 2 3600 20230615000000 20230515000000 54321 example.com. [signature]

Establishing the Chain of Trust

  • Step 4: DS Record Creation and Publication
    • DNS operator creates a DS record from the KSK's DNSKEY:
      • Calculates a cryptographic hash of the KSK's DNSKEY record
      • Example: example.com. IN DS 54321 8 2 [hash value]
    • Domain owner provides this DS record to the registrar (e.g., Route 53 domain registration)
    • Registrar forwards the DS record to the TLD registry (.com operated by Verisign)
    • TLD registry publishes the DS record in the .com zone

DNSSEC Validation Process

  • Step 5: DNS Query and Validation
    • User queries for www.example.com
    • DNSSEC-aware resolver follows validation chain:
      • Starts with root zone trust anchor (pre-configured)
      • Validates .com zone using root zone signatures
      • Retrieves DS record for example.com from .com zone
      • Retrieves DNSKEY records from example.com zone
      • Validates DNSKEY using DS record from parent (.com)
      • Validates A record for www.example.com using ZSK
      • Returns validated result to user

Key Rollover Scenario

  • Step 6: ZSK Rollover (Routine Maintenance)
    • DNS operator generates new ZSK pair
    • Publishes new ZSK in DNSKEY record alongside old ZSK
    • Signs DNSKEY set with KSK
    • Gradually transitions to signing records with new ZSK
    • After transition period, removes old ZSK
  • Step 7: KSK Rollover (Less Frequent)
    • DNS operator generates new KSK pair
    • Publishes new KSK in DNSKEY record alongside old KSK
    • Signs DNSKEY set with both old and new KSK
    • Creates new DS record from new KSK
    • Submits new DS record to registrar
    • Registrar forwards to TLD registry
    • After new DS is published and propagated, removes old KSK

DNSSEC Key Ownership, Management, Creation, and Renewal

Key Ownership and Responsibilities

  • Zone Signing Key (ZSK)
    • Owner: DNS service provider (e.g., Route 53, third-party DNS provider)
    • Creation: Generated by the DNS service provider's key management system
    • Storage: Private key stored securely by DNS provider, often in HSMs (Hardware Security Modules)
    • Publication: Public key published in zone's DNSKEY records
    • Renewal: Typically every 1-3 months, managed entirely by DNS provider
    • Customer Involvement: Usually none; transparent to domain owner
  • Key Signing Key (KSK)
    • Owner: DNS service provider
    • Creation: Generated by the DNS service provider's key management system
    • Storage: Private key stored in highly secure systems, often with stronger protections than ZSK
    • Publication: Public key published in zone's DNSKEY records
    • Renewal: Typically every 1-2 years
    • Customer Involvement: May require domain owner to update DS record with registrar during KSK rollover
  • Delegation Signer (DS) Record
    • Owner: TLD registry (e.g., Verisign for .com)
    • Creation: Hash generated from KSK by DNS provider
    • Submission: Domain owner or DNS provider submits to domain registrar
    • Publication: Published in parent zone by TLD registry
    • Renewal: Updated whenever KSK changes
    • Customer Involvement: May need to submit new DS record to registrar during KSK rollover

Key Management Models

  • Fully Managed (e.g., Route 53 DNSSEC Signing)
    • DNS provider handles all key generation, storage, and rotation
    • Provider manages communication with registrar for DS records
    • Customer simply enables DNSSEC with minimal configuration
    • Example: Route 53 DNSSEC signing with integrated registrar
    • Customer involvement: One-click enablement, no manual key management
  • Partially Managed
    • DNS provider manages ZSK and KSK generation and rotation
    • Customer responsible for updating DS records with registrar
    • DNS provider generates and provides DS record information
    • Example: Route 53 DNSSEC signing with external registrar
    • Customer involvement: Updating DS records during initial setup and KSK rollovers
  • Self-Managed
    • Customer manages own DNSSEC implementation (e.g., BIND, PowerDNS)
    • Customer generates and manages all keys
    • Customer handles all key rotation procedures
    • Customer submits DS records to registrar
    • Example: On-premises DNS servers or custom cloud deployments
    • Customer involvement: Complete management of entire DNSSEC process

Key Creation and Generation

  • Key Generation Methods
    • Automated Systems: Most DNS providers use automated systems to generate keys
    • Hardware Security Modules (HSMs): High-security environments use specialized hardware
    • Software-based: BIND's dnssec-keygen, OpenSSL, or similar tools for self-managed scenarios
    • Parameters: Algorithm selection, key size, validity period, flags (KSK vs ZSK)
  • Route 53 Implementation
    • Uses AWS Key Management Service (KMS) for key generation and protection
    • Customer-managed customer master key (CMK) for signing
    • Automated key generation when DNSSEC is enabled
    • Keys never exposed to customers or AWS operators
    • Automatic ZSK rotation every 90 days
    • Managed KSK rotation approximately every year

Key Renewal and Rotation Processes

  • ZSK Rotation (Pre-Publication Method)
    • Who: DNS service provider
    • How:
      • Generate new ZSK
      • Add new ZSK to DNSKEY RRset (alongside existing ZSK)
      • Sign DNSKEY RRset with KSK
      • Wait for propagation (TTL period)
      • Begin signing zone records with new ZSK
      • Wait for propagation
      • Remove old ZSK from DNSKEY RRset
    • Customer Involvement: None in managed services
  • KSK Rotation (Double-DS Method)
    • Who: DNS service provider with domain owner/registrar involvement
    • How:
      • Generate new KSK
      • Add new KSK to DNSKEY RRset
      • Sign DNSKEY RRset with both old and new KSK
      • Generate new DS record from new KSK
      • Submit new DS record to registrar (alongside existing DS)
      • Wait for DS propagation in parent zone
      • Remove old KSK from DNSKEY RRset
      • Remove old DS record from parent zone
    • Customer Involvement: May need to submit new DS record to registrar and later remove old DS

Emergency Key Management

  • Key Compromise Scenarios
    • ZSK Compromise:
      • Emergency ZSK rotation by DNS provider
      • No registrar involvement needed
      • Minimal customer involvement
    • KSK Compromise:
      • Emergency KSK rotation
      • Urgent DS record update at registrar
      • May require customer involvement for DS update
    • Complete Compromise:
      • Temporarily disable DNSSEC by removing DS record
      • Generate all new keys
      • Re-establish DNSSEC with new keys
      • Requires customer involvement with registrar

Route 53 DNSSEC Implementation

Route 53 DNSSEC Architecture

  • Key Management Responsibilities:
    • Route 53 manages Zone Signing Keys (ZSKs) automatically
    • Customer manages Key Signing Keys (KSKs) through AWS KMS
    • Route 53 handles all signing operations transparently
    • Customer responsible for establishing chain of trust
  • AWS KMS Integration:
    • KSKs are based on AWS KMS customer managed keys
    • Must use asymmetric KMS keys with ECC_NIST_P256 key spec
    • KMS keys must be in US East (N. Virginia) Region
    • Requires specific key policy permissions for Route 53 access
  • Signing Operations:
    • Route 53 performs online signing (not offline batch signing)
    • ZSK signs all zone records
    • KSK signs only the DNSKEY record set
    • RRSIG records refreshed every 1-7 days
    • 15 RRSIGs generated per refresh for redundancy

Implementation Constraints

  • TTL Limitations:
    • Maximum TTL enforced at one week (604,800 seconds)
    • Longer TTLs are automatically capped
    • Applies only to DNSSEC-enabled zones
  • Configuration Restrictions:
    • Multi-vendor configurations not supported
    • White-label name servers must be from single provider
    • Parent zone must support DS records
    • Maximum two KSKs per hosted zone
  • Monitoring Requirements:
    • CloudWatch alarms strongly recommended
    • Must monitor for DNSSECInternalFailure errors
    • Must monitor for DNSSECKeySigningKeysNeedingAction errors
    • Quick resolution required to prevent outages

ZSK Management by Route 53

  • ZSK Generation:
    • One active ZSK per DNSSEC-enabled zone
    • Generated and managed entirely by Route 53
    • Uses ECDSAP256SHA256 algorithm
    • Customer has no direct access or control
  • ZSK Rotation:
    • Automatic rotation every 7-30 days
    • Uses Pre-Publish Key Rollover method
    • Introduces new ZSK before removing old one
    • Suspended if any KSK is in ACTION_NEEDED status
    • Resumes automatically when issues resolved

Proof of Nonexistence Implementation

  • NSEC Implementation:
    • Uses BL method for compact representation
    • Prevents zone walking (security enhancement)
    • Returns minimal NSEC record for type nonexistence
    • No NSEC record for wildcard synthesis
  • Online Signing Advantage:
    • Generates response-specific RRSIGs
    • Prevents RRSIG reuse for spoofing
    • More efficient than traditional offline signing

Implementation Process

  • Preparation Phase:
    • Monitor zone availability
    • Lower zone's maximum TTL (recommended: 1 hour)
    • Lower SOA TTL and minimum field
    • Verify changes are propagated
  • Enabling Phase:
    • Create KSK (or use existing one)
    • Enable DNSSEC signing
    • Wait for previous zone's maximum TTL
    • Monitor for customer issues (2 weeks recommended)
  • Chain of Trust Phase:
    • Create DS record from KSK information
    • Add DS record to parent zone
    • Set DS TTL to 5 minutes if possible
    • Wait for maximum NS TTL
    • Monitor name resolution (2 weeks recommended)

Troubleshooting DNSSEC Issues

  • KSK Status: ACTION_NEEDED
    • Cause: Route 53 lost access to AWS KMS key
    • Risk: Will cause zone outage for validating resolvers
    • Solution: Verify KMS key is enabled with correct permissions
    • Prevention: Set up CloudWatch alarms
  • KSK Status: INTERNAL_FAILURE
    • Cause: System error in DNSSEC operations
    • Impact: Blocks all other DNSSEC operations
    • Solution: Retry activating/deactivating KSK
    • Urgency: Must be resolved promptly
  • Disabling DNSSEC Issues:
    • Route 53 checks if zone is in chain of trust
    • Verifies parent zone NS and DS records
    • May fail if zone is not publicly resolvable
    • Solution: Contact parent zone administrator
