AWS Cloud WAN Study Guide - Part 1: Core Concepts

Introduction to AWS Cloud WAN

  • Definition:
    • Managed wide-area networking (WAN) service
    • Builds, manages, and monitors unified global networks
    • Connects cloud and on-premises resources
    • Provides central dashboard for network management
  • Key Benefits:
    • Centralized network management
    • Policy-based configuration
    • Automated security tasks
    • Complete visibility of global network

Key Concepts and Terminology

  • Global Network:
    • Single, private network container
    • High-level container for network objects
    • Can contain Transit Gateways and core networks
    • Visible in Network Manager console
  • Core Network:
    • Part of global network managed by AWS
    • Includes regional connection points and attachments
    • Operates in regions defined in policy document
    • Limited to one core network per global network
  • Core Network Policy:
    • Declarative document applied to core network
    • Defines segments, routing, and attachment mapping
    • Captures intent and deploys automatically
    • Supports multiple versions (only one active)

Network Components

  • Attachments:
    • Connections or resources added to core network
    • Includes VPCs, VPNs, Transit Gateway attachments, Connect attachments
    • Each attachment connects to one segment
  • Core Network Edge:
    • Regional connection point managed by AWS
    • Created in each region defined in policy
    • Also known as AWS Transit Gateway
    • Forms full-mesh peering with other edges
  • Network Segments:
    • Dedicated routing domains
    • Similar to globally consistent VRF tables
    • Provides network isolation
    • By default, only attachments in same segment can communicate

Advanced Features

  • Segment Actions and Attachment Policies:
    • Define routing between segments
    • Map attachments to segments via tags
    • Control route sharing between segments
    • Configure centralized internet routing
  • Ownership Models:
    • Core Network Owner: Full control and visibility
    • Attachment Owner: Limited to creating connections and attachments
    • Sharing via AWS Resource Access Manager
  • Additional Capabilities:
    • Peering with Transit Gateways
    • AWS PrivateLink support (limited regions)
    • IPv6 support on dual-stack endpoints
    • Home Region concept (US West - Oregon)

Exam Tips

  • Key Limitations:
    • One core network per global network
    • Home Region fixed to US West (Oregon)
    • PrivateLink only in us-west-2 and us-gov-west-1
    • IPv6 only on dual-stack endpoints
  • Important Distinctions:
    • Global Network vs. Core Network roles
    • Core Network Owner vs. Attachment Owner permissions
    • Segments vs. Traditional VRFs
    • Policy-based vs. Manual configuration
  • Remember:
    • Cloud WAN aggregates data to US West (Oregon)
    • Uses Amazon Location Service for maps
    • Available in 30+ AWS Regions
    • Segments provide isolation by default

AWS Cloud WAN Study Guide - Part 3: Core Network Policies and Segments

Core Network Policy Fundamentals

  • Policy Definition:
    • Declarative JSON document
    • Captures network intent
    • Automatically deployed by AWS
    • Central to Cloud WAN operation
  • Policy Components:
    • Core network configuration
    • Segment definitions
    • Attachment policies
    • Segment actions (routing rules)
    • Edge locations (AWS Regions)

Network Segments

  • Segment Characteristics:
    • Dedicated routing domains
    • Isolated by default
    • Similar to VRF tables or Layer 3 IP VPNs
    • Limit of 100 segments per core network
  • Segment Use Cases:
    • Business unit isolation (e.g., retail vs. IT)
    • Environment separation (e.g., dev, test, prod)
    • Compliance requirements (e.g., PCI, HIPAA)
    • Shared services segments

Policy Versioning

  • Version Management:
    • Multiple versions can exist simultaneously
    • Only one version is LIVE at a time
    • Previous versions can be restored
    • Changes applied through updated JSON policy
  • Policy States:
    • PENDING_ATTACHMENT_ACCEPTANCE: Waiting for approval
    • READY: Validated but not yet applied
    • FAILED: Validation or execution failed
    • EXECUTING: Currently being applied
    • LIVE: Active and deployed

Attachment Policies

  • Attachment Mapping:
    • Explicit mapping: Direct resource-to-segment assignment
    • Tag-based mapping: Using resource tags
    • Automatic assignment based on criteria
  • Attachment Requirements:
    • Each attachment connects to exactly one segment
    • Approval can be required for segment access
    • Route filters can be applied before attachment

Segment Actions

  • Action Types:
    • Route sharing between segments
    • Shared service access configuration
    • Centralized internet routing
    • Inter-segment traffic through firewalls
  • Action Configuration:
    • Defined in core network policy
    • Applied consistently across all regions
    • Can be unidirectional or bidirectional
    • Supports complex routing patterns

Policy Creation Methods

  • Console Visual Editor:
    • Graphical interface for policy creation
    • Guided workflow for segments and attachments
    • Validation during creation
  • JSON Editor:
    • Direct JSON policy editing
    • More flexibility for complex configurations
    • Supports advanced policy features
    • Requires understanding of policy structure

Exam Tips

  • Policy Structure:
    • Core network policy is declarative, not imperative
    • Changes require new policy version
    • Policy deployment can take time
    • Understand policy states and lifecycle
  • Segment Design:
    • Default isolation between segments
    • Explicit sharing required for communication
    • Segment actions control routing behavior
    • Consistent configuration across regions
  • Common Scenarios:
    • Creating shared services segments
    • Implementing network isolation
    • Configuring centralized internet access
    • Managing policy versions

AWS Cloud WAN Study Guide - Part 4: Attachments and Connectivity Options

Attachment Types

  • VPC Attachments:
    • Connect Amazon VPCs to core network
    • Support both IPv4 and IPv6 (dual-stack)
    • Can be tagged for automatic segment assignment
    • Supports multiple subnets across AZs
  • Site-to-Site VPN Attachments:
    • Connect on-premises networks via IPsec VPN
    • Support BGP for dynamic routing
    • Provide encrypted connectivity
    • Require customer gateway device
  • Connect Attachments:
    • Support third-party SD-WAN appliances
    • Include tunnel-less and GRE options
    • Enable software-defined connectivity
    • Integrate with partner solutions
  • Direct Connect Attachments:
    • Dedicated private connections
    • Higher bandwidth and consistent performance
    • Lower latency than internet-based connections
    • Support for transit virtual interfaces
  • Transit Gateway Route Table Attachments:
    • Connect existing transit gateway infrastructure
    • Enable peering between core network and transit gateways
    • Support end-to-end network segmentation
    • Require peering connection

Attachment States and Lifecycle

  • Creation States:
    • CREATING: Initial attachment creation
    • PENDING_ATTACHMENT_ACCEPTANCE: Waiting for approval
    • AVAILABLE: Successfully created and ready
    • FAILED: Creation process failed
  • Operational States:
    • AVAILABLE: Attachment is operational
    • PENDING_NETWORK_UPDATE: Changes being applied
    • DELETING: Attachment is being removed
    • DELETED: Attachment has been removed

Attachment Tags

  • Tag-Based Mapping:
    • Key-value pairs assigned to attachments
    • Used for automatic segment assignment
    • Defined in core network policy
    • Simplifies attachment management at scale
  • Tag Configuration:
    • Applied during attachment creation
    • Can be modified after creation
    • Policy evaluates tags for segment mapping
    • Supports complex mapping rules

Peering Connections

  • Transit Gateway Peering:
    • Interconnects core network edge with transit gateway
    • Must be in same AWS Region
    • Supports dynamic routing
    • Enables end-to-end segmentation
  • Peering Configuration:
    • Created from core network dashboard
    • Requires transit gateway ID
    • Route table attachments created over peering
    • Supports multiple route tables

Shared Attachments

  • Resource Sharing:
    • Uses AWS Resource Access Manager (RAM)
    • Supports VPC attachments
    • Supports transit gateway route table attachments
    • Enables multi-account architectures
  • Sharing Process:
    • Core network owner initiates sharing
    • Recipient account accepts shared resource
    • Permissions limited to attachment operations
    • Can be shared with accounts or organizations

AWS PrivateLink Integration

  • PrivateLink Features:
    • Private connectivity between Cloud WAN and VPCs
    • No internet gateway or NAT required
    • Secure communication path
    • Separate cost structure from Cloud WAN
  • Limitations:
    • Only supports IPv6 dual-stack endpoints
    • Available only in us-west-2 and us-gov-west-1
    • Requires specific configuration

Exam Tips

  • Attachment Selection:
    • Choose attachment type based on connectivity requirements
    • Understand attachment state lifecycle
    • Consider bandwidth, latency, and security needs
    • Know which attachments support sharing
  • Integration Points:
    • Transit Gateway peering requirements
    • PrivateLink regional limitations
    • Connect attachment partner integrations
    • Direct Connect virtual interface requirements
  • Common Scenarios:
    • Hybrid connectivity implementation
    • Multi-account network architecture
    • Tag-based segment assignment
    • Integrating existing transit gateway networks

AWS Cloud WAN Study Guide for ANS-C01 - Part 1: Core Components and Architecture

Introduction to AWS Cloud WAN

  • Definition and Purpose:
    • Fully managed service that creates a unified global network for all your AWS and on-premises locations
    • Eliminates manual configuration of complex connectivity between VPCs, AWS Regions, and on-premises networks
    • Launched in July 2022 as a generally available service
    • Built on top of AWS Transit Gateway technology but with global management capabilities
  • Specific Use Cases:
    • Connecting 100+ VPCs across multiple regions with consistent policy
    • Implementing global network segmentation with different security domains
    • Centralizing egress traffic through security inspection points
    • Simplifying SD-WAN integration with AWS infrastructure

Core Components - Exact Definitions

  • Global Network:
    • Container object that holds all network resources
    • Can contain exactly ONE core network and multiple transit gateways
    • Multiple global networks can exist in a single AWS account
    • Created instantly (unlike core network which takes up to 30 minutes)
  • Core Network:
    • AWS-managed infrastructure that provides connectivity between attachments
    • Consists of regional "edges" (based on Transit Gateway technology)
    • Takes exactly 30 minutes to deploy initially
    • Hard limit of ONE core network per global network
    • Requires at least ONE edge location (AWS Region) to be specified
  • Core Network Edge:
    • Regional connection point for attachments
    • Created in each AWS Region specified in the core network policy
    • Automatically forms full-mesh connectivity with other edges
    • Based on AWS Transit Gateway technology (but managed by AWS)
    • Each edge has a specific capacity and scaling properties

Network Segments - Specific Details

  • Segment Definition:
    • Isolated routing domains similar to VRFs (Virtual Routing and Forwarding)
    • Hard limit of 100 segments per core network
    • Each attachment connects to exactly ONE segment
    • Segments are defined in the core network policy
    • Segments are global (consistent across all regions)
  • Segment Isolation:
    • By default, segments are completely isolated from each other
    • Resources in different segments cannot communicate without explicit segment actions
    • Segment isolation is enforced consistently across all regions
    • Segment names are case-sensitive and must be unique within a core network
  • Common Segment Examples:
    • "Production", "Development", "Test" for environment separation
    • "Shared", "NonProduction", "Production" for service-based segmentation
    • "PCI", "NonPCI" for compliance-based segmentation
    • "Egress" for centralized internet access

Attachment Types - Specific Capabilities

  • VPC Attachments:
    • Requires subnet allocation in EACH Availability Zone where connectivity is needed
    • Supports both IPv4 and IPv6 (dual-stack) addressing
    • Appliance mode option available for symmetric routing
    • Maximum of 5 Gbps per flow (same as Transit Gateway)
    • Supports automatic route propagation to core network
  • Site-to-Site VPN Attachments:
    • Maximum throughput of 1.25 Gbps per tunnel
    • Supports BGP for dynamic routing
    • Requires customer gateway device information (IP address and ASN)
    • Supports both static and dynamic routing options
    • Creates two tunnels for redundancy
  • Connect Attachments:
    • Supports GRE tunnels with maximum MTU of 8500 bytes
    • Supports tunnel-less connections for higher throughput
    • Requires third-party SD-WAN device compatibility
    • Connect peers must be created after attachment
    • Supports BGP for dynamic routing
  • Transit Gateway Route Table Attachments:
    • Must be in the same AWS Region as the core network edge
    • Requires existing peering connection
    • Maps to exactly one segment in the core network
    • Routes are exchanged between transit gateway route table and core network segment
    • Transit gateway cannot be part of another core network

Critical Limitations - Exact Numbers

  • Hard Limits:
    • 1 core network per global network
    • 100 segments per core network
    • 5 Gbps per flow (same as Transit Gateway)
    • 30 minute deployment time for core network
    • 500 attachments per core network (default, can be increased)
    • 5000 routes per segment (default, can be increased)
  • ASN Ranges:
    • Private ASN range: 64512-65534
    • 4-byte ASN range: 4200000000-4294967294
    • Default ASN assignment if not specified
    • Cannot be changed after core network creation

AWS Cloud WAN Study Guide for ANS-C01 - Part 2: Core Network Policy

Core Network Policy Structure

  • Policy Format:
    • JSON document with specific required sections
    • Must include "core-network-configuration" section
    • Must include at least one segment definition
    • Must specify at least one AWS Region as edge location
    • Case-sensitive throughout the entire document
  • Required Policy Elements:
    • Edge locations: List of AWS Regions where core network edges will be deployed
    • ASN ranges: Optional, defaults if not specified
    • Segments: At least one segment must be defined
    • Segment names: Case-sensitive, alphanumeric with hyphens, no spaces
    • Attachment policies: Rules for mapping attachments to segments
  • Example Policy Structure: 
    	{
	  "core-network-configuration": {
		"asn-ranges": ["64512-65534"],
		"edge-locations": [
		  { "location": "us-east-1" },
		  { "location": "us-west-2" }
		],
		"vpn-ecmp-support": true
	  },
	  "segments": [
		{ "name": "Production", "description": "Production segment" },
		{ "name": "Development", "description": "Development segment" }
	  ],
	  "segment-actions": [...],
	  "attachment-policies": [...]
	}

Policy Versioning and States

  • Version Management:
    • Each policy change creates a new numbered version
    • Only ONE policy version can be LIVE at any time
    • Previous versions are preserved and can be restored
    • Policy versions are immutable once created
    • New version required for ANY policy change
  • Exact Policy States:
    • PENDING_ATTACHMENT_ACCEPTANCE: Waiting for attachment approvals
    • READY: Validated but not yet applied to network
    • EXECUTING: Currently being applied to network (can take several minutes)
    • LIVE: Active and controlling network behavior
    • FAILED: Validation or execution errors occurred
  • Change Set States:
    • NOT_STARTED: Change set not yet created
    • PENDING: Change set created but not executed
    • IN_PROGRESS: Change set being applied
    • FAILED: Change set application failed
    • SUCCEEDED: Change set successfully applied

Attachment Policies - Exact Configuration

  • Tag-Based Mapping:
    • Maps attachments to segments based on resource tags
    • Tags are key-value pairs (both case-sensitive)
    • Example tag: "Environment": "Production"
    • Tags must match EXACTLY as defined in policy
    • Multiple tags can be evaluated with AND/OR logic
  • Attachment Policy Structure: 
    	"attachment-policies": [
	  {
		"rule-number": 100,
		"condition-logic": "and",
		"conditions": [
		  {
			"type": "tag-value",
			"key": "Environment",
			"value": "Production"
		  }
		],
		"action": {
		  "association-method": "constant",
		  "segment": "Production"
		}
	  }
	]
  • Rule Processing:
    • Rules are processed in order of rule-number (lowest first)
    • First matching rule determines segment assignment
    • If no rules match, attachment remains unassigned
    • Rule numbers must be unique within policy
    • Rule numbers typically use increments of 100 (100, 200, 300)

Segment Actions - Exact Configuration

  • Action Types:
    • create-route: Creates routes between segments
    • share-attachment-routes: Shares routes from attachments
    • share-segment: Shares entire segment with another
    • action-type determines behavior of segment action
  • Segment Action Structure: 
    	"segment-actions": [
	  {
		"action": "create-route",
		"description": "Share Shared segment with Production",
		"destination-cidr-blocks": ["0.0.0.0/0"],
		"destinations": ["Production"],
		"segment": "Shared"
	  }
	]
  • Routing Behavior:
    • Unidirectional by default (source to destination)
    • Bidirectional requires two separate actions
    • Can specify exact CIDR blocks or use "0.0.0.0/0" for all IPv4
    • Can specify "::/0" for all IPv6 routes
    • Actions apply consistently across all regions
  • Common Segment Action Patterns:
    • Shared Services: Allow specific segments to access shared segment
    • Centralized Egress: Allow segments to access egress segment
    • Segmentation: Keep segments isolated by default
    • Hub-and-Spoke: Central segment connects to all others

Policy Deployment Process

  • Exact Deployment Steps:
    • 1. Create new policy version (JSON document)
    • 2. Submit policy for validation (moves to READY state)
    • 3. Generate change set to review changes
    • 4. Execute policy (moves to EXECUTING state)
    • 5. Wait for deployment completion (moves to LIVE state)
  • Deployment Timing:
    • Initial core network creation: 30 minutes
    • Policy updates: 5-15 minutes depending on complexity
    • Adding new edge locations: 15-20 minutes
    • Simple segment changes: 5-10 minutes
    • No attachments can be created during EXECUTING state
  • Policy Update Restrictions:
    • Cannot delete segments that have attachments
    • Cannot remove edge locations that have attachments
    • Cannot modify ASN ranges after initial creation
    • Cannot have multiple policies in EXECUTING state
    • Must wait for LIVE state before creating attachments

Exam Tips for Core Network Policy

  • Remember These Specifics:
    • Policy is declarative (describes desired state) not imperative
    • Case sensitivity matters throughout policy document
    • Tag-based mapping requires exact matches
    • Segment actions are unidirectional by default
    • Policy changes require new version and full deployment
    • Only one policy can be LIVE at a time
  • Common Policy Scenarios:
    • Adding new AWS Region to existing core network
    • Creating new segment for new business unit
    • Modifying segment actions to allow new communication paths
    • Adding new attachment policies for automatic segment assignment
    • Troubleshooting failed policy deployment
