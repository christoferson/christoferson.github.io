AWS Route 53 Study Guide - Part 1: Core Concepts

Introduction to Route 53

  • Definition:
    • Highly available and scalable Domain Name System (DNS) web service
    • Performs three main functions: domain registration, DNS routing, health checking

Key Components and Terminology

  • Domain Registration Components:
    • Domain Name: Human-readable website address (e.g., example.com)
    • Domain Registrar: ICANN-accredited company processing registrations
    • Domain Registry: Company owning rights to specific TLDs
    • Top-level Domain (TLD): Last part of domain (.com, .org, etc.)
  • DNS Components:
    • Hosted Zone: Container for DNS records
    • DNS Record: Defines how to route traffic
    • Name Servers: Translate domain names to IP addresses
    • TTL (Time to Live): Cache duration for DNS records

Architecture and Operation

  • Control Plane vs Data Plane:
    • Control Plane: Located in us-east-1
      • Manages DNS entries
      • Handles API operations
      • Failover to us-west-2 if impaired
    • Data Plane: Globally distributed
      • 200+ Points of Presence (PoP)
      • Handles DNS queries
      • Optimized for availability

DNS Resolution Process

  • Step-by-Step Flow:
    • 1. User enters domain name in browser
    • 2. Request goes to DNS resolver (ISP)
    • 3. Resolver queries root name servers
    • 4. TLD servers provide Route 53 name servers
    • 5. Route 53 provides IP address
    • 6. Browser connects to web server

Exam Tips

  • Key Points to Remember:
    • Route 53 is both a registrar and DNS service
    • Control plane in us-east-1, data plane global
    • DNS resolution follows specific hierarchy
    • Automatic creation of hosted zone upon domain registration
  • Common Exam Scenarios:
    • Understanding DNS resolution flow
    • Distinguishing between control and data plane
    • Identifying components in DNS architecture
    • Understanding high availability aspects

AWS Route 53 Study Guide - Part 2: Routing Policies and Record Types

Record Types

  • Standard DNS Records:
    • A Record: Maps domain to IPv4 address
    • AAAA Record: Maps domain to IPv6 address
    • CNAME: Maps domain to another domain
    • MX: Defines mail servers
    • TXT: Text records for various purposes
  • Route 53 Specific:
    • Alias Records:
      • AWS-specific feature
      • Maps to AWS resources (S3, CloudFront, ALB)
      • No additional charge for Alias queries
      • Automatically updates if target IP changes

Routing Policies

  • Simple Routing:
    • Basic routing to single resource
    • Multiple values return random selection
    • Cannot be associated with health checks
  • Weighted Routing:
    • Route traffic based on assigned weights
    • Percentage of traffic = (Weight for a record / Sum of all weights)
    • Use cases: A/B testing, gradual migrations
  • Latency-based Routing:
    • Routes based on lowest latency for end user
    • Uses AWS Region latency measurements
    • Requires creating records in multiple regions
  • Failover Routing:
    • Active-passive failover configuration
    • Requires health check on primary resource
    • Automatically routes to secondary if primary fails
  • Geolocation Routing:
    • Routes based on user location
    • Can specify continents, countries, or US states
    • Requires default record for non-matched locations
  • Geoproximity Routing:
    • Routes based on geographic location of resources
    • Can shift traffic with bias values (-99 to +99)
    • Requires Route 53 Traffic Flow
  • IP-based Routing:
    • Routes based on client IP addresses
    • Uses CIDR blocks to define IP ranges
    • Requires CIDR location mapping
  • Multivalue Answer:
    • Returns multiple healthy records randomly
    • Can be associated with health checks
    • Returns up to 8 healthy records

Key Considerations

  • Alias vs CNAME:
    • Alias is free, CNAME is charged
    • Alias can be used at zone apex, CNAME cannot
    • Alias automatically updates with IP changes
  • Health Checks Integration:
    • Not all routing policies support health checks
    • Health checks are separate from records
    • Additional charges for health checks

Exam Tips

  • Common Scenarios:
    • Choosing between routing policies for specific use cases
    • Understanding when to use Alias vs CNAME
    • Identifying appropriate health check configurations
    • Calculating weighted routing distributions
  • Remember:
    • Only Alias records can point to AWS resources
    • Geolocation ≠ Geoproximity
    • Simple routing doesn't support health checks
    • Weighted routing requires all weights to calculate percentage

AWS Route 53 Study Guide - Part 3: Health Checks and DNS Failover

Health Check Fundamentals

  • Types of Health Checks:
    • Endpoint Monitoring:
      • Monitor specific IP address or domain
      • Supports HTTP, HTTPS, TCP protocols
      • Can monitor external endpoints
    • Status of Other Health Checks:
      • Calculated health checks
      • Monitor multiple endpoints
      • Useful for complex failover scenarios
    • State of CloudWatch Alarm:
      • Monitor CloudWatch metrics
      • Broader monitoring capabilities
      • Custom metric integration

Health Check Configuration

  • Key Parameters:
    • Request Interval:
      • Standard: 30 seconds
      • Fast: 10 seconds (higher cost)
    • Failure Threshold:
      • Number of consecutive failures before marking unhealthy
      • Default: 3 consecutive failures
    • String Matching:
      • Search for specific string in response
      • First 5120 bytes checked
  • Monitoring Options:
    • SNS Notifications for status changes
    • CloudWatch metrics integration
    • Latency measurements

DNS Failover Configuration

  • Active-Passive Failover:
    • Primary and backup resources
    • Automatic failover when primary is unhealthy
    • Uses failover routing policy
  • Active-Active Failover:
    • All resources active simultaneously
    • Uses multiple routing policies
    • Resources can be weighted
  • Private Endpoint Monitoring:
    • Requires CloudWatch metric
    • Use CloudWatch alarm health check
    • Useful for internal resources

Health Check Process

  • Operation Flow:
    • 1. Health checkers send requests at configured interval
    • 2. Response evaluated against configured criteria
    • 3. Status updated based on failure threshold
    • 4. DNS responses updated based on health status
  • Global Perspective:
    • Health checks from multiple regions
    • Minimum 3 regions must report healthy
    • Global status determination

Integration with Other Services

  • CloudWatch Integration:
    • Health check metrics published to CloudWatch
    • Custom alarms possible
    • Automated actions via CloudWatch
  • SNS Integration:
    • Notifications for status changes
    • Multiple notification endpoints
    • Custom message formats

Exam Tips

  • Key Concepts to Remember:
    • Health checks are separate from DNS records
    • Minimum 3 regions must report healthy
    • Fast interval (10s) vs Standard interval (30s)
    • Failure threshold affects failover time
  • Common Scenarios:
    • Setting up failover for web applications
    • Monitoring private endpoints
    • Implementing complex health check logic
    • Integrating with CloudWatch metrics

AWS Route 53 Study Guide - Part 4: Route 53 Resolver and DNS Firewall

Route 53 Resolver Overview

  • Core Functionality:
    • Recursive DNS service for VPCs
    • Handles hybrid DNS resolution
    • Supports both inbound and outbound DNS queries
    • Regional service - runs independently in each AWS Region
  • Components:
    • Resolver Endpoints:
      • Inbound: Allows on-premises to resolve AWS DNS names
      • Outbound: Allows VPCs to resolve on-premises DNS names
      • Each endpoint requires 2+ IP addresses in different AZs
    • Resolver Rules:
      • Forward rules: Direct queries to custom DNS servers
      • System rules: Default AWS resolution behavior
      • Recursive lookups: Handle standard DNS resolution

Route 53 Resolver DNS Firewall

  • Key Features:
    • Domain List Management:
      • Allow lists: Permitted domains
      • Block lists: Denied domains
      • Custom domain lists
    • Rule Groups:
      • Collections of firewall rules
      • Priority-based processing
      • Association with VPCs
  • Response Actions:
    • ALLOW: Permit the DNS query
    • BLOCK: Deny the DNS query
    • ALERT: Allow but log the query
    • Custom response: Return specific DNS response

Resolver on AWS Outposts

  • Capabilities:
    • Local DNS resolution on Outposts racks
    • Integration with on-premises DNS
    • Reduced latency for local queries
  • Configuration:
    • Requires Resolver endpoints
    • Uses existing Resolver rules
    • Supports hybrid DNS architecture

Route 53 Resolver Query Logging

  • Logging Features:
    • Log DNS queries from VPCs
    • Integration with CloudWatch Logs
    • Supports multiple destinations
    • Includes query metadata
  • Log Contents:
    • Query timestamp
    • Domain name requested
    • Query type
    • Response code
    • Source IP address

Security and Access Control

  • IAM Integration:
    • Resource-based policies
    • Service-linked roles
    • Cross-account access
  • Network Security:
    • Security group configuration
    • VPC endpoint policies
    • Network ACL considerations

Exam Tips

  • Key Points:
    • Resolver endpoints require 2+ IP addresses in different AZs
    • DNS Firewall rules process in priority order
    • Resolver is a regional service
    • Query logging supports multiple destinations
  • Common Scenarios:
    • Hybrid DNS resolution setup
    • DNS security implementation
    • Cross-account DNS management
    • DNS query monitoring and logging
  • Design Considerations:
    • High availability requirements
    • Network security integration
    • Performance optimization
    • Cost implications

AWS Route 53 Study Guide - Part 5: Traffic Flow and Route 53 Profiles

Traffic Flow Overview

  • Purpose:
    • Visual traffic management tool
    • Complex routing configuration management
    • Global traffic optimization
    • Policy version control
  • Key Components:
    • Traffic Policy:
      • Collection of routing rules
      • Visual editor for configuration
      • Support for multiple versions
      • Reusable across multiple domains
    • Policy Records:
      • DNS records created from traffic policies
      • Associates policy with domain/subdomain
      • Supports multiple endpoints

Traffic Flow Rules and Configurations

  • Rule Types:
    • Geoproximity:
      • Based on geographic distance
      • Bias adjustments (-99 to +99)
      • AWS Region or custom coordinates
    • Weighted:
      • Proportional traffic distribution
      • Percentage-based routing
      • A/B testing capabilities
    • Latency:
      • Based on network latency
      • Region-specific routing
      • Automatic measurements

Route 53 Profiles

  • Core Features:
    • Centralized DNS Management:
      • Multi-VPC configurations
      • Cross-account management
      • Standardized DNS settings
    • Profile Components:
      • DNS settings templates
      • VPC associations
      • Rule configurations
  • Profile Management:
    • Version Control:
      • Multiple profile versions
      • Rollback capabilities
      • Change tracking
    • Access Control:
      • IAM integration
      • Resource sharing
      • Permission management

Implementation Best Practices

  • Traffic Flow:
    • Start with simple policies
    • Test configurations before deployment
    • Use version control effectively
    • Monitor policy performance
  • Profiles:
    • Standardize DNS configurations
    • Implement least privilege access
    • Regular profile reviews
    • Document profile changes

Cost Considerations

  • Traffic Flow Pricing:
    • Policy creation charges
    • Monthly charges per policy record
    • Version management costs
  • Profile Costs:
    • Based on number of associated VPCs
    • Cross-account configurations
    • Rule evaluation charges

Exam Tips

  • Key Concepts:
    • Traffic Flow is a visual editor for complex routing
    • Profiles standardize DNS across accounts/VPCs
    • Version control is built into both features
    • Geoproximity requires Traffic Flow
  • Common Scenarios:
    • Global application deployment
    • Multi-region failover
    • Cross-account DNS management
    • Standardized DNS configurations
  • Remember:
    • Traffic Flow policies are reusable
    • Profiles work across multiple VPCs
    • Bias values range from -99 to +99
    • Changes require version updates

AWS Route 53 - Comprehensive Study Guide Summary

1. Core Architecture (Must Know)

  • Service Structure:
    • Control Plane: us-east-1 (failover to us-west-2)
    • Data Plane: 200+ global Points of Presence
    • Regional Service: Resolver components
  • Critical Numbers:
    • Health Check Intervals: 30s standard, 10s fast
    • Health Check Threshold: Minimum 3 regions must report healthy
    • Resolver Endpoints: Minimum 2 IP addresses in different AZs
    • Geoproximity Bias: -99 to +99

2. Routing Policies (Frequently Tested)

  • Key Differentiators:
    • Simple: Single resource, no health checks
    • Weighted: Specified percentages (weights/sum of weights)
    • Latency: Based on AWS Region measurements
    • Failover: Requires health checks for primary
    • Geolocation: Based on user location (continent/country/state)
    • Geoproximity: Based on resource location (requires Traffic Flow)
    • IP-based: Uses CIDR blocks
    • Multivalue: Returns up to 8 healthy records

3. DNS Record Types (Common Exam Scenarios)

  • Critical Distinctions:
    • Alias vs CNAME:
      • Alias: Free, works at zone apex, AWS-specific
      • CNAME: Charged, cannot use at apex, standard DNS
    • A vs AAAA:
      • A: IPv4 addresses
      • AAAA: IPv6 addresses

4. Health Checks and Failover (Key Concepts)

  • Health Check Types:
    • Endpoint Monitoring (HTTP, HTTPS, TCP)
    • Calculated Health Checks (monitor other health checks)
    • CloudWatch Alarm Status
  • Failover Configurations:
    • Active-Passive: Primary with health check, automatic failover
    • Active-Active: All resources active, weighted distribution

5. Resolver and DNS Firewall (Advanced Features)

  • Resolver Components:
    • Inbound Endpoints: On-premises → AWS resolution
    • Outbound Endpoints: AWS → On-premises resolution
    • Rules: Forward, System, Recursive
  • DNS Firewall Actions:
    • ALLOW: Permit query
    • BLOCK: Deny query
    • ALERT: Allow but log

6. Common Exam Scenarios

  • High Availability:
    • Multiple routing policies with health checks
    • Cross-Region failover configurations
    • Resolver endpoint redundancy
  • Hybrid DNS:
    • Resolver endpoint configuration
    • Conditional forwarding rules
    • DNS security implementation
  • Global Applications:
    • Traffic Flow policy design
    • Geoproximity configuration
    • Latency-based routing

7. Cost Optimization Tips

  • Remember:
    • Alias records are free for AWS resources
    • Health check pricing varies by interval
    • Traffic Flow has additional charges
    • Query logging affects CloudWatch costs

AWS Route 53 - DNS Domain Name Formatting

Introduction

  • Basic Structure:
    • Series of labels separated by dots
    • Each label: Maximum 63 bytes
    • Total domain name: Maximum 255 bytes (including dots)

Domain Registration Rules

  • Allowed Characters:
    • Letters: a-z (case insensitive)
    • Numbers: 0-9
    • Hyphen (-)
  • Restrictions:
    • No hyphen at start or end of label
    • Case-insensitive storage (always stored as lowercase)

Hosted Zones and Records Formatting

  • Supported Characters:
    • Letters: a-z
    • Numbers: 0-9
    • Special characters: ! " # $ % & ' ( ) * + , - / : ; < = > ? @ [ \ ] ^ _ ` { | } ~ .
  • Escape Code Requirements:
    • Required for:
      • Characters 000 to 040 octal
      • Characters 177 to 377 octal
      • Period (.) when used as character (not delimiter)
    • Format: \three-digit octal code

Wildcard (*) Usage Rules

  • Hosted Zones:
    • Cannot use * in leftmost label
    • Treated as ASCII 42 in other positions
  • Records:
    • Must be leftmost label (*.example.com)
    • Must replace entire label
    • Specific names take precedence
    • Applies to specified level and all subdomains
    • Not allowed in NS records

Internationalized Domain Names (IDN)

  • Support:
    • Requires TLD support for IDN
    • Stored in Punycode format
    • Example: 中国.asia becomes xn--fiqs8s.asia
  • Implementation Methods:
    • API/SDK: Programmatic conversion to Punycode
    • Console (Domain Registration): Automatic conversion
    • Console (Hosted Zones/Records): Manual conversion required

Exam Tips

  • Key Points to Remember:
    • Domain registration is more restrictive than hosted zones
    • Wildcard must be leftmost label in records
    • IDNs require Punycode conversion
    • Maximum lengths: 63 bytes per label, 255 bytes total
  • Common Scenarios:
    • Wildcard DNS configuration
    • International domain setup
    • Special character handling
    • Domain name validation

AWS Route 53 - Domain Registration and Management

Introduction

  • Core Functions:
    • Register new domains
    • Transfer domains to/from Route 53
    • Manage domain settings
    • Monitor domain status

Key Registration Considerations

  • Before Registration:
    • Domain names cannot be changed after registration
    • AWS credits cannot be used for domain registration
    • Special/premium domain prices not supported
    • Automatic hosted zone creation (additional charges apply)
  • Cost Considerations:
    • Registration fees are non-refundable
    • Hosted zone monthly fee
    • DNS query charges
    • Optional automatic renewal fees

Registration Process

  • Required Information:
    • Domain name and TLD selection
    • Registration period (1-10 years)
    • Contact information:
      • Registrant (domain owner)
      • Administrative contact
      • Technical contact
    • Privacy protection preferences
  • Verification Requirements:
    • Email verification for new registrant contacts
    • Verification emails from:
      • noreply@registrar.amazon (Amazon Registrar)
      • noreply@domainnameverification.net (Gandi)
    • Domain suspension if not verified

Post-Registration

  • Automatic Actions:
    • Hosted zone creation
    • Name server assignment
    • DNS service configuration
  • Next Steps:
    • Create DNS records in hosted zone
    • Configure routing preferences
    • Set up domain services (web, email, etc.)

Exam Tips

  • Remember:
    • Domain registration requires valid contact verification
    • Hosted zones are automatically created
    • Registration fees are non-refundable
    • Privacy protection options vary by TLD
  • Common Scenarios:
    • Domain registration process
    • Contact verification requirements
    • Hosted zone management
    • Privacy protection configuration
