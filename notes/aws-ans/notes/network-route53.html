AWS Route 53 Study Guide - Part 1: Core Concepts

Introduction to Route 53

  • Definition:
    • Highly available and scalable Domain Name System (DNS) web service
    • Performs three main functions: domain registration, DNS routing, health checking

Key Components and Terminology

  • Domain Registration Components:
    • Domain Name: Human-readable website address (e.g., example.com)
    • Domain Registrar: ICANN-accredited company processing registrations
    • Domain Registry: Company owning rights to specific TLDs
    • Top-level Domain (TLD): Last part of domain (.com, .org, etc.)
  • DNS Components:
    • Hosted Zone: Container for DNS records
    • DNS Record: Defines how to route traffic
    • Name Servers: Translate domain names to IP addresses
    • TTL (Time to Live): Cache duration for DNS records

Architecture and Operation

  • Control Plane vs Data Plane:
    • Control Plane: Located in us-east-1
      • Manages DNS entries
      • Handles API operations
      • Failover to us-west-2 if impaired
    • Data Plane: Globally distributed
      • 200+ Points of Presence (PoP)
      • Handles DNS queries
      • Optimized for availability

DNS Resolution Process

  • Step-by-Step Flow:
    • 1. User enters domain name in browser
    • 2. Request goes to DNS resolver (ISP)
    • 3. Resolver queries root name servers
    • 4. TLD servers provide Route 53 name servers
    • 5. Route 53 provides IP address
    • 6. Browser connects to web server

Exam Tips

  • Key Points to Remember:
    • Route 53 is both a registrar and DNS service
    • Control plane in us-east-1, data plane global
    • DNS resolution follows specific hierarchy
    • Automatic creation of hosted zone upon domain registration
  • Common Exam Scenarios:
    • Understanding DNS resolution flow
    • Distinguishing between control and data plane
    • Identifying components in DNS architecture
    • Understanding high availability aspects

AWS Route 53 Study Guide - Part 2: Routing Policies and Record Types

Record Types

  • Standard DNS Records:
    • A Record: Maps domain to IPv4 address
    • AAAA Record: Maps domain to IPv6 address
    • CNAME: Maps domain to another domain
    • MX: Defines mail servers
    • TXT: Text records for various purposes
  • Route 53 Specific:
    • Alias Records:
      • AWS-specific feature
      • Maps to AWS resources (S3, CloudFront, ALB)
      • No additional charge for Alias queries
      • Automatically updates if target IP changes

Routing Policies

  • Simple Routing:
    • Basic routing to single resource
    • Multiple values return random selection
    • Cannot be associated with health checks
  • Weighted Routing:
    • Route traffic based on assigned weights
    • Percentage of traffic = (Weight for a record / Sum of all weights)
    • Use cases: A/B testing, gradual migrations
  • Latency-based Routing:
    • Routes based on lowest latency for end user
    • Uses AWS Region latency measurements
    • Requires creating records in multiple regions
  • Failover Routing:
    • Active-passive failover configuration
    • Requires health check on primary resource
    • Automatically routes to secondary if primary fails
  • Geolocation Routing:
    • Routes based on user location
    • Can specify continents, countries, or US states
    • Requires default record for non-matched locations
  • Geoproximity Routing:
    • Routes based on geographic location of resources
    • Can shift traffic with bias values (-99 to +99)
    • Requires Route 53 Traffic Flow
  • IP-based Routing:
    • Routes based on client IP addresses
    • Uses CIDR blocks to define IP ranges
    • Requires CIDR location mapping
  • Multivalue Answer:
    • Returns multiple healthy records randomly
    • Can be associated with health checks
    • Returns up to 8 healthy records

Key Considerations

  • Alias vs CNAME:
    • Alias is free, CNAME is charged
    • Alias can be used at zone apex, CNAME cannot
    • Alias automatically updates with IP changes
  • Health Checks Integration:
    • Not all routing policies support health checks
    • Health checks are separate from records
    • Additional charges for health checks

Exam Tips

  • Common Scenarios:
    • Choosing between routing policies for specific use cases
    • Understanding when to use Alias vs CNAME
    • Identifying appropriate health check configurations
    • Calculating weighted routing distributions
  • Remember:
    • Only Alias records can point to AWS resources
    • Geolocation ≠ Geoproximity
    • Simple routing doesn't support health checks
    • Weighted routing requires all weights to calculate percentage

AWS Route 53 Study Guide - Part 3: Health Checks and DNS Failover

Health Check Fundamentals

  • Types of Health Checks:
    • Endpoint Monitoring:
      • Monitor specific IP address or domain
      • Supports HTTP, HTTPS, TCP protocols
      • Can monitor external endpoints
    • Status of Other Health Checks:
      • Calculated health checks
      • Monitor multiple endpoints
      • Useful for complex failover scenarios
    • State of CloudWatch Alarm:
      • Monitor CloudWatch metrics
      • Broader monitoring capabilities
      • Custom metric integration

Health Check Configuration

  • Key Parameters:
    • Request Interval:
      • Standard: 30 seconds
      • Fast: 10 seconds (higher cost)
    • Failure Threshold:
      • Number of consecutive failures before marking unhealthy
      • Default: 3 consecutive failures
    • String Matching:
      • Search for specific string in response
      • First 5120 bytes checked
  • Monitoring Options:
    • SNS Notifications for status changes
    • CloudWatch metrics integration
    • Latency measurements

DNS Failover Configuration

  • Active-Passive Failover:
    • Primary and backup resources
    • Automatic failover when primary is unhealthy
    • Uses failover routing policy
  • Active-Active Failover:
    • All resources active simultaneously
    • Uses multiple routing policies
    • Resources can be weighted
  • Private Endpoint Monitoring:
    • Requires CloudWatch metric
    • Use CloudWatch alarm health check
    • Useful for internal resources

Health Check Process

  • Operation Flow:
    • 1. Health checkers send requests at configured interval
    • 2. Response evaluated against configured criteria
    • 3. Status updated based on failure threshold
    • 4. DNS responses updated based on health status
  • Global Perspective:
    • Health checks from multiple regions
    • Minimum 3 regions must report healthy
    • Global status determination

Integration with Other Services

  • CloudWatch Integration:
    • Health check metrics published to CloudWatch
    • Custom alarms possible
    • Automated actions via CloudWatch
  • SNS Integration:
    • Notifications for status changes
    • Multiple notification endpoints
    • Custom message formats

Exam Tips

  • Key Concepts to Remember:
    • Health checks are separate from DNS records
    • Minimum 3 regions must report healthy
    • Fast interval (10s) vs Standard interval (30s)
    • Failure threshold affects failover time
  • Common Scenarios:
    • Setting up failover for web applications
    • Monitoring private endpoints
    • Implementing complex health check logic
    • Integrating with CloudWatch metrics

AWS Route 53 Study Guide - Part 4: Route 53 Resolver and DNS Firewall

Route 53 Resolver Overview

  • Core Functionality:
    • Recursive DNS service for VPCs
    • Handles hybrid DNS resolution
    • Supports both inbound and outbound DNS queries
    • Regional service - runs independently in each AWS Region
  • Components:
    • Resolver Endpoints:
      • Inbound: Allows on-premises to resolve AWS DNS names
      • Outbound: Allows VPCs to resolve on-premises DNS names
      • Each endpoint requires 2+ IP addresses in different AZs
    • Resolver Rules:
      • Forward rules: Direct queries to custom DNS servers
      • System rules: Default AWS resolution behavior
      • Recursive lookups: Handle standard DNS resolution

Route 53 Resolver DNS Firewall

  • Key Features:
    • Domain List Management:
      • Allow lists: Permitted domains
      • Block lists: Denied domains
      • Custom domain lists
    • Rule Groups:
      • Collections of firewall rules
      • Priority-based processing
      • Association with VPCs
  • Response Actions:
    • ALLOW: Permit the DNS query
    • BLOCK: Deny the DNS query
    • ALERT: Allow but log the query
    • Custom response: Return specific DNS response

Resolver on AWS Outposts

  • Capabilities:
    • Local DNS resolution on Outposts racks
    • Integration with on-premises DNS
    • Reduced latency for local queries
  • Configuration:
    • Requires Resolver endpoints
    • Uses existing Resolver rules
    • Supports hybrid DNS architecture

Route 53 Resolver Query Logging

  • Logging Features:
    • Log DNS queries from VPCs
    • Integration with CloudWatch Logs
    • Supports multiple destinations
    • Includes query metadata
  • Log Contents:
    • Query timestamp
    • Domain name requested
    • Query type
    • Response code
    • Source IP address

Security and Access Control

  • IAM Integration:
    • Resource-based policies
    • Service-linked roles
    • Cross-account access
  • Network Security:
    • Security group configuration
    • VPC endpoint policies
    • Network ACL considerations

Exam Tips

  • Key Points:
    • Resolver endpoints require 2+ IP addresses in different AZs
    • DNS Firewall rules process in priority order
    • Resolver is a regional service
    • Query logging supports multiple destinations
  • Common Scenarios:
    • Hybrid DNS resolution setup
    • DNS security implementation
    • Cross-account DNS management
    • DNS query monitoring and logging
  • Design Considerations:
    • High availability requirements
    • Network security integration
    • Performance optimization
    • Cost implications

AWS Route 53 - DNS Domain Name Formatting

Introduction

  • Basic Structure:
    • Series of labels separated by dots
    • Each label: Maximum 63 bytes
    • Total domain name: Maximum 255 bytes (including dots)

Domain Registration Rules

  • Allowed Characters:
    • Letters: a-z (case insensitive)
    • Numbers: 0-9
    • Hyphen (-)
  • Restrictions:
    • No hyphen at start or end of label
    • Case-insensitive storage (always stored as lowercase)

Hosted Zones and Records Formatting

  • Supported Characters:
    • Letters: a-z
    • Numbers: 0-9
    • Special characters: ! " # $ % & ' ( ) * + , - / : ; < = > ? @ [ \ ] ^ _ ` { | } ~ .
  • Escape Code Requirements:
    • Required for:
      • Characters 000 to 040 octal
      • Characters 177 to 377 octal
      • Period (.) when used as character (not delimiter)
    • Format: \three-digit octal code

Wildcard (*) Usage Rules

  • Hosted Zones:
    • Cannot use * in leftmost label
    • Treated as ASCII 42 in other positions
  • Records:
    • Must be leftmost label (*.example.com)
    • Must replace entire label
    • Specific names take precedence
    • Applies to specified level and all subdomains
    • Not allowed in NS records

Internationalized Domain Names (IDN)

  • Support:
    • Requires TLD support for IDN
    • Stored in Punycode format
    • Example: 中国.asia becomes xn--fiqs8s.asia
  • Implementation Methods:
    • API/SDK: Programmatic conversion to Punycode
    • Console (Domain Registration): Automatic conversion
    • Console (Hosted Zones/Records): Manual conversion required

Exam Tips

  • Key Points to Remember:
    • Domain registration is more restrictive than hosted zones
    • Wildcard must be leftmost label in records
    • IDNs require Punycode conversion
    • Maximum lengths: 63 bytes per label, 255 bytes total
  • Common Scenarios:
    • Wildcard DNS configuration
    • International domain setup
    • Special character handling
    • Domain name validation

AWS Route 53 - Privacy Protection for Domain Contact Information

    • Enabled by default for new registrations

    • Reduces spam by hiding contact information

      • All contact information hidden by default
      • Subject to TLD registry regulations
    • Gandi:
      • All information hidden except organization name
      • Subject to TLD registry regulations
  • TLD Registry Factors:
    • Some automatically hide all information
    • Some allow choice to hide all information
    • Some allow partial information hiding
    • Some don't allow any information hiding

    • .co.uk

AWS Route 53 - DNSSEC Configuration for Domains

Introduction to DNSSEC

  • Security Problem Addressed:
    • DNS spoofing (man-in-the-middle attacks)
    • Traffic hijacking to fake websites
    • Interception of DNS queries
    • Unauthorized response substitution
  • DNSSEC Solution:
    • Protocol for securing DNS traffic
    • Creates chain of trust for DNS responses
    • Uses cryptographic signatures to verify authenticity
    • Prevents acceptance of spoofed DNS records

Route 53 DNSSEC Support

  • Dual Functionality:
    • DNSSEC signing (for DNS service)
    • DNSSEC for domain registration
  • Implementation Options:
    • Route 53 as both registrar and DNS service
    • Route 53 as registrar with external DNS service
    • External registrar with Route 53 DNS service

How DNSSEC Works

  • Configuration Process:
    • DNS provider signs zone records with private key
    • Public key provided to domain registrar
    • Registrar forwards key to TLD registry
    • Creates chain of trust from TLD to authoritative nameservers
  • Verification Flow:
    • DNS resolver requests DS record from TLD registry
    • Resolver obtains signed records from authoritative nameservers
    • Resolver verifies signature using public key
    • If verification fails, resolver returns error
    • If verification succeeds, resolver returns authentic record

Prerequisites and Limitations

  • Requirements:
    • TLD registry must support DNSSEC
    • DNS service provider must support DNSSEC
    • DNSSEC must be configured at DNS provider first
  • Key Limitations:
    • .com and .net domains: Up to 13 public keys
    • All other domains: Up to 4 public keys
    • Keys must be in base64 format

Exam Tips

  • Key Concepts:
    • DNSSEC creates cryptographic chain of trust
    • Implementation requires both DNS provider and registrar configuration
    • Not all DNS resolvers support DNSSEC validation
    • Proper key management is critical
  • Common Scenarios:
    • Securing domains against spoofing
    • Key rotation procedures
    • Troubleshooting DNSSEC validation failures
    • Understanding DNSSEC deployment requirements
  • Remember:
    • Configure DNSSEC at DNS provider before registrar
    • Key limits vary by TLD
    • Route 53 supports both signing and registration aspects
    • Improper DNSSEC removal can cause SERVFAIL errors

    • Website/application accessibility
    • Email delivery
    • DNS resolution
    • Be prepared to revert if issues occur

Step 9: Restore TTL Values

    • Reduces DNS query load

Exam Tips

AWS Route 53 - Routing Traffic for Subdomains

Introduction

  • Subdomain Concepts:
    • Subdomains are prefixes to main domain (e.g., blog.example.com)
    • Can have multiple levels (e.g., dev.api.example.com)
    • Each subdomain can route to different resources
    • Provides organizational structure to domain namespace
  • Routing Options:
    • Standard approach: Records in parent domain's hosted zone
    • Advanced approach: Separate hosted zone for subdomain
    • Different use cases for each approach

Standard Approach: Records in Parent Domain

  • Implementation:
    • Create records in main domain's hosted zone
    • Use subdomain name in record name field
    • Example: Create "blog.example.com" record in example.com hosted zone
  • Advantages:
    • Simplest configuration
    • All records managed in one place
    • No delegation required
    • Most common approach

Advanced Approach: Separate Hosted Zone

  • Implementation Overview:
    • Create new hosted zone for subdomain
    • Create records in subdomain hosted zone
    • Create NS record in parent domain pointing to subdomain's name servers
    • Called "delegating responsibility" or "subdomain delegation"
  • Use Cases:
    • Different teams managing different subdomains
    • IAM permission separation
    • Using different DNS services for subdomains
    • Complex subdomain configurations

Creating a Subdomain Hosted Zone

  • Step-by-Step Process:
    • 1. Create hosted zone with subdomain name (e.g., acme.example.com)
    • 2. Note the four name servers assigned to subdomain
    • 3. Create records in subdomain hosted zone
    • 4. Create NS record in parent domain with subdomain's name servers
  • Important Considerations:
    • Don't create additional NS/SOA records in subdomain zone
    • Don't delete existing NS/SOA records
    • Create all subdomain records in subdomain zone only

DNS Resolution Flow

  • How Delegation Works:
    • DNS resolver queries parent domain for subdomain
    • Parent domain returns subdomain's name servers
    • Resolver queries subdomain's name servers
    • Subdomain name servers return requested record
  • Performance Impact:
    • Small performance impact on first query
    • Resolver must make additional query
    • Results cached until TTL expires
    • Subsequent queries use cached information

Common Pitfalls

  • Record Duplication Issues:
    • Records in both parent and subdomain zones cause inconsistent behavior
    • Depends on which name servers resolver queries
    • Can result in NXDOMAIN responses even when record exists
    • All subdomain records must be in one place only
  • Delegation Chain Issues:
    • Incorrect name servers in NS record
    • Missing NS record in parent zone
    • Broken chain of delegation

Multi-Level Subdomains

  • Nested Delegation:
    • Can delegate sub-subdomains (e.g., dev.api.example.com)
    • Create NS record in immediate parent zone
    • Example: NS for dev.api.example.com goes in api.example.com zone
    • Maintains proper delegation chain
  • Best Practice:
    • Keep delegation hierarchy logical
    • Document delegation structure
    • Limit delegation depth when possible

Exam Tips

  • Key Decision Points:
    • Standard approach: Simple management, single zone
    • Separate zone approach: Administrative separation, complex management
    • Consider IAM permissions requirements
    • Consider management boundaries
  • Critical Concepts:
    • NS records create delegation
    • Delegation creates chain of trust
    • Records must exist in only one zone
    • Proper delegation requires correct name servers
  • Common Scenarios:
    • Subdomain management strategies
    • Troubleshooting delegation issues
    • Multi-team DNS management
    • Hybrid DNS architectures

AWS Route 53 - White-Label Name Servers

Introduction

  • White-Label Name Servers Concept:
    • Custom domain names for Route 53 name servers
    • Also known as vanity or private name servers
    • Example: ns1.example.com instead of ns-2048.awsdns-64.com
    • Creates professional, branded DNS infrastructure
  • Implementation Benefits:
    • Consistent branding across all domain elements
    • Reusable across multiple domains
    • Simplified DNS migration
    • Consistent name server configuration

Implementation Overview

  • Key Components:
    • Reusable delegation set: Set of four Route 53 name servers
    • Custom hosted zones using the delegation set
    • A/AAAA records for white-label name servers
    • Glue records at registrar
  • Implementation Process:
    • 10-step process
    • Requires API/CLI usage
    • Careful TTL management
    • Precise IP address configuration

Step 1: Create Reusable Delegation Set

  • Process Details:
    • Must use API, CLI, or SDK (not available in console)
    • CreateReusableDelegationSet API operation
    • Returns four Route 53 name servers
    • Same AWS account must own delegation set and hosted zones

Step 2-3: Create Hosted Zones and Records

  • Hosted Zone Creation:
    • Create new hosted zones with reusable delegation set
    • Cannot modify existing hosted zones
    • Lower TTL values initially (60 seconds or less)
    • Recreate all records in new hosted zones
  • Critical Warning:
    • Cannot change name servers for existing zones
    • Must recreate zones to use reusable delegation set
    • Requires careful migration planning

Step 4-5: Configure White-Label Records

  • IP Address Collection:
    • Get IPv4 and IPv6 addresses for all name servers
    • Use dig or nslookup commands
    • Document name server and IP mappings
  • Record Creation:
    • Create A and AAAA records in primary domain's hosted zone
    • One A record per name server (IPv4)
    • One AAAA record per name server (IPv6)
    • Use low TTL values initially
    • Critical: Correct IP addresses essential

Step 6-7: Update NS/SOA Records and Create Glue Records

  • NS/SOA Updates:
    • Update SOA record with white-label name server
    • Update NS record with all four white-label name servers
    • Document original name servers for rollback
  • Glue Record Creation:
    • Create at registrar
    • Link white-label name server names to IP addresses
    • Include both IPv4 and IPv6 addresses
    • Update name servers for domain to white-label servers

Step 8-10: Monitoring and Finalization

  • Traffic Monitoring:
    • Monitor website/application traffic
    • Be prepared to revert if issues occur
    • Repeat process for additional domains
  • TTL Restoration:
    • Increase NS record TTL to normal values (e.g., 172800 seconds)
    • Increase SOA minimum TTL to normal values
    • Improves resolver performance
  • Optional DNS Service Notification:
    • Contact recursive DNS services
    • Notify about white-label name servers
    • Important for geolocation routing
    • Ensures optimal routing based on client location

Exam Tips

  • Key Concepts:
    • Reusable delegation sets require API/CLI
    • Cannot modify name servers for existing zones
    • White-label servers require glue records
    • TTL management critical during migration
  • Common Scenarios:
    • Multi-domain DNS management
    • Professional DNS branding requirements
    • DNS migration planning
    • Troubleshooting white-label configuration
  • Critical Warnings:
    • Incorrect IP addresses cause domain unavailability
    • Cannot modify existing hosted zone name servers
    • Requires careful planning and execution
    • Document original configuration for rollback

AWS Route 53 - NS and SOA Records

Introduction

  • Automatic Record Creation:
    • Route 53 automatically creates NS and SOA records
    • Created for every public hosted zone
    • Rarely need modification
    • Critical for DNS functionality
  • Record Importance:
    • Fundamental to DNS operation
    • Define authoritative name servers
    • Provide zone metadata
    • Enable domain resolution

Name Server (NS) Records

  • Purpose and Function:
    • Lists authoritative name servers for the zone
    • Contains four Route 53 name servers
    • Same name as hosted zone
    • Used by registrars to delegate DNS authority
  • Name Server Format:
    • Format: ns-XXXX.awsdns-XX.TLD
    • Example: ns-2048.awsdns-64.com
    • Four different name servers per zone
    • Distributed across different TLDs for redundancy
  • Best Practices:
    • Don't add, change, or delete name servers
    • Use all four name servers at registrar
    • Maintain consistent configuration

Start of Authority (SOA) Record

  • Record Format:
    • Example: ns-2048.awsdns-64.net. hostmaster.example.com. 1 7200 900 1209600 86400
    • Contains seven distinct elements
    • Provides zone metadata
    • Controls various DNS behaviors
  • SOA Components:
    • 1. Primary name server: Route 53 name server that created the record
    • 2. Administrator email: @ replaced with period (default is amazon.com address)
    • 3. Serial number: Optional version number (not auto-incremented)
    • 4. Refresh time: 7200 seconds (2 hours)
    • 5. Retry interval: 900 seconds (15 minutes)
    • 6. Expiry time: 1209600 seconds (2 weeks)
    • 7. Minimum TTL: 86400 seconds (1 day)

SOA Minimum TTL Significance

  • Negative Caching Control:
    • Controls caching of negative responses
    • Affects NXDOMAIN responses (domain doesn't exist)
    • Affects NODATA responses (record type doesn't exist)
    • Lesser of SOA minimum TTL and SOA record TTL
  • Cost and Performance Impact:
    • Longer TTL: Fewer queries, lower cost
    • Shorter TTL: More queries, higher cost
    • Longer TTL: Longer recovery from errors
    • Shorter TTL: Faster recovery from errors

Finding NS and SOA Records

  • Viewing Name Servers:
    • Hosted zone details page
    • Listed under "Name servers" section
    • Always shows four name servers
    • Critical for domain registration configuration
  • Viewing SOA Record:
    • Records section of hosted zone
    • Filter for SOA record type
    • Contains all seven components
    • Can be edited (though rarely needed)

Exam Tips

  • Key Concepts:
    • NS records define authoritative name servers
    • SOA record contains zone metadata
    • Both created automatically
    • Rarely should be modified
  • SOA Components to Remember:
    • Minimum TTL controls negative caching
    • Serial number not auto-incremented
    • Administrator email format (period replaces @)
    • Primary name server is a Route 53 name server
  • Common Scenarios:
    • Identifying authoritative name servers
    • Understanding negative caching behavior
    • Troubleshooting DNS propagation
    • Optimizing DNS query costs
  • Best Practices:
    • Don't modify NS record content
    • Consider TTL adjustments carefully
    • Understand negative caching implications
    • Use all four name servers at registrar

AWS Route 53 Hosted Zones - Exam Critical Summary

1. Hosted Zone Fundamentals

  • EXAM CRITICAL:
    • Hosted zone = container for DNS records with same name as domain
    • Public hosted zone routes internet traffic; private hosted zone routes VPC traffic
    • NS and SOA records created automatically - do not modify or delete
    • Maximum one active hosted zone per domain (though multiple can exist)

2. Name Server (NS) Records

  • EXAM CRITICAL:
    • Four name servers assigned to each hosted zone
    • Format: ns-XXXX.awsdns-XX.TLD (e.g., ns-2048.awsdns-64.com)
    • All four must be configured at registrar for DNS to work
    • Default TTL: 172800 seconds (2 days)
    • Name server changes take up to 48 hours to fully propagate

3. Start of Authority (SOA) Records

  • EXAM CRITICAL:
    • Format: [nameserver] [admin-email] [serial] [refresh] [retry] [expiry] [minimum-ttl]
    • Minimum TTL controls negative caching (NXDOMAIN/NODATA responses)
    • Negative caching = lesser of SOA minimum TTL and SOA record TTL
    • Negative responses are charged same as standard queries

4. DNS Query Metrics

  • EXAM CRITICAL:
    • Metrics available only in US East (N. Virginia) region
    • Only counts queries forwarded to Route 53 (not cached responses)
    • DNSQueries metric sent automatically to CloudWatch
    • No charge for DNS query metrics
    • More detailed query logging available (separate feature)

5. Hosted Zone Deletion

  • EXAM CRITICAL:
    • Can only delete zones with no records except NS and SOA
    • Must disable DNSSEC signing before deletion
    • Service-created zones (AWS Cloud Map, ECS) require special deletion process
    • For subdomain delegation: delete parent NS record first, wait for TTL, then delete child zone
    • Deletion is permanent and cannot be undone

6. White-Label Name Servers

  • EXAM CRITICAL:
    • Requires reusable delegation set (API/CLI only, not console)
    • Cannot change name servers for existing zones - must recreate zones
    • Requires both A and AAAA records for each name server
    • Requires glue records at registrar
    • Incorrect IP addresses cause domain unavailability
    • Lower TTLs during migration for faster recovery

7. DNS Response Checking Tool

  • EXAM CRITICAL:
    • Shows configuration-based responses, not actual DNS queries
    • Works regardless of whether zone is active
    • Supports geolocation/latency testing with specific IPs
    • Only available for public hosted zones
    • Cannot test subdomain delegation name server responses

8. Multiple Hosted Zones with Same Name

  • EXAM CRITICAL:
    • Possible to create multiple zones with identical names
    • Each gets different name servers
    • Only one set can be active at registrar
    • Route 53 never returns values from inactive zones
    • Useful for testing configuration changes

9. Reusable Delegation Sets

  • EXAM CRITICAL:
    • Same four name servers across multiple hosted zones
    • Created programmatically only (API, CLI, SDK)
    • Must be specified when creating hosted zone
    • Cannot be changed for existing zones
    • Simplifies DNS migration (same name servers for multiple domains)

10. Common Exam Scenarios

  • EXAM CRITICAL:
    • Troubleshooting DNS resolution issues
    • Migrating DNS service to Route 53
    • Configuring white-label name servers
    • Understanding negative caching behavior
    • Properly removing subdomain delegation
    • Interpreting DNS query metrics
    • Managing multiple hosted zones
    • Optimizing DNS query costs

11. Key Numerical Values

  • EXAM CRITICAL:
    • 4 name servers per hosted zone
    • Maximum 6 name servers when using white-label configuration
    • Default NS TTL: 172800 seconds (2 days)
    • Default SOA minimum TTL: 86400 seconds (1 day)
    • DNS metrics available only in us-east-1 region
    • Up to 48 hours for name server changes to propagate
