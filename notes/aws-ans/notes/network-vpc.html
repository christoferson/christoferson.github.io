AWS Certified Advanced Networking - Specialty (ANS-C01)

VPC Connectivity Options Study Guide

Introduction to VPC Connectivity

  • Overview:
    • AWS Virtual Private Cloud (VPC) provides isolated network environments in the cloud
    • Multiple connectivity options enable communication between VPCs and external networks
    • Connectivity solutions address different use cases: internet access, VPC-to-VPC communication, and hybrid networking
    • Each option has distinct characteristics, benefits, and limitations
  • Key Connectivity Categories:
    • Internet Access: Connecting VPC resources to the public internet
    • VPC-to-VPC Communication: Connecting resources across different VPCs
    • Hybrid Connectivity: Connecting VPCs to on-premises networks
    • Service Access: Connecting to AWS services privately

Internet Gateway

  • Core Concepts:
    • Horizontally scaled, redundant, and highly available VPC component
    • Enables two-way communication between VPC resources and the internet
    • Supports both IPv4 and IPv6 traffic
    • No availability risks or bandwidth constraints
  • Key Characteristics:
    • Performs Network Address Translation (NAT) for IPv4 traffic
    • No NAT required for IPv6 traffic (globally routable)
    • Attached at the VPC level, not subnet level
    • No charge for the gateway itself (only data transfer charges)
  • Public Subnet Concept:
    • Subnet with route table entry pointing to internet gateway (0.0.0.0/0 → igw-id)
    • Instances must have public IP addresses or Elastic IP addresses
    • Instances only aware of private IP addresses; IGW handles NAT
    • Default VPC comes with internet gateway and public subnets

Egress-Only Internet Gateway

  • Core Concepts:
    • Specialized gateway for IPv6 traffic only
    • Allows outbound IPv6 communication while preventing inbound connections
    • Stateful component that remembers outbound requests
    • IPv4 equivalent is a NAT gateway
  • Key Characteristics:
    • No charge for the gateway itself (only data transfer charges)
    • Cannot associate security groups with it
    • Control traffic using network ACLs on the subnet
    • Route table configuration: ::/0 → eigw-id

NAT Devices (Gateways and Instances)

  • Purpose and Function:
    • Enable resources in private subnets to access the internet or other networks
    • Allow outbound connections while preventing inbound connections
    • Perform both address translation and port address translation (PAT)
    • Two types: AWS-managed NAT gateways and self-managed NAT instances
  • NAT Gateway Types:
    • Public NAT Gateway: For internet access via internet gateway
    • Private NAT Gateway: For access to other VPCs or on-premises networks
    • Both types map source private IPs to NAT gateway's private IP
    • Public NAT gateway uses EIP for internet communication
  • NAT Gateway Characteristics:
    • Created in specific Availability Zone with redundancy in that zone
    • Supports TCP, UDP, and ICMP protocols
    • Bandwidth: 5 Gbps base with auto-scaling to 100 Gbps
    • Packet processing: 1 million packets/second scaling to 10 million
    • Connection limit: 55,000 connections per unique destination
    • Can associate up to 8 IP addresses to increase connection limits
    • MTU support up to 8500 bytes
  • NAT Instance Characteristics:
    • Self-managed EC2 instance with NAT configuration
    • Requires manual setup and maintenance
    • Performance depends on instance type
    • Requires disabling source/destination checks
    • Can serve additional functions (bastion host, port forwarding)

Exam Tips

  • Internet Gateway vs. NAT Gateway vs. Egress-Only Internet Gateway:
    • Internet Gateway: Two-way IPv4/IPv6 communication, requires public IPs
    • NAT Gateway: Outbound-only IPv4 communication from private IPs
    • Egress-Only Internet Gateway: Outbound-only IPv6 communication
  • NAT Gateway vs. NAT Instance Decision Factors:
    • Availability requirements: NAT gateway offers better availability
    • Bandwidth needs: NAT gateway scales up to 100 Gbps
    • Maintenance preference: NAT gateway is fully managed by AWS
    • Special requirements: NAT instance for port forwarding or bastion server needs
  • High Availability Considerations:
    • Create NAT gateways in each AZ for zone-independent architecture
    • Resources in one AZ lose internet if that AZ's NAT gateway fails
    • Internet gateway is inherently highly available across all AZs
  • Common Scenarios:
    • Public subnet with IGW for internet-facing resources
    • Private subnet with NAT gateway for outbound-only internet access
    • IPv6 subnet with egress-only gateway for outbound-only IPv6 traffic
    • Private NAT gateway for communication with overlapping networks

VPC Connectivity Options Study Guide - Part 2

Transit Gateway

  • Core Concepts:
    • Regional network transit hub that connects VPCs and on-premises networks
    • Highly available and scalable with built-in redundancy
    • Acts as a cloud router - single connection point for all networks
    • Simplifies network architecture by eliminating complex peering relationships
  • Key Capabilities:
    • Supports thousands of VPC attachments per Region
    • Handles transitive routing between all connected networks
    • Integrates with Direct Connect gateways and VPN connections
    • Supports IP multicast (the only AWS service that does)
    • Enables centralized network policy enforcement
  • Routing Control:
    • Route tables control traffic flow between attachments
    • Support for route propagation and static routes
    • Blackhole routes for traffic filtering
    • Multiple route tables for network segmentation
    • Appliance Mode for stateful inspection requirements
  • Advanced Features:
    • Transit Gateway Network Manager for global network visualization
    • Cross-account sharing via AWS RAM
    • Inter-region peering for global connectivity
    • Multicast support for one-to-many applications
    • Flow logs for traffic monitoring and analysis

AWS Virtual Private Network (VPN)

  • Site-to-Site VPN:
    • Encrypted connection between VPC and on-premises network
    • Uses IPsec protocol to establish secure tunnels
    • Two tunnels per connection for high availability
    • Supports static routes or dynamic routing with BGP
    • Can connect to Virtual Private Gateway or Transit Gateway
  • Client VPN:
    • Managed client-based VPN service
    • Enables secure access to AWS and on-premises resources
    • Uses OpenVPN clients
    • Supports authentication via Active Directory, certificates, or SAML
    • Split-tunnel and full-tunnel modes available
  • VPN CloudHub:
    • Hub-and-spoke model for connecting multiple sites
    • Uses Virtual Private Gateway as the hub
    • Each site establishes VPN connection to the hub
    • Sites can communicate with each other through the hub
    • Cost-effective solution for connecting multiple remote networks
  • Accelerated Site-to-Site VPN:
    • Uses AWS Global Accelerator to improve VPN performance
    • Traffic routed through AWS global network instead of public internet
    • Reduces jitter and latency
    • Higher cost than standard Site-to-Site VPN
    • Ideal for latency-sensitive applications

VPC Peering

  • Core Concepts:
    • Direct network connection between two VPCs
    • Uses private IP addresses for communication
    • Traffic stays on the AWS network (not over internet)
    • No single point of failure or bandwidth bottleneck
    • No gateway or VPN connection required
  • Key Characteristics:
    • Non-transitive relationship (A↔B, B↔C does not mean A↔C)
    • Can peer across AWS accounts and regions
    • Cannot have overlapping CIDR blocks
    • Maximum of 125 peering connections per VPC
    • Security groups can reference peer VPC security groups
  • Routing Configuration:
    • Requires route table entries in both VPCs
    • Routes point to the peering connection (pcx-id)
    • Can be specific CIDR blocks or the entire VPC CIDR
    • Most specific route takes precedence
    • No automatic route propagation
  • Limitations:
    • No transitive peering
    • No edge to edge routing through a VPC peering connection
    • No support for overlapping CIDR blocks
    • No support for IPv6 communication if VPCs have matching IPv4 CIDRs
    • Cannot reference security groups across peering if using different regions

AWS Direct Connect

  • Core Concepts:
    • Dedicated private network connection from on-premises to AWS
    • Bypasses the public internet for improved reliability and performance
    • Reduces network costs for high-volume data transfer
    • Lower and more consistent latency than internet-based connections
    • Supports both public and private AWS resources
  • Connection Types:
    • Dedicated Connection: 1, 10, 100, or 400 Gbps direct to AWS
    • Hosted Connection: Sub-1Gbps to 10Gbps through AWS partners
    • Direct Connect Gateway: Access to multiple regions from single connection
    • Link Aggregation Groups (LAG): Combine multiple connections for higher bandwidth
  • Virtual Interfaces (VIFs):
    • Private VIF: Connect to resources in a VPC using private IP addresses
    • Public VIF: Connect to public AWS services using public IP addresses
    • Transit VIF: Connect to resources in VPCs through Transit Gateway
    • Each VIF requires BGP configuration
  • Resiliency Models:
    • Maximum Resiliency: Multiple connections at multiple locations
    • High Resiliency: Multiple connections at multiple locations
    • Development and Test: Multiple connections at a single location
    • Direct Connect with VPN backup for additional security

Exam Tips

  • Connectivity Solution Selection:
    • Transit Gateway: For complex networks with many VPCs and on-premises connections
    • VPC Peering: For simple point-to-point connectivity between two VPCs
    • Site-to-Site VPN: For encrypted connections over the internet
    • Direct Connect: For high-bandwidth, consistent, private connectivity
  • Transitive Routing Considerations:
    • VPC Peering: Non-transitive (A↔B, B↔C does not mean A↔C)
    • Transit Gateway: Supports transitive routing
    • Direct Connect Gateway: Enables access to multiple VPCs across regions
  • High Availability Design:
    • Site-to-Site VPN: Deploy to multiple VGWs or TGW attachments
    • Direct Connect: Use multiple connections across locations
    • Transit Gateway: Deploy across multiple AZs
    • Hybrid: Combine Direct Connect with VPN backup
  • Cost Considerations:
    • VPC Peering: Lowest cost for VPC-to-VPC communication
    • Transit Gateway: Higher cost but simplified management for many connections
    • Site-to-Site VPN: Cost-effective for lower bandwidth requirements
    • Direct Connect: Higher fixed costs but lower data transfer costs

AWS VPC Connectivity Options - Comprehensive Summary

1. Internet Connectivity Options

  • Internet Gateway (IGW):
    • Two-way communication between VPC and internet
    • Supports both IPv4 and IPv6 traffic
    • Performs NAT for IPv4 (not for IPv6)
    • Highly available and horizontally scaled
    • Required for public subnets
    • No additional charge beyond data transfer
  • Egress-Only Internet Gateway:
    • Outbound-only IPv6 traffic
    • Prevents inbound connections from internet
    • Stateful device that tracks connections
    • IPv6 equivalent of NAT gateway
    • No additional charge beyond data transfer
  • NAT Gateway:
    • Enables outbound internet access for private subnets
    • Two types: Public (internet access) and Private (VPC/on-premises access)
    • Highly available within a single AZ
    • Scales from 5 Gbps to 100 Gbps
    • Supports up to 55,000 connections per destination (expandable with additional IPs)
    • Fully managed by AWS
  • NAT Instance:
    • Self-managed EC2 instance with NAT configuration
    • Requires manual setup and maintenance
    • Performance limited by instance type
    • Supports additional features like port forwarding and bastion access
    • Requires disabling source/destination checks

2. VPC-to-VPC Connectivity

  • VPC Peering:
    • Direct network connection between two VPCs
    • Non-transitive relationship
    • No overlapping CIDR blocks
    • Cross-account and cross-region support
    • No bandwidth constraints
    • Maximum 125 peering connections per VPC
    • Ideal for simple point-to-point connectivity
  • Transit Gateway:
    • Regional network transit hub
    • Connects multiple VPCs and on-premises networks
    • Supports transitive routing
    • Multiple route tables for traffic control
    • Cross-account sharing via AWS RAM
    • Inter-region peering capability
    • Ideal for complex network architectures
  • PrivateLink:
    • Private connectivity to services across VPCs
    • No need for VPC peering, internet gateway, NAT, etc.
    • Uses interface endpoints in your VPC
    • Traffic stays on AWS network
    • Ideal for accessing services without exposing to internet

3. Hybrid Connectivity

  • Site-to-Site VPN:
    • Encrypted IPsec connection over internet
    • Two tunnels per connection for redundancy
    • Supports static routes or BGP
    • Can connect to Virtual Private Gateway or Transit Gateway
    • Accelerated option available for improved performance
    • Cost-effective solution for moderate bandwidth needs
  • Direct Connect:
    • Dedicated private connection to AWS
    • 1, 10, 100, or 400 Gbps bandwidth options
    • Lower latency and consistent performance
    • Private VIF for VPC access, Public VIF for AWS services
    • Transit VIF for Transit Gateway access
    • Multiple resiliency models available
    • Ideal for high bandwidth and consistent latency requirements
  • Client VPN:
    • Managed OpenVPN-based remote access VPN
    • Secure access for remote users
    • Supports authentication via AD, certificates, or SAML
    • Split-tunnel or full-tunnel options
    • Access to both AWS and on-premises resources

4. Decision Matrix: Selecting the Right Connectivity Option

  • For Internet Access:
    • Public resources: Internet Gateway
    • Private resources needing outbound IPv4: NAT Gateway
    • Private resources needing outbound IPv6: Egress-Only Internet Gateway
    • Special requirements (port forwarding, bastion): NAT Instance
  • For VPC-to-VPC Connectivity:
    • Simple connection between two VPCs: VPC Peering
    • Multiple VPCs with transitive routing: Transit Gateway
    • Access to services without exposure: PrivateLink
  • For Hybrid Connectivity:
    • Quick setup, moderate bandwidth: Site-to-Site VPN
    • High bandwidth, consistent performance: Direct Connect
    • Maximum reliability: Direct Connect + VPN backup
    • Remote user access: Client VPN
    • Multiple branch offices: Transit Gateway + Site-to-Site VPN

5. Key Architectural Considerations

  • High Availability:
    • NAT Gateway: Deploy in each AZ with resources
    • Transit Gateway: Inherently highly available
    • VPN: Use multiple tunnels across different AZs
    • Direct Connect: Use multiple connections across locations
  • Scalability:
    • NAT Gateway: Scales to 100 Gbps, 10M packets per second
    • Transit Gateway: Thousands of attachments per region
    • VPC Peering: Limited to 125 connections per VPC
    • Direct Connect: Use LAGs or multiple connections for higher bandwidth
  • Security:
    • All traffic within AWS network is automatically encrypted
    • VPN connections provide encrypted tunnels
    • Direct Connect is private but not encrypted by default
    • Security groups and NACLs control traffic at VPC level
  • Cost Optimization:
    • Minimize cross-AZ traffic to reduce data transfer costs
    • Use Direct Connect for high-volume data transfer
    • Consider Transit Gateway for many VPC connections
    • VPC Peering is most cost-effective for simple connections

6. Common Exam Scenarios

  • Scenario 1: Multi-VPC Architecture
    • Few VPCs, simple connectivity: VPC Peering
    • Many VPCs, complex routing: Transit Gateway
    • Shared services model: PrivateLink or Transit Gateway
  • Scenario 2: Hybrid Cloud
    • Quick deployment, moderate bandwidth: Site-to-Site VPN
    • High bandwidth, critical workloads: Direct Connect
    • Maximum resilience: Direct Connect + VPN backup
    • Multiple sites: Transit Gateway + Site-to-Site VPN
  • Scenario 3: Internet Access Architecture
    • Public-facing resources: Internet Gateway
    • Private resources needing internet: NAT Gateway
    • IPv6-enabled private resources: Egress-Only Internet Gateway
    • Centralized egress: Transit Gateway + NAT Gateway

VPC Connectivity Options Study Guide - Part 3

Elastic IP Addresses

  • Core Concepts:
    • Static, public IPv4 addresses designed for dynamic cloud computing
    • Can be associated with any instance or network interface in a VPC
    • Allows masking instance failures by remapping to another instance
    • Provides consistent public endpoint for your applications
  • Key Characteristics:
    • Associated with a network interface, not directly with an instance
    • Limited to 5 EIPs per region by default (adjustable)
    • IPv6 not supported with Elastic IPs
    • Can be tagged for cost allocation tracking
    • Regional resource (not global)
  • Rules and Limitations:
    • One EIP can be associated with one instance/interface at a time
    • Associating an EIP replaces any existing public IPv4 address
    • Disassociating an EIP from primary interface auto-assigns new public IPv4
    • Security groups and NACLs control access to the EIP
    • Internet gateway required for reply traffic
  • IP Address Sources:
    • Amazon's pool of IPv4 addresses
    • Your own IP addresses (BYOIP)
    • Customer-owned pool (for Outposts)

Elastic IP Management

  • Allocation Process:
    • Allocate from chosen IP pool
    • Select appropriate network border group
    • Apply tags for organization and cost tracking
  • Association and Disassociation:
    • Can associate with instance or network interface
    • Disassociate before changing associated resource
    • Primary network interface gets new public IP after disassociation
  • Transfer Capabilities:
    • Can transfer EIPs between AWS accounts in same region
    • Two-step handshake process with 7-day acceptance window
    • Tags not transferred with the EIP
    • BYOIP addresses cannot be transferred
  • Release and Recovery:
    • Release unused EIPs to avoid charges
    • Can recover recently released EIPs if still available
    • Recovery not possible if allocated to another account
    • Recovery counts against your EIP quota

Transit Gateway

  • Core Concepts:
    • Regional network transit hub for VPCs and on-premises networks
    • Simplifies network architecture by eliminating complex peering relationships
    • Acts as a cloud router with a single connection point
    • Highly available and scalable with built-in redundancy
  • Key Use Cases:
    • Centralized router connecting all VPCs, Direct Connect, and VPN connections
    • Isolated VPCs with multiple logical routers
    • Shared services model with controlled access
    • Hub-and-spoke network topology
  • Routing Control:
    • Route tables control traffic flow between attachments
    • Multiple route tables for network segmentation
    • Blackhole routes for traffic filtering
    • Support for route propagation and static routes

VPN Connectivity Options

  • AWS Site-to-Site VPN:
    • IPsec VPN connection between VPC and remote network
    • Virtual private gateway or transit gateway provides two VPN endpoints
    • Automatic failover between tunnels
    • Customer gateway device required on remote side
  • AWS Client VPN:
    • Managed client-based VPN service
    • Secure access to AWS or on-premises resources
    • OpenVPN-based client connections
    • TLS VPN sessions for security
  • AWS VPN CloudHub:
    • Connect multiple remote networks (branch offices)
    • Uses virtual private gateway with multiple VPN connections
    • Enables communication between remote sites
    • Hub-and-spoke model for remote network connectivity
  • Third-party VPN Solutions:
    • EC2 instances running third-party VPN software
    • Available through AWS Marketplace
    • Customer manages the VPN appliance
    • Flexible configuration options

VPC Peering

  • Core Concepts:
    • Networking connection between two VPCs
    • Enables private communication as if in same network
    • Traffic never traverses public internet
    • No gateway, VPN, or hardware required
  • Key Capabilities:
    • Connect VPCs in same or different AWS accounts
    • Connect VPCs in same or different AWS regions
    • No single point of failure or bandwidth bottleneck
    • All traffic stays within AWS network
  • Use Cases:
    • Communication between application tiers
    • Resource sharing between teams or business units
    • Hybrid cloud architectures
    • Simple point-to-point connectivity needs

Exam Tips

  • Elastic IP Addresses:
    • Remember the 5 EIP default limit per region
    • Understand EIP is a property of network interface, not instance
    • Know that releasing unused EIPs avoids charges
    • Recognize transfer process between accounts
  • Transit Gateway:
    • Identify use cases for centralized routing
    • Understand route table functionality for traffic control
    • Know integration points with other AWS services
    • Recognize regional nature of the service
  • VPN Options:
    • Differentiate between Site-to-Site, Client VPN, and CloudHub
    • Understand when to use AWS-managed vs. third-party solutions
    • Know integration points with Direct Connect
    • Recognize high availability configurations
  • VPC Peering:
    • Remember non-transitive nature of peering
    • Understand cross-account and cross-region capabilities
    • Know that overlapping CIDR blocks are not supported
    • Recognize when peering is appropriate vs. other solutions

VPC Monitoring Study Guide - Part 1

Introduction to VPC Monitoring

  • Available Monitoring Tools:
    • VPC Flow Logs
    • CloudWatch Internet Monitor
    • IP Address Manager (IPAM)
    • Traffic Mirroring
    • Reachability Analyzer
    • Network Access Analyzer
    • CloudTrail Logs

VPC Flow Logs Deep Dive

  • Core Concepts:
    • Captures IP traffic information for network interfaces in VPC
    • Operates outside network path (no performance impact)
    • Can be created or deleted without affecting network performance
    • Supports multiple destination options for logs
  • Publishing Destinations:
    • CloudWatch Logs
    • Amazon S3
    • Amazon Kinesis Data Firehose
  • Key Use Cases:
    • Diagnose restrictive security group rules
    • Monitor instance traffic
    • Determine traffic direction
    • Security analysis
    • Compliance requirements

Flow Logs to CloudWatch Logs

  • Structure:
    • Published to a log group
    • Each network interface gets unique log stream
    • Multiple flow logs can publish to same log group
    • Combined log stream for same interface in multiple logs
  • Timestamp Handling:
    • Timestamp field: Flow start time
    • IngestionTime: When CloudWatch received the record
    • IngestionTime is always later than flow end time

Flow Logs to S3

  • Key Features:
    • Published to existing S3 bucket
    • Creates series of log file objects
    • Captures all network interfaces in VPC if VPC-level logging
    • Supports data processing with tools like Athena

Flow Logs to Kinesis Data Firehose

  • Characteristics:
    • Real-time delivery to Firehose delivery stream
    • Plain text format
    • Supports transformation and processing
    • Can deliver to multiple destinations

Cost Considerations

  • Pricing Components:
    • Data ingestion charges
    • Data archival charges
    • Vended logs pricing applies
    • Cost allocation tags available for tracking

Exam Tips

  • Key Points to Remember:
    • Flow Logs operate outside network path - no performance impact
    • Three destination options with different use cases
    • CloudWatch integration provides metrics and alerting
    • S3 destination enables long-term storage and analysis
    • Firehose enables real-time processing
  • Common Scenarios:
    • Troubleshooting security group issues
    • Network traffic analysis
    • Security and compliance monitoring
    • Cost tracking with tags
  • Design Considerations:
    • Choose destination based on analysis needs
    • Consider retention requirements
    • Plan for cost management
    • Understand IAM requirements for each destination

VPC Monitoring - Network Address Usage (NAU) Metrics

NAU Fundamentals

  • Definition and Purpose:
    • Metric for measuring VPC size and resource consumption
    • Helps plan and monitor VPC capacity
    • No additional cost for monitoring
    • Essential for resource planning and quota management
  • Business Impact:
    • Exhausting NAU quota prevents:
      • Launching new EC2 instances
      • Provisioning Network Load Balancers
      • Creating VPC endpoints
      • Deploying Lambda functions
      • Adding transit gateway attachments
      • Creating NAT gateways

CloudWatch Metrics for NAU

  • AWS/EC2 Namespace:
    • NetworkAddressUsage:
      • Measures NAU count per individual VPC
      • 24-hour reporting interval
      • Uses Per-VPC Metrics dimension with VPC ID
    • NetworkAddressUsagePeered:
      • Measures NAU count for VPC and all peered VPCs
      • 24-hour reporting interval
      • Uses Per-VPC Metrics dimension with VPC ID

Monitoring Configuration

  • Enabling NAU Monitoring:
    • Must be enabled per VPC
    • Not enabled by default
    • Can be enabled/disabled through console or CLI
    • No additional charge for NAU monitoring
  • Best Practices:
    • Enable monitoring for critical VPCs
    • Set up alarms for threshold monitoring
    • Monitor growth trends
    • Plan capacity based on usage patterns

Exam Tips

  • Key Points:
    • NAU monitoring is free but must be explicitly enabled
    • Metrics are reported every 24 hours
    • Two main metrics: individual VPC and peered VPCs
    • Critical for capacity planning and quota management
  • Common Scenarios:
    • VPC capacity planning
    • Resource provisioning decisions
    • Growth trend analysis
    • Quota management
