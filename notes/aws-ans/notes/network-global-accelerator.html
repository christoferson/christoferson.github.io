AWS Global Accelerator - Comprehensive Exam Guide Summary

1. Core Concepts

  • Service Definition:
    • AWS service that improves application performance and availability
    • Routes traffic over AWS global network instead of public internet
    • Provides static IP addresses as entry points
    • Two types: Standard and Custom Routing accelerators
  • Key Components:
    • Static IP addresses: 2 IPv4 or 4 addresses (2 IPv4 + 2 IPv6)
    • Network zones: Isolated infrastructure units for fault isolation
    • Listeners: Process inbound connections based on ports/protocols
    • Endpoint groups: Regional collections of endpoints
    • Endpoints: Resources that receive traffic (ALB, NLB, EC2, EIP, VPC subnets)

2. Accelerator Types

  • Standard Accelerator:
    • Routes based on proximity, health, and weights
    • Supports all endpoint types
    • Includes health checks and automatic failover
    • Supports traffic dials and endpoint weights
    • Ideal for general web applications and APIs
  • Custom Routing Accelerator:
    • Deterministic routing to specific EC2 instances and ports
    • Supports VPC subnet endpoints only
    • No health checks or automatic failover
    • IPv4 only (no dual-stack)
    • Ideal for VoIP, gaming, and real-time applications

3. Security and Connectivity

  • VPC Connectivity:
    • Requires internet gateway attached to VPC
    • No public IP addresses needed on resources
    • No internet gateway route required for subnet
    • Traffic returns through Global Accelerator, not internet gateway
  • Security Features:
    • Origin masking behind static IPs
    • AWS Shield protection for static IPs
    • Private VPC connections using private IPs
    • Security groups still apply to all traffic
  • Protocol Requirements:
    • TLS 1.2 minimum (1.3 recommended)
    • Perfect Forward Secrecy cipher suites (DHE/ECDHE)
    • ICMP traffic cannot be blocked at accelerator level

4. Traffic Management

  • Traffic Dials:
    • Controls percentage of traffic to endpoint group (0-100%)
    • Applied to traffic already directed to the group
    • Regional-level traffic control
    • Default: 100%
  • Endpoint Weights:
    • Controls proportion of traffic to individual endpoints (0-255)
    • Endpoint-level traffic control
    • Default: 128
    • Traffic proportion = endpoint weight รท sum of all weights
  • Health Checks:
    • Standard accelerator only
    • Continuous endpoint monitoring
    • New connections route to healthy endpoints
    • Existing connections maintained until completion

5. Critical Timeouts and Limits

  • Idle Timeouts:
    • TCP connections: 340 seconds (fixed)
    • UDP connections: 30 seconds (fixed)
    • Not customizable
    • Requires at least 1 byte of data within timeout window
  • IP Address Persistence:
    • Static IPs remain assigned for accelerator lifetime
    • Persist even when accelerator is disabled
    • Lost permanently when accelerator is deleted
  • Regional Support:
    • Available in most AWS Regions
    • Some specific AZ exclusions exist
    • API operations must use us-west-2 Region

6. Integration Points

  • AWS Service Integration:
    • Amazon Application Recovery Controller (ARC)
    • VPC Block Public Access
    • AWS Shield for DDoS protection
    • AWS WAF (depends on routing method)
    • Route 53 health checkers
  • Client IP Address Handling:
    • Custom Routing: Always preserves client IP
    • Standard: Optional preservation for some endpoint types
    • Affects AWS WAF rule application

7. Primary Use Cases

  • Standard Accelerator:
    • Reducing latency for global applications
    • Improving availability with multi-region failover
    • Simplifying management with static IPs
    • Enhancing security by masking origins
  • Custom Routing Accelerator:
    • VoIP applications with specific instance mapping
    • Online gaming with player session assignment
    • Real-time applications requiring deterministic routing

8. API Operations

  • Key Differences:
    • Separate API sets for standard and custom routing accelerators
    • All operations require us-west-2 Region specification
    • API deletion requires component removal first
    • Console deletion handles component removal automatically
  • Cross-Account Support:
    • Dedicated API operations for cross-account attachments
    • Enables resource sharing across accounts

9. Most Frequently Tested Concepts

  • Critical Numbers:
    • TCP timeout: 340 seconds
    • UDP timeout: 30 seconds
    • Traffic dial range: 0-100%
    • Endpoint weight range: 0-255 (default 128)

AWS Global Accelerator - Custom Routing Accelerators Summary

1. Core Concepts and Architecture

  • Definition and Purpose:
    • Enables application-controlled routing to specific destinations
    • Maps users to specific EC2 instances and ports
    • Uses AWS global network for improved performance
    • Ideal for gaming, VoIP, and real-time collaboration
  • Component Hierarchy:
    • Accelerator: Container for the entire configuration
    • Listeners: Define port ranges Global Accelerator listens on
    • Endpoint Groups: Regional containers with destination port configurations
    • VPC Subnet Endpoints: Subnets containing EC2 instance destinations

2. Key Limitations and Requirements

  • Endpoint Restrictions:
    • EC2 instances only (no load balancers or other resources)
    • VPC subnets only (minimum /28, maximum /17)
    • IPv4 only (no dual-stack support)
    • Unsupported EC2 types: C1, CC1, CC2, CG1, CG2, CR1, CS1, G1, G2, HI1, HS1, M1, M2, M3, T1
  • Port Range Requirements:
    • Listener port ranges: Minimum 16 ports each
    • Cannot decrease existing port ranges
    • Formula for sizing: subnet size ร destination ports ร protocols
    • Add buffer for Global Accelerator requirements

3. Port Mapping and Traffic Flow

  • Port Mapping Behavior:
    • Static mapping created when subnet added
    • Maps listener ports to destination IP:port combinations
    • Mapping never changes for a specific subnet
    • Retrieved via ListCustomRoutingPortMappings API
  • Traffic Flow Process:
    • Application retrieves port mapping
    • Application directs users to specific Global Accelerator port
    • User connects to Global Accelerator static IP and port
    • Traffic routes deterministically to mapped destination

4. Traffic Management

  • Default Deny Behavior:
    • All traffic denied by default to all destinations
    • Must explicitly allow traffic to destinations
    • Allow all traffic to subnet or specific IP:port combinations
    • Changes take time to propagate
  • No Health Checks or Failover:
    • No automatic health monitoring
    • No automatic failover to healthy endpoints
    • Traffic routed deterministically regardless of health
    • Application must handle failover logic

5. Port Range Distinctions

  • Two Types of Port Ranges:
    • Listener port ranges:
      • Ports on Global Accelerator static IPs
      • Where clients connect to
      • Must include enough ports for all destinations
    • Destination port ranges:
      • Ports on EC2 instances
      • Where traffic is delivered to
      • Can specify UDP, TCP, or both for each range

6. Implementation Considerations

  • Client IP Preservation:
    • VPC subnets always preserve client IP addresses
    • Security groups must account for client IPs
    • Review security configuration for client IP preservation
  • Connection Collision Prevention:
    • Avoid sending direct internet traffic to same endpoints
    • Use dedicated instances for Global Accelerator traffic
  • Container Workloads:
    • Can use with EKS/ECS using VPC-CNI plugin
    • Routes traffic to Kubernetes Pods through ENI

7. Key API Operations

  • Essential APIs:
    • ListCustomRoutingPortMappings: Get all port mappings
    • ListCustomRoutingPortMappingsByDestination: Get mappings for specific destination
    • AllowCustomRoutingTraffic: Enable traffic to destination
    • DenyCustomRoutingTraffic: Disable traffic to destination
    • DescribeCustomRoutingAccelerator: Check propagation status

8. Most Frequently Tested Concepts

  • Key Differences from Standard Accelerators:
    • Deterministic routing vs. automatic routing
    • VPC subnets only vs. multiple endpoint types
    • No health checks or failover vs. automatic failover
    • IPv4 only vs. IPv4 or dual-stack
    • Application-controlled routing vs. AWS-controlled routing

  • IP Address Types:
    • Standard: IPv4 (2 addresses) or dual-stack (2 IPv4 + 2 IPv6)

  • Listeners:
    • Process inbound connections based on ports/protocols
    • Support TCP and UDP protocols
    • Standard: Client affinity options (None or Source IP)
    • Custom Routing: Minimum 16 ports per range
  • Endpoint Groups:
    • Regional containers for endpoints
    • Standard: Traffic dials control percentage (0-100%)
    • Custom Routing: Define destination port ranges
    • One endpoint group per Region
  • Endpoints:
    • Standard: ALB, NLB, EC2, Elastic IP
    • Custom Routing: VPC subnets with EC2 instances only
    • Standard: Weights control traffic proportion (0-255)
    • Custom Routing: All traffic denied by default

3. Cross-Account Access

  • Resource Sharing:
    • Resource owners create cross-account attachments
    • Attachments specify resources and authorized principals
    • Principals can add shared resources as endpoints
    • Principals can use shared BYOIP addresses
  • Key Requirements:
    • Resources must be owned by attachment creator
    • Cannot share resources that were shared with you
    • BYOIP addresses must be provisioned before sharing
    • Principals must specify attachment when using resources

4. DNS and Custom Domains

    • IPv4 format: a1234567890abcdef.awsglobalaccelerator.com

    • PTR records created automatically (except for BYOIP)
    • Round-robin DNS returns IPs in random order
  • Custom Domain Options:
    • A records: Point directly to static IPs
    • CNAME: Point to accelerator DNS name (not for apex domains)
    • Route 53 Alias: Works for apex and subdomains
    • Alias records automatically update if IPs change

5. BYOIP (Bring Your Own IP)

  • Requirements:
    • IPv4 only, /24 ranges only
    • Registered with ARIN, RIPE, or APNIC
    • Business/institutional entity (not individual)
    • Clean reputation history
    • Cannot be multihomed (advertised by multiple providers)
  • Authorization Process:
    • Create ROA to authorize Amazon ASN 16509
    • Create self-signed X.509 certificate
    • Add certificate to RDAP record
    • Create signed authorization message
    • Provision and advertise address range

6. Client IP Address Preservation

AWS Global Accelerator - How It Works

Traffic Flow Overview

  • Basic Operation:
    • Static IP addresses serve as fixed entry points
    • Traffic enters AWS global network at closest edge location
    • Routed over optimized AWS backbone network
    • Directed to appropriate regional endpoints
  • Routing Logic by Accelerator Type:
    • Standard Accelerator:
      • Routes to optimal endpoint based on multiple factors
      • Considers user location, endpoint health, configured weights
      • Optimizes for performance and availability
    • Custom Routing Accelerator:
      • Routes each client to specific EC2 instance and port
      • Based on external IP address and listener port mapping
      • Deterministic routing with no automatic failover

Performance Optimization

  • TCP Connection Handling:
    • Terminates TCP connections at AWS edge locations
    • Establishes new TCP connection with endpoints
    • Results in faster response times (lower latency)
    • Increases throughput for applications
  • Path Optimization:
    • Maximizes time traffic spends on AWS network
    • Routes over well-monitored, congestion-free paths
    • Uses redundant AWS global network
    • Ensures optimal network path selection

Client IP Address Handling

  • IP Preservation Options:
    • Custom Routing Accelerator: Always preserves client IP
    • Standard Accelerator: Optional preservation for some endpoint types
    • Depends on endpoint type and configuration
  • Requirements:
    • Specific endpoint types support preservation
    • May require additional configuration
    • Affects security group settings

Health Monitoring and Failover

  • Standard Accelerator:
    • Continuous endpoint health monitoring
    • Instant redirection of new connections from unhealthy endpoints
    • Enables high-availability architecture
    • Existing connections maintained until completion or timeout
  • Custom Routing Accelerator:
    • No health checks used
    • No automatic failover
    • Destination specified by application logic
    • Application must handle failover scenarios

Traffic Management Features

  • Traffic Dials:
    • Control percentage of traffic to endpoint groups
    • Dial up/down traffic to specific Regions
    • Useful for performance testing or stack upgrades
    • Available for standard accelerators only
  • Endpoint Weights:
    • Fine-grained control within endpoint groups
    • Specify proportion of traffic to each endpoint
    • Values from 0-255 (default: 128)
    • Available for standard accelerators only

Special Considerations

  • Endpoint Weight Overrides:
    • In limited scenarios, Global Accelerator may override weights
    • Prioritizes availability over exact weight adherence
    • Helps avoid connection collisions
    • Common with client IP preservation
  • IP Fragmentation Handling:
    • UDP fragments forwarded to configured endpoint
    • TCP fragments dropped at edge (not supported)
    • TCP MSS negotiated automatically
    • Large packets (1500+ bytes) may require fragmentation

Exam Tips

  • Key Operational Concepts:

AWS Global Accelerator - Traffic Dials and Endpoint Weights

Traffic Management Overview

  • Available Controls:
    • Two primary mechanisms for traffic management:
      • Traffic dials (endpoint group level)
      • Endpoint weights (individual endpoint level)
    • Available for standard accelerators only
    • Not applicable to custom routing accelerators

Traffic Dials

  • Purpose and Function:
    • Controls percentage of traffic to an entire endpoint group
    • Applied to traffic already directed to the endpoint group
    • Does not affect initial traffic distribution across groups
    • Regional-level traffic control
  • Configuration Details:
    • Value range: 0 to 100 (percentage)
    • Default value: 100 (100%)
    • Setting to 0 stops all traffic to the group
    • Partial values limit percentage of accepted traffic
  • Example Scenario:
    • Traffic dial set to 50% for us-east-1 endpoint group
    • 100 user requests directed to the group
    • Only 50 requests accepted by the group
    • Remaining 50 requests redirected to other regions

Endpoint Weights

  • Purpose and Function:
    • Controls proportion of traffic to individual endpoints
    • Divides traffic within an endpoint group
    • Endpoint-level traffic control
    • Useful for performance testing within a region
  • Configuration Details:
    • Value range: 0 to 255
    • Default value: 128 (half of maximum)
    • Setting to 0 stops traffic to the endpoint
    • Traffic proportion based on endpoint's weight relative to sum of all weights
  • Calculation Example:
    • Endpoint A: Weight 100
    • Endpoint B: Weight 200
    • Total weight: 300
    • Endpoint A receives: 100/300 = 33.3% of traffic
    • Endpoint B receives: 200/300 = 66.7% of traffic

Comparing Traffic Dials and Weights

  • Key Differences:
    • Scope:
      • Traffic dials: Entire endpoint group (region)
      • Weights: Individual endpoints within a group
    • Purpose:
      • Traffic dials: Limit traffic to a region
      • Weights: Distribute traffic within a region
    • Use cases:
      • Traffic dials: Regional failover, blue/green deployments
      • Weights: A/B testing, gradual rollouts within region

Weight Override Scenarios

  • Special Circumstances:
    • Global Accelerator may override endpoint weights in limited scenarios
    • Prioritizes availability over exact weight adherence
    • Helps prevent connection collisions
    • Common with client IP preservation enabled
  • Decision Logic:
    • Balance between:
      • Preserving availability for client traffic
      • Adhering to configured endpoint weights
    • Availability takes precedence when conflicts arise

Failover Impact

  • How Failover Works with Traffic Controls:
    • Unhealthy endpoints excluded from traffic distribution
    • Weights recalculated among remaining healthy endpoints
    • Traffic dials still apply to endpoint groups
    • All-unhealthy scenario routes to all endpoints regardless of weights

Exam Tips

  • Key Concepts:
    • Traffic dials: Regional percentage control (0-100)
    • Endpoint weights: Individual endpoint proportion (0-255)
    • Both features for standard accelerators only
    • Weight calculation: endpoint weight รท sum of all weights
  • Common Scenarios:
    • Regional traffic management
    • Blue/green deployments
    • A/B testing
    • Performance testing
    • Gradual rollout of changes
  • Remember:
    • Traffic dials affect traffic already directed to a region
    • Weights distribute traffic within a region
    • Default values: 100% for dials, 128 for weights
    • Weights may be overridden to preserve availability

AWS Global Accelerator - ICMP Response Messages

ICMP Fundamentals

  • Role in Internet Availability:
    • ICMP response messages ensure internet availability
    • Critical for proper network communication
    • Provides feedback for network conditions
    • Enables path MTU discovery
  • Common ICMP Message Types:
    • ICMP echo messages (pings)
    • ICMP Packet Too Big
    • Fragmentation Needed
    • Destination Unreachable

Global Accelerator ICMP Handling

  • Edge Response Behavior:
    • Responds to ICMP echo messages (pings) at the edge
    • Responds for all global IP addresses
    • Pings not forwarded to customer endpoints
    • Cannot be used for accurate performance testing
  • ICMP Traffic Control:
    • Cannot block ICMP traffic at accelerator level
    • Blocking would drop critical ICMP messages
    • Would prevent Path MTU Discovery (PMTUD)
    • Would affect TCP and other protocols

Path MTU Discovery

  • PMTUD Process:
    • Determines maximum packet size between devices
    • Maximum Transmission Unit (MTU) varies across networks
    • Larger MTU allows more data per packet
    • Prevents fragmentation or packet loss
  • Operation:
    • Oversized packets dropped by intermediate devices
    • ICMP messages notify sender about size limits
    • Sender adjusts packet size accordingly
    • Ensures optimal packet sizing

Critical ICMP Messages

  • Essential Message Types:
    • ICMPv6 Packet Too Big (Type 2)
    • Destination Unreachable: Fragmentation Needed (Type 3, Code 4)
    • Required for traffic to reach originating host
    • Necessary for PMTUD functionality
  • Impact of Blocking:
    • Drops messages needed for path discovery
    • Prevents successful communication with clients
    • Affects networks with smaller-than-typical MTU
    • Breaks PMTUD functionality

Endpoint Security Group Configuration

  • Required Configuration:
    • Endpoint security groups must allow ICMP traffic
    • Required for PMTUD to work properly
    • Prevents availability issues on specific networks
    • Especially important for networks with non-standard MTU
  • Troubleshooting:
    • For network-specific availability issues:
    • Verify endpoint security groups allow ICMP traffic
    • Check for ICMP filtering at any network layer
    • Ensure PMTUD can function properly

Performance Testing

  • Testing Recommendations:
    • Don't use ping (ICMP echo) for performance testing
    • Pings are handled at edge, not forwarded to endpoints
    • Use deeper protocol tests instead
    • TCP/HTTP tests provide more accurate performance metrics

Exam Tips

  • Key Points:
    • ICMP traffic cannot be blocked at accelerator level
    • Pings are responded to at edge, not forwarded
    • Endpoint security groups must allow ICMP
    • PMTUD requires ICMP message support
  • Common Scenarios:
    • Troubleshooting network connectivity issues
    • Understanding performance testing limitations
    • Configuring security groups for proper operation
    • Diagnosing client-specific availability problems
  • Remember:
    • Don't use ping for performance testing
    • ICMP is essential for internet availability
    • Blocking ICMP breaks PMTUD
    • Security groups must be properly configured

  • Step 2: Add Listeners
    • Specify ports or port ranges (1-65535)
    • Select protocols (TCP, UDP, or both)
    • Configure client affinity (optional)
      • None: Default, no client affinity
      • Source IP: Ensures client connections route to same endpoint
    • Add additional listeners if needed
  • Step 3: Add Endpoint Groups
    • Select AWS Region for each group
    • Configure traffic dial (0-100%)
    • Optionally configure health checks
    • Add additional endpoint groups as needed
  • Step 4: Add Endpoints
    • Select endpoint type from available resources
    • Configure endpoint weight (0-255)
    • Enable/disable client IP preservation (where available)
    • Add additional endpoints as needed
  • Step 5: Test Your Accelerator
    • Wait for "Active" status
    • Test using curl or similar tool
    • Verify traffic routing to endpoints
    • Check distribution based on weights
  • Step 6: Delete Accelerator (Optional)
    • Disable accelerator first
    • Then delete accelerator
    • Note: Static IPs will be lost permanently

  • Prerequisites:

    • Open Global Accelerator console

  • Step 2: Add Listeners

  • Step 3: Add Endpoint Groups
    • Select AWS Region

    • Add additional endpoint groups as needed
  • Step 4: Add VPC Subnet Endpoints
    • Select subnet ID for endpoint
    • Configure traffic allowance:

    • Specify IP addresses and ports to allow

  • Step 5: Delete Accelerator (Optional)

AWS Global Accelerator - Listeners for Standard Accelerators

Listener Fundamentals

  • Definition and Purpose:
    • Process inbound connections from clients

    • Act as the entry point for traffic to your accelerator
    • Direct traffic to associated endpoint groups
  • Supported Protocols:
    • TCP: Transmission Control Protocol
    • UDP: User Datagram Protocol
    • Can support both protocols simultaneously
  • Port Configuration:
    • Specific ports (e.g., 80, 443)
    • Port ranges (e.g., 1024-65535)
    • Multiple port configurations possible
    • Supports ports 1-65535

Client Affinity

  • Purpose:
    • Maintains consistent routing for specific clients
    • Directs all requests from a specific source IP to same endpoint
    • Essential for stateful applications
    • Preserves session continuity
  • Configuration Options:
    • None (Default):
      • Equal traffic distribution between endpoints
      • Uses 5-tuple properties for hashing
      • Source IP, source port, destination IP, destination port, protocol
      • Optimizes for performance
    • Source IP:
      • Routes based on client source IP address
      • Uses 2-tuple properties for hashing
      • Source IP and destination IP only
      • Maintains client affinity

Client Affinity Limitations

  • Edge Location Shifts:
    • Network maintenance may shift traffic to different edge locations
    • New edge location may prefer different AWS Region
    • Client affinity not guaranteed during these shifts
    • Can affect session persistence
  • Endpoint Weight Overrides:
    • Global Accelerator may override endpoint weights in specific scenarios
    • Prioritizes availability over weight adherence
    • Common with client IP address preservation
    • Helps avoid connection collisions

Listener Management

  • Adding Listeners:

  • Editing Listeners:
    • Modify port configurations
    • Update protocol settings
    • Change client affinity settings
    • Add or remove endpoint group associations
  • Removing Listeners:
    • Removes the entry point for specified ports/protocols
    • Does not affect other listeners
    • Does not delete associated endpoint groups

AWS Global Accelerator - Endpoints for Standard Accelerators

Endpoint Fundamentals

  • Definition and Types:
    • Resources that receive traffic from Global Accelerator
    • Supported endpoint types:
      • Network Load Balancers
      • Application Load Balancers
      • Amazon EC2 instances
      • Elastic IP addresses
    • Must be valid and active when added
    • Can be added to multiple endpoint groups (different listeners)
  • Traffic Distribution:
    • Routes only to healthy endpoints
    • Continuously monitors endpoint health
    • If no healthy endpoints, routes to all endpoints
    • Proportional routing based on endpoint weights

Endpoint Requirements

  • Application Load Balancer Requirements:
    • Can be internet-facing or internal
    • Dual-stack ALBs supported
    • Must run inside AWS Region (not Local Zone)
  • Network Load Balancer Requirements:
    • Can be internet-facing or internal
    • Dual-stack NLBs have restrictions:
      • For dual-stack accelerators: Cannot have target group with target type of ip
      • For dual-stack accelerators: Cannot have target type of instance with IPv6 address type
      • For IPv4 accelerators: Cannot enable client IP preservation
    • Must run inside AWS Region (not Local Zone)
    • Disable cross-zone traffic to avoid connection collisions
    • No shared subnet support with client IP preservation
  • EC2 Instance Requirements:
    • Cannot be instance types: C1, CC1, CC2, CG1, CG2, CR1, CS1, G1, G2, HI1, HS1, M1, M2, M3, or T1
    • Supported only in specific AWS Regions
    • Must be inside AWS Region (not Local Zone)
    • Remove from endpoint group before termination
    • Dual-stack instances need primary IPv6 ENI
  • Elastic IP Address Requirements:
    • Dual-stack Elastic IPs not supported
  • General Requirements:
    • Avoid sending direct internet traffic to same endpoints
    • Resources must be owned by same account as accelerator (unless using cross-account)
    • Dual-stack accelerators require dual-stack endpoints

Endpoint Weights

  • Purpose and Function:
    • Controls proportion of traffic to individual endpoints
    • Endpoint-level traffic control
    • Enables weighted routing within endpoint group
    • Useful for load balancing and testing
  • Configuration Details:
    • Value range: 0 to 255
    • Default value: 128 (half of maximum)
    • Setting to 0 stops traffic to the endpoint
    • Traffic proportion = endpoint weight รท sum of all weights
  • Example Calculation:
    • Endpoint A: Weight 1
    • Endpoint B: Weight 255
    • Total weight: 256
    • Endpoint A receives: 1/256 = 0.39% of traffic
    • Endpoint B receives: 255/256 = 99.61% of traffic
  • Weight Override Scenarios:
    • Global Accelerator may override weights in specific scenarios
    • Prioritizes availability over weight adherence
    • Common with client IP preservation
    • Helps avoid connection collisions

Failover Behavior

  • No Healthy Endpoints with Weight > 0:
    • Tries to fail over to healthy endpoint in another endpoint group
    • Ignores traffic dial settings during failover
    • Attempts three closest endpoint groups
    • If no healthy endpoint found, routes to random endpoint in closest group
  • Recovery Behavior:
    • Returns to regular routing when Regions healthy again
    • Typically resumes in about 30 seconds
    • Established connections not moved
    • Existing connections continue until reset or terminated

Connection Collision Prevention

  • Collision Scenarios:
    • Same resource as endpoint with multiple accelerators
    • Resources receiving both accelerator and direct internet traffic
    • NLB endpoints with cross-zone traffic enabled
  • Prevention Methods:
    • Use port overrides to map to different destination ports
    • Disable cross-zone traffic for NLBs
    • Configure different port mappings for multiple accelerators
    • Use separate ports for direct and accelerator traffic
  • Example Configuration:
    • Accelerator-A: Listener TCP:443 โ Override to 8443
    • Accelerator-B: Listener TCP:443 โ Override to 9443
    • ALB endpoint: Configure listeners on ports 8443 and 9443
    • Result: Traffic from both accelerators reaches ALB without collisions

AWS Global Accelerator - DNS Addressing and Custom Domains

DNS Addressing Fundamentals

  • Default DNS Assignment:
    • Global Accelerator assigns default DNS name to each accelerator
    • IPv4 accelerator format: a1234567890abcdef.awsglobalaccelerator.com
    • Dual-stack accelerator format: a1234567890abcdef.dualstack.awsglobalaccelerator.com
    • DNS name points to static IP addresses assigned to accelerator
  • DNS Resolution Process:
    • DNS servers use round-robin DNS to resolve accelerator name
    • Returns static IP addresses in random order
    • Clients typically use first IP address returned
    • Provides automatic load balancing at DNS level
  • Reverse DNS (PTR Records):
    • Global Accelerator creates PTR record for each IP address
    • Maps static IP address to corresponding DNS name
    • Supports reverse DNS lookup
    • Not created for BYOIP addresses
    • Cannot be configured to point to custom domain names

Custom Domain Configuration

  • Use Cases:
    • Brand consistency with existing domain names
    • Simplified DNS management
    • Easier migration from existing services
    • User-friendly URLs
  • Configuration Methods:
    • Direct IP Method:
      • Create A records pointing to Global Accelerator static IPs
      • Works with any DNS provider
      • Requires manual updates if IPs change
    • CNAME Method:
      • Create CNAME record pointing to accelerator DNS name
      • Automatically updates if IPs change
      • Cannot be used for apex/root domain (example.com)
    • Route 53 Alias Method:
      • Create alias record pointing to accelerator
      • Works for both apex and subdomains
      • No additional charge for alias queries
      • Automatically updates if IPs change

Route 53 Alias Configuration

  • Benefits of Alias Records:
    • Route 53 extension to DNS (similar to CNAME)
    • Works for apex/root domain (example.com)
    • Works for subdomains (www.example.com)
    • No additional charge for alias queries
    • Automatically updates if accelerator IPs change
  • Configuration Steps:
    • Create hosted zone in Route 53 for your domain
    • Create alias record in hosted zone
    • Select "AWS Global Accelerator accelerators" as record type
    • Select your accelerator from dropdown list
    • Wait for DNS propagation

BYOIP (Bring Your Own IP) Overview

  • Purpose and Benefits:
    • Use your own public IPv4 address ranges with Global Accelerator
    • Maintain ownership of address ranges
    • AWS advertises ranges on the internet
    • Provides consistency with existing IP-based security policies
    • Simplifies migration of existing services
  • Key Limitations:
    • IPv4 only (IPv6 not supported)
    • Only /24 address ranges supported
    • Cannot use same IP addresses for multiple AWS services
    • Address range must have clean reputation history
    • Must be registered with ARIN, RIPE, or APNIC
  • IP Address Assignment:
    • Appears as address pool in your account
    • Can assign one IP address from range to accelerator
    • Second IP comes from Amazon IP address range
    • Each address range assigned to different network zone

Exam Tips

  • DNS Addressing:
    • Default DNS name format for different accelerator types

AWS Global Accelerator - Client IP Address Preservation

Introduction to Client IP Preservation

  • Definition and Purpose:
    • Preserves original client IP address in packets arriving at endpoints
    • Allows endpoints to see actual source IP of client requests
    • Enables IP-based security policies and analytics
    • Provides visibility into client geographic distribution
  • Supported Endpoint Types:
    • Application Load Balancers (internal and internet-facing)
    • Network Load Balancers with security groups
    • Amazon EC2 instances
    • Custom routing accelerator endpoints (always preserved)
  • Unsupported Endpoint Types:
    • Network Load Balancers without security groups
    • Elastic IP addresses

Guidelines and Restrictions

  • Security Configuration:
    • Update security groups to include client IP addresses before enabling
    • AWS WAF will see client IPs instead of Global Accelerator IPs
    • Security groups must be configured for client IP ranges
  • Regional Support:
    • Supported in all AWS Regions where Global Accelerator is available
  • Default Behavior:
    • New accelerators in console: Enabled by default for supported endpoints
    • Internet-facing ALBs: Can be enabled or disabled
    • Internal ALBs and EC2 instances: Always enabled (cannot be disabled)
    • NLBs with security groups: Not enabled by default
  • API/CLI Default Behavior:
    • Internet-facing ALBs: Enabled by default
    • NLBs with security groups: Not enabled by default

Endpoint-Specific Requirements

  • Network Load Balancer Restrictions:
    • Not supported for NLBs without security groups
    • Not supported for NLBs with security groups that have TLS listeners
    • Not supported for NLBs with IPv4 to IPv6 NAT translation
    • Targets must be in same VPC as the NLB
    • Traffic must flow directly from NLB to target
  • Elastic Network Interfaces:
    • Global Accelerator creates ENIs in your account
    • One ENI for each subnet where an endpoint is present
    • ENIs facilitate client IP preservation
  • Private Subnet Requirements:
    • VPC must have internet gateway attached
    • Use private subnets to ensure traffic only from Global Accelerator
    • Configure security groups to allow appropriate traffic

Network Configuration

  • Security Group Configuration:
    • Update to include client IP addresses on allow lists
    • Remove Global Accelerator IPs from allow lists
    • EC2 security groups apply to all traffic including Global Accelerator
  • Network ACL Configuration:
    • Apply to egress (outbound) traffic with client IP preservation
    • Must configure both inbound and outbound rules
    • Example: Allow TCP/UDP ephemeral ports (1024-65535)
    • Create matching inbound and outbound rules
  • AWS WAF Integration:
    • WAF sees client IP addresses instead of Global Accelerator IPs
    • Update WAF rules to account for client IPs

Transitioning to Client IP Preservation

  • Recommended Approach:
    • Transition slowly to minimize disruption
    • Add new endpoint with client IP preservation enabled
    • Gradually shift traffic using endpoint weights
    • Test with small traffic percentage first
  • Transition Steps:
    • 1. Add duplicate endpoint with client IP preservation enabled
    • 2. Set low weight for new endpoint (e.g., 5)
    • 3. Test with small traffic percentage
    • 4. Gradually increase weight for new endpoint
    • 5. Gradually decrease weight for original endpoint
    • 6. Set original endpoint weight to 0 when transition complete
  • Reverting Changes:
    • Increase weight for non-preserving endpoint
    • Decrease weight for preserving endpoint to 0

Exam Tips

  • Key Concepts:
    • Supported endpoints: ALBs, NLBs with security groups, EC2 instances
    • Unsupported endpoints: NLBs without security groups, Elastic IPs
    • Internal ALBs and EC2 instances always preserve client IPs
    • Custom routing accelerators always preserve client IPs
  • Security Considerations:
    • Update security groups before enabling
    • Configure both inbound and outbound network ACL rules
    • AWS WAF sees client IPs instead of Global Accelerator IPs
  • Remember:
    • VPC must have internet gateway attached
    • Global Accelerator creates ENIs in your account
    • Transition slowly using endpoint weights
    • Test with small traffic percentage first
