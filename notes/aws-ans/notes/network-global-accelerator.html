AWS Global Accelerator - Comprehensive Exam Guide Summary

1. Core Concepts

  • Service Definition:
    • AWS service that improves application performance and availability
    • Routes traffic over AWS global network instead of public internet
    • Provides static IP addresses as entry points
    • Two types: Standard and Custom Routing accelerators
  • Key Components:
    • Static IP addresses: 2 IPv4 or 4 addresses (2 IPv4 + 2 IPv6)
    • Network zones: Isolated infrastructure units for fault isolation
    • Listeners: Process inbound connections based on ports/protocols
    • Endpoint groups: Regional collections of endpoints
    • Endpoints: Resources that receive traffic (ALB, NLB, EC2, EIP, VPC subnets)

2. Accelerator Types

  • Standard Accelerator:
    • Routes based on proximity, health, and weights
    • Supports all endpoint types
    • Includes health checks and automatic failover
    • Supports traffic dials and endpoint weights
    • Ideal for general web applications and APIs
  • Custom Routing Accelerator:
    • Deterministic routing to specific EC2 instances and ports
    • Supports VPC subnet endpoints only
    • No health checks or automatic failover
    • IPv4 only (no dual-stack)
    • Ideal for VoIP, gaming, and real-time applications

3. Security and Connectivity

  • VPC Connectivity:
    • Requires internet gateway attached to VPC
    • No public IP addresses needed on resources
    • No internet gateway route required for subnet
    • Traffic returns through Global Accelerator, not internet gateway
  • Security Features:
    • Origin masking behind static IPs
    • AWS Shield protection for static IPs
    • Private VPC connections using private IPs
    • Security groups still apply to all traffic
  • Protocol Requirements:
    • TLS 1.2 minimum (1.3 recommended)
    • Perfect Forward Secrecy cipher suites (DHE/ECDHE)
    • ICMP traffic cannot be blocked at accelerator level

4. Traffic Management

  • Traffic Dials:
    • Controls percentage of traffic to endpoint group (0-100%)
    • Applied to traffic already directed to the group
    • Regional-level traffic control
    • Default: 100%
  • Endpoint Weights:
    • Controls proportion of traffic to individual endpoints (0-255)
    • Endpoint-level traffic control
    • Default: 128
    • Traffic proportion = endpoint weight รท sum of all weights
  • Health Checks:
    • Standard accelerator only
    • Continuous endpoint monitoring
    • New connections route to healthy endpoints
    • Existing connections maintained until completion

5. Critical Timeouts and Limits

  • Idle Timeouts:
    • TCP connections: 340 seconds (fixed)
    • UDP connections: 30 seconds (fixed)
    • Not customizable
    • Requires at least 1 byte of data within timeout window
  • IP Address Persistence:
    • Static IPs remain assigned for accelerator lifetime
    • Persist even when accelerator is disabled
    • Lost permanently when accelerator is deleted
  • Regional Support:
    • Available in most AWS Regions
    • Some specific AZ exclusions exist
    • API operations must use us-west-2 Region

6. Integration Points

  • AWS Service Integration:
    • Amazon Application Recovery Controller (ARC)
    • VPC Block Public Access
    • AWS Shield for DDoS protection
    • AWS WAF (depends on routing method)
    • Route 53 health checkers
  • Client IP Address Handling:
    • Custom Routing: Always preserves client IP
    • Standard: Optional preservation for some endpoint types
    • Affects AWS WAF rule application

7. Primary Use Cases

  • Standard Accelerator:
    • Reducing latency for global applications
    • Improving availability with multi-region failover
    • Simplifying management with static IPs
    • Enhancing security by masking origins
  • Custom Routing Accelerator:
    • VoIP applications with specific instance mapping
    • Online gaming with player session assignment
    • Real-time applications requiring deterministic routing

8. API Operations

  • Key Differences:
    • Separate API sets for standard and custom routing accelerators
    • All operations require us-west-2 Region specification
    • API deletion requires component removal first
    • Console deletion handles component removal automatically
  • Cross-Account Support:
    • Dedicated API operations for cross-account attachments
    • Enables resource sharing across accounts

9. Most Frequently Tested Concepts

  • Critical Numbers:
    • TCP timeout: 340 seconds
    • UDP timeout: 30 seconds
    • Traffic dial range: 0-100%
    • Endpoint weight range: 0-255 (default 128)
    • Static IPs: 2 IPv4 or 4 addresses (2 IPv4 + 2 IPv6)
  • Key Distinctions:
    • Standard vs. Custom Routing accelerator capabilities
    • Traffic dials (regional) vs. Endpoint weights (individual)
    • Client IP preservation options and implications
    • Health check behavior and failover scenarios

AWS Global Accelerator - How It Works

Traffic Flow Overview

  • Basic Operation:
    • Static IP addresses serve as fixed entry points
    • Traffic enters AWS global network at closest edge location
    • Routed over optimized AWS backbone network
    • Directed to appropriate regional endpoints
  • Routing Logic by Accelerator Type:
    • Standard Accelerator:
      • Routes to optimal endpoint based on multiple factors
      • Considers user location, endpoint health, configured weights
      • Optimizes for performance and availability
    • Custom Routing Accelerator:
      • Routes each client to specific EC2 instance and port
      • Based on external IP address and listener port mapping
      • Deterministic routing with no automatic failover

Performance Optimization

  • TCP Connection Handling:
    • Terminates TCP connections at AWS edge locations
    • Establishes new TCP connection with endpoints
    • Results in faster response times (lower latency)
    • Increases throughput for applications
  • Path Optimization:
    • Maximizes time traffic spends on AWS network
    • Routes over well-monitored, congestion-free paths
    • Uses redundant AWS global network
    • Ensures optimal network path selection

Client IP Address Handling

  • IP Preservation Options:
    • Custom Routing Accelerator: Always preserves client IP
    • Standard Accelerator: Optional preservation for some endpoint types
    • Depends on endpoint type and configuration
  • Requirements:
    • Specific endpoint types support preservation
    • May require additional configuration
    • Affects security group settings

Health Monitoring and Failover

  • Standard Accelerator:
    • Continuous endpoint health monitoring
    • Instant redirection of new connections from unhealthy endpoints
    • Enables high-availability architecture
    • Existing connections maintained until completion or timeout
  • Custom Routing Accelerator:
    • No health checks used
    • No automatic failover
    • Destination specified by application logic
    • Application must handle failover scenarios

Traffic Management Features

  • Traffic Dials:
    • Control percentage of traffic to endpoint groups
    • Dial up/down traffic to specific Regions
    • Useful for performance testing or stack upgrades
    • Available for standard accelerators only
  • Endpoint Weights:
    • Fine-grained control within endpoint groups
    • Specify proportion of traffic to each endpoint
    • Values from 0-255 (default: 128)
    • Available for standard accelerators only

Special Considerations

  • Endpoint Weight Overrides:
    • In limited scenarios, Global Accelerator may override weights
    • Prioritizes availability over exact weight adherence
    • Helps avoid connection collisions
    • Common with client IP preservation
  • IP Fragmentation Handling:
    • UDP fragments forwarded to configured endpoint
    • TCP fragments dropped at edge (not supported)
    • TCP MSS negotiated automatically
    • Large packets (1500+ bytes) may require fragmentation

Exam Tips

  • Key Operational Concepts:
    • Traffic enters at edge, travels over AWS backbone
    • Standard accelerator optimizes for performance and health
    • Custom routing accelerator provides deterministic mapping
    • TCP connections terminated at edge for performance
  • Remember:
    • Health checks only apply to standard accelerators
    • Client IP preservation varies by accelerator type
    • Traffic management features (dials/weights) for standard accelerators only
    • TCP fragments dropped, UDP fragments supported

AWS Global Accelerator - Types and Idle Timeout

Accelerator Types in Detail

  • Standard Accelerator:
    • Optimizes for availability and performance
    • Routes based on:
      • Geographic proximity (lowest latency)
      • Endpoint health status
      • Configured weights and traffic dials
    • Ideal for:
      • General web applications
      • APIs and microservices
      • Multi-region deployments
      • High-availability architectures
    • Supports traffic management features:
      • Blue/green deployments
      • A/B testing
      • Traffic shifting across regions
  • Custom Routing Accelerator:
    • Provides deterministic routing to specific destinations
    • Maps users to specific EC2 instances and ports
    • No automatic health-based routing
    • Ideal for:
      • VoIP applications
      • Online gaming (player session mapping)
      • Real-time applications requiring specific instance routing
    • Limitations:
      • IPv4 support only (no dual-stack)
      • VPC subnet endpoints only
      • No automatic failover

Idle Timeout Behavior

  • Timeout Settings:
    • TCP connections: 340 seconds (fixed)
    • UDP connections: 30 seconds (fixed)
    • Not customizable
  • Connection Management:
    • Global Accelerator closes connection after idle timeout
    • Requires at least 1 byte of data within timeout window
    • Data can be in either direction (ingress or egress)
    • TCP keep-alive packets insufficient to maintain connection
  • Traffic Routing Behavior:
    • Continues routing established connections to endpoint until:
      • Idle timeout is reached
      • Connection naturally terminates
    • Even if endpoint is marked unhealthy or removed
    • New endpoint selection only occurs:
      • When new connection starts
      • After idle timeout occurs

Exam Tips

  • Key Differences Between Types:
    • Standard: Performance optimization, automatic health routing
    • Custom: Deterministic mapping, application-controlled routing
    • Standard: Supports all endpoint types
    • Custom: VPC subnets only
  • Timeout Considerations:
    • Fixed values: 340s for TCP, 30s for UDP
    • Cannot be modified
    • TCP keep-alive packets don't prevent timeout
    • Requires actual data transmission
  • Common Scenarios:
    • Choosing accelerator type based on application needs
    • Understanding connection persistence during endpoint changes
    • Planning for timeout-related connection behavior
    • Implementing proper keep-alive strategies

AWS Global Accelerator - Traffic Dials and Endpoint Weights

Traffic Management Overview

  • Available Controls:
    • Two primary mechanisms for traffic management:
      • Traffic dials (endpoint group level)
      • Endpoint weights (individual endpoint level)
    • Available for standard accelerators only
    • Not applicable to custom routing accelerators

Traffic Dials

  • Purpose and Function:
    • Controls percentage of traffic to an entire endpoint group
    • Applied to traffic already directed to the endpoint group
    • Does not affect initial traffic distribution across groups
    • Regional-level traffic control
  • Configuration Details:
    • Value range: 0 to 100 (percentage)
    • Default value: 100 (100%)
    • Setting to 0 stops all traffic to the group
    • Partial values limit percentage of accepted traffic
  • Example Scenario:
    • Traffic dial set to 50% for us-east-1 endpoint group
    • 100 user requests directed to the group
    • Only 50 requests accepted by the group
    • Remaining 50 requests redirected to other regions

Endpoint Weights

  • Purpose and Function:
    • Controls proportion of traffic to individual endpoints
    • Divides traffic within an endpoint group
    • Endpoint-level traffic control
    • Useful for performance testing within a region
  • Configuration Details:
    • Value range: 0 to 255
    • Default value: 128 (half of maximum)
    • Setting to 0 stops traffic to the endpoint
    • Traffic proportion based on endpoint's weight relative to sum of all weights
  • Calculation Example:
    • Endpoint A: Weight 100
    • Endpoint B: Weight 200
    • Total weight: 300
    • Endpoint A receives: 100/300 = 33.3% of traffic
    • Endpoint B receives: 200/300 = 66.7% of traffic

Comparing Traffic Dials and Weights

  • Key Differences:
    • Scope:
      • Traffic dials: Entire endpoint group (region)
      • Weights: Individual endpoints within a group
    • Purpose:
      • Traffic dials: Limit traffic to a region
      • Weights: Distribute traffic within a region
    • Use cases:
      • Traffic dials: Regional failover, blue/green deployments
      • Weights: A/B testing, gradual rollouts within region

Weight Override Scenarios

  • Special Circumstances:
    • Global Accelerator may override endpoint weights in limited scenarios
    • Prioritizes availability over exact weight adherence
    • Helps prevent connection collisions
    • Common with client IP preservation enabled
  • Decision Logic:
    • Balance between:
      • Preserving availability for client traffic
      • Adhering to configured endpoint weights
    • Availability takes precedence when conflicts arise

Failover Impact

  • How Failover Works with Traffic Controls:
    • Unhealthy endpoints excluded from traffic distribution
    • Weights recalculated among remaining healthy endpoints
    • Traffic dials still apply to endpoint groups
    • All-unhealthy scenario routes to all endpoints regardless of weights

Exam Tips

  • Key Concepts:
    • Traffic dials: Regional percentage control (0-100)
    • Endpoint weights: Individual endpoint proportion (0-255)
    • Both features for standard accelerators only
    • Weight calculation: endpoint weight รท sum of all weights
  • Common Scenarios:
    • Regional traffic management
    • Blue/green deployments
    • A/B testing
    • Performance testing

AWS Global Accelerator - ICMP Response Messages

ICMP Fundamentals

  • Role in Internet Availability:
    • ICMP response messages ensure internet availability
    • Critical for proper network communication
    • Provides feedback for network conditions
    • Enables path MTU discovery
  • Common ICMP Message Types:
    • ICMP echo messages (pings)
    • ICMP Packet Too Big
    • Fragmentation Needed
    • Destination Unreachable

Global Accelerator ICMP Handling

  • Edge Response Behavior:
    • Responds to ICMP echo messages (pings) at the edge
    • Responds for all global IP addresses
    • Pings not forwarded to customer endpoints
    • Cannot be used for accurate performance testing
  • ICMP Traffic Control:
    • Cannot block ICMP traffic at accelerator level
    • Blocking would drop critical ICMP messages
    • Would prevent Path MTU Discovery (PMTUD)
    • Would affect TCP and other protocols

Path MTU Discovery

  • PMTUD Process:
    • Determines maximum packet size between devices
    • Maximum Transmission Unit (MTU) varies across networks
    • Larger MTU allows more data per packet
    • Prevents fragmentation or packet loss
  • Operation:
    • Oversized packets dropped by intermediate devices
    • ICMP messages notify sender about size limits
    • Sender adjusts packet size accordingly
    • Ensures optimal packet sizing

Critical ICMP Messages

  • Essential Message Types:
    • ICMPv6 Packet Too Big (Type 2)
    • Destination Unreachable: Fragmentation Needed (Type 3, Code 4)
    • Required for traffic to reach originating host
    • Necessary for PMTUD functionality
  • Impact of Blocking:
    • Drops messages needed for path discovery
    • Prevents successful communication with clients
    • Affects networks with smaller-than-typical MTU
    • Breaks PMTUD functionality

Endpoint Security Group Configuration

  • Required Configuration:
    • Endpoint security groups must allow ICMP traffic
    • Required for PMTUD to work properly
    • Prevents availability issues on specific networks
    • Especially important for networks with non-standard MTU
  • Troubleshooting:
    • For network-specific availability issues:
    • Verify endpoint security groups allow ICMP traffic
    • Check for ICMP filtering at any network layer
    • Ensure PMTUD can function properly

Performance Testing

  • Testing Recommendations:
    • Don't use ping (ICMP echo) for performance testing
    • Pings are handled at edge, not forwarded to endpoints
    • Use deeper protocol tests instead
    • TCP/HTTP tests provide more accurate performance metrics

Exam Tips

  • Key Points:
    • ICMP traffic cannot be blocked at accelerator level
    • Pings are responded to at edge, not forwarded
    • Endpoint security groups must allow ICMP
    • PMTUD requires ICMP message support
  • Common Scenarios:
    • Troubleshooting network connectivity issues
    • Understanding performance testing limitations
    • Configuring security groups for proper operation
    • Diagnosing client-specific availability problems
  • Remember:
    • Don't use ping for performance testing
    • ICMP is essential for internet availability
    • Blocking ICMP breaks PMTUD
    • Security groups must be properly configured

AWS Global Accelerator - Getting Started

Getting Started with Standard Accelerator

  • Prerequisites:
    • Create resources to use as endpoints:
      • EC2 instances
      • Network Load Balancers
      • Application Load Balancers
      • Elastic IP addresses
    • Ensure VPC has internet gateway attached
    • Configure Route 53 health checker access
  • Step 1: Create a Standard Accelerator
    • Open Global Accelerator console
    • Choose "Create accelerator"
    • Provide accelerator name
    • Select "Standard" accelerator type
    • Choose IP address type (IPv4 or Dual-stack)
    • Optionally add tags
  • Step 2: Add Listeners
    • Specify ports or port ranges (1-65535)
    • Select protocols (TCP, UDP, or both)
    • Configure client affinity (optional)
      • None: Default, no client affinity
      • Source IP: Ensures client connections route to same endpoint
    • Add additional listeners if needed
  • Step 3: Add Endpoint Groups
    • Select AWS Region for each group
    • Configure traffic dial (0-100%)
    • Optionally configure health checks
    • Add additional endpoint groups as needed
  • Step 4: Add Endpoints
    • Select endpoint type from available resources
    • Configure endpoint weight (0-255)
    • Enable/disable client IP preservation (where available)
    • Add additional endpoints as needed
  • Step 5: Test Your Accelerator
    • Wait for "Active" status
    • Test using curl or similar tool
    • Verify traffic routing to endpoints
    • Check distribution based on weights
  • Step 6: Delete Accelerator (Optional)
    • Disable accelerator first
    • Then delete accelerator
    • Note: Static IPs will be lost permanently

Getting Started with Custom Routing Accelerator

  • Prerequisites:
    • Create VPC subnet for endpoint
    • Optionally launch EC2 instances in subnet
    • Ensure VPC has internet gateway attached
    • Review guidelines and restrictions
  • Step 1: Create a Custom Routing Accelerator
    • Open Global Accelerator console
    • Choose "Create accelerator"
    • Provide accelerator name
    • Select "Custom routing" accelerator type
    • Optionally add tags
  • Step 2: Add Listeners
    • Specify port ranges (minimum 16 ports)
    • Select protocols (TCP, UDP, or both)
    • Add additional listeners if needed
    • Note: Larger port ranges provide more flexibility
  • Step 3: Add Endpoint Groups
    • Select AWS Region
    • Configure port ranges and protocols for EC2 instances
    • Specify From port and To port
    • Select protocol for each range
    • Add additional endpoint groups as needed
  • Step 4: Add VPC Subnet Endpoints
    • Select subnet ID for endpoint
    • Configure traffic allowance:
      • Allow all traffic, or
      • Allow traffic to specific destinations
    • Specify IP addresses and ports to allow
    • Note: By default, all traffic is denied
  • Step 5: Delete Accelerator (Optional)
    • Disable accelerator first
    • Then delete accelerator
    • Note: Static IPs will be lost permanently

Important Considerations

  • Static IP Address Management:
    • IPs remain assigned for accelerator lifetime
    • Persist even when accelerator is disabled
    • Lost permanently when accelerator is deleted
    • Use IAM policies to prevent accidental deletion
  • EC2 Instance Handling:
    • If EC2 instance is terminated before removal from endpoint group:
    • New instance with same IP will receive traffic if health checks pass
    • Remove EC2 instance from endpoint group before termination

Exam Tips

  • Key Differences in Setup:
    • Standard: Endpoints are specific resources
    • Custom routing: Endpoints are VPC subnets
    • Standard: Health checks used
    • Custom routing: Traffic allowance configuration
  • Remember:
    • VPC requires internet gateway for both types
    • Custom routing requires minimum 16 ports per range
    • Standard accelerator supports client affinity
    • Deleting accelerator permanently loses static IPs
  • Console vs. API Differences:
    • Console: Can delete without removing components
    • API: Must remove listeners and endpoint groups first
    • Both require disabling accelerator before deletion
