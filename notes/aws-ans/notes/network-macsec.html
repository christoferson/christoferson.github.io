AWS Direct Connect MACsec - Key Concepts Summary

MACsec Fundamentals

  • Key Components:
    • CAK (Connectivity Association Key): Pre-shared secret key
    • CKN (Connectivity Association Key Name): Identifies which CAK is being used
    • SAK (Secure Association Key): Session keys derived from CAK
    • MKA (MACsec Key Agreement): Protocol for key exchange
  • Critical Specifications:
    • CKN/CAK Length: 32 Octets (64 Hex digits)
    • Cryptographic Algorithm: AES-CMAC-256
    • Cipher Suite: GCM-AES-XPN-256 (100Gbps)
    • Cipher Suite Options: GCM-AES-XPN-256 and GCM-AES-256 (10Gbps)

Cipher Suite Details

  • Evolution:
    • Original: GCM-AES-128
    • Extended: GCM-AES-256
    • Advanced: GCM-AES-XPN-128 and GCM-AES-XPN-256
  • XPN Advantage:
    • Addresses packet number exhaustion
    • Extends 32-bit packet number field
    • Prevents frequent key rollovers on high-speed links
    • Critical for 100G interfaces (prevents 5-minute SA cycles)

Exam Tips

  • Remember:
    • CKN/CAK must be exactly 32 Octets (64 Hex digits)
    • AWS uses GCM-AES-XPN-256 for 100Gbps connections
    • 10Gbps connections support both GCM-AES-XPN-256 and GCM-AES-256
    • MKA control packets use AES-CMAC-256
  • Key Relationships:
    • CAK is the pre-shared secret
    • CKN identifies which CAK is in use
    • SAKs are derived from CAK for session encryption
    • MKA distributes SAKs in control packets (MKPDUs)

AWS Direct Connect MACsec - Configuration Summary

Prerequisites and Requirements

  • Connection Requirements:
    • MACsec-capable Direct Connect circuit
    • Dedicated connection transparent to Layer 2 traffic
    • Direct Layer 2 adjacency with AWS Direct Connect device
    • Last-mile provider verification for MACsec compatibility
  • Equipment Requirements:
    • MACsec-capable on-premises equipment
    • Support for AWS default configuration options
    • Ability to configure matching CKN/CAK parameters

Supported Topologies

  • Topology 1: Router-Switch-Direct Connect Device
    • MACsec between on-premises switch and AWS Direct Connect device
    • IP routing handled by on-premises router
    • Switch must support MACsec configuration
  • Topology 2: Router-Direct Connect Device
    • MACsec directly between on-premises router and AWS Direct Connect device
    • Router must support MACsec configuration
    • Simpler topology with fewer components
  • LAG Support:
    • Both topologies support Link Aggregation Groups
    • All LAG member connections must support MACsec
    • MACsec configuration at LAG level, not connection level

Configuration Workflow

  • Step 1: Create MACsec-enabled connection
    • Use AWS Console or CreateConnection API with requestMACSec field
    • Example CLI: aws directconnect create-connection --location IAD66 --bandwidth 10Gbps --connection-name "MACsec connection" --request-mac-sec
  • Step 2: Associate CKN/CAK with connection
    • Use AssociateMacSecKey API
    • Two options:
      • Reference existing AWS Secrets Manager secret
      • Directly specify CKN/CAK values (creates secret automatically)
  • Step 3: Verify connection status
    • Use DescribeConnections API to check:
      • MACsec support status
      • Encryption port status
      • Current CKN in use
      • Encryption mode
      • Associated MACsec keys
  • Step 4: Migrate traffic
    • Associate existing VIFs using AssociateVIF API
    • Or build additional VIFs and update routing

Encryption Modes

  • Available Options:
    • should_encrypt (default):
      • Attempts MKA, if successful sends encrypted traffic
      • Falls back to unencrypted if MKA fails
    • must_encrypt:
      • Attempts MKA, if successful sends encrypted traffic
      • Connection goes down if MKA fails
      • Authentication retried after timeout
    • no_encrypt:
      • No MKA performed
      • MKA frames ignored
      • Only unencrypted frames sent/received

Exam Tips

  • Remember:
    • Default encryption mode is "should_encrypt"
    • LAG configuration is at LAG level, not connection level
    • Layer 2 adjacency required for MACsec
    • Cannot modify MACsec key after association (must disassociate and create new)
  • API Knowledge:
    • CreateConnection with requestMACSec
    • AssociateMacSecKey for key management
    • DescribeConnections for status verification
    • UpdateConnection for changing encryption mode

AWS Direct Connect MACsec Security - Study Guide Summary

Introduction to MACsec

  • What is MACsec:
    • IEEE 802.1AE standard for encrypting packets at Layer 2
    • Provides point-to-point encryption for 10 Gbps and 100 Gbps links
    • Available for dedicated connections only (not hosted)
    • Available at select Direct Connect locations
  • Key Benefits:
    • Layer 2 confidentiality and integrity
    • High-speed hardware-based encryption (near line-rate)
    • Secures all control plane protocol packets (ARP, DHCP, LLDP, BGP)
    • Ideal for regulated industries and high-bandwidth workloads

Technical Specifications

  • Security Features:
    • Confidentiality: Encrypts frame's ethertype and payload
    • Data Integrity: Ensures data cannot be modified in transit
    • Origin Authenticity: Verifies frames sent by trusted peers
    • Replay Protection: Prevents out-of-order frame processing
  • AWS Implementation Details:
    • Cipher Suite: GCM-AES-XPN-256 (100 Gbps)
    • Cipher Suite: GCM-AES-XPN-256 or GCM-AES-256 (10 Gbps)
    • MKA Control Packet Encryption: AES-CMAC-256
    • CKN/CAK Length: 32 Octets (64 Hex digits)
    • Confidentiality Offset: 0

Key Concepts

  • Secure Channel:
    • Unidirectional communication path
    • Identified by Secure Channel Identifier (SCI)
    • Two channels per connection (send/receive)
  • Secure Associations:
    • Transient sessions within secure channels
    • Contains Secure Association Key (SAK)
    • Maintains Packet Number (PN) counters
    • Up to two SAs per channel during key rotation

Exam Tips

  • Remember These Points:
    • MACsec operates at Layer 2 (unlike IPsec or TLS)
    • Requires direct Layer 2 adjacency with AWS device
    • Only available for dedicated connections (not hosted)
    • Supports 10 Gbps and 100 Gbps connections
    • Uses AWS Secrets Manager for key storage
  • Key Terminology:
    • CAK: Connectivity Association Key (pre-shared key)
    • CKN: Connectivity association Key Name
    • SAK: Secure Association Key (session key)
    • MKA: MACsec Key Agreement protocol

AWS Direct Connect MACsec - Configuration and Key Management

MACsec Configuration Process

  • Implementation Workflow:
    • 1. Create new connection with MACsec support
    • 2. Associate CKN/CAK with the connection
    • 3. Verify connection status
    • 4. Migrate traffic to new connection
  • API Commands:
    • CreateConnection: With requestMACSec parameter
    • AssociateMacSecKey: Links CKN/CAK to connection
    • DescribeConnections: Shows MACsec status
    • UpdateConnection: Changes encryption mode

Encryption Modes

  • Available Options:
    • should_encrypt (default):
      • Attempts MACsec encryption
      • Falls back to unencrypted if MACsec fails
      • Prioritizes connectivity over encryption
    • must_encrypt:
      • Requires successful MACsec
      • Connection goes down if encryption fails
      • Prioritizes security over connectivity
    • no_encrypt:
      • No MACsec attempted
      • Ignores MACsec frames
      • Only unencrypted traffic allowed

Topology Requirements

  • Supported Configurations:
    • Router-Switch-Direct Connect Device:
      • MACsec between on-premises switch and AWS
      • IP routing handled by on-premises router
    • Router-Direct Connect Device:
      • MACsec directly between router and AWS
      • Direct Layer 2 adjacency required
  • LAG Support:
    • MACsec configuration at LAG level (not connection)
    • All member connections must support MACsec
    • Same CKN/CAK used for all member connections

Key Management

  • AWS Secrets Manager Integration:
    • Automatic storage of CKN/CAK pairs
    • Uses service-linked IAM role
    • Two options for key association:
      • Direct specification of CKN/CAK values
      • Reference to existing Secret ARN
  • Key Rotation:
    • Supports hitless key rollover
    • MACsec keychain stores up to 3 CKN/CAK pairs
    • Manual rotation process (no automatic lifetime-based)
    • Keys cannot be modified after association

Exam Tips

  • Configuration Details:
    • Default encryption mode: should_encrypt
    • Maximum keychain capacity: 3 CKN/CAK pairs
    • CKN/CAK format: 32 Octets (64 Hex digits)
    • Service-linked role required for Secrets Manager
  • Common Scenarios:
    • Migrating from non-MACsec to MACsec connection
    • Configuring encryption modes
    • Key rotation procedures
    • LAG with MACsec configuration

AWS Direct Connect MACsec - Technical Implementation and Considerations

MACsec Frame Encapsulation

  • Frame Modification:
    • 16-byte SecTAG inserted before EtherType
    • 16-byte Integrity Check Value (ICV) added at frame end
    • Original EtherType and payload encrypted
    • Original Dot1q tags included in encryption
  • Packet Processing:
    • Hardware-based encryption (ASIC/PHY)
    • Near line-rate performance
    • Packet Numbers (PN) used for replay protection

Pre-Implementation Requirements

  • Hardware Compatibility:
    • MACsec-capable on-premises equipment
    • Direct Layer 2 adjacency with AWS device
    • Transparent Layer 2 last-mile provider
  • Configuration Alignment:
    • CKN/CAK format: 32 Octets (64 Hex digits)
    • Cryptographic Algorithm: AES-CMAC-256
    • SAK Cipher Suite: GCM-AES-XPN-256 (100 Gbps)
    • SAK Cipher Suite: GCM-AES-XPN-256 or GCM-AES-256 (10 Gbps)
    • Confidentiality Offset: 0
    • ICV Indicator: No
    • SAK Rekey Time: PN Rollover

Connection Monitoring

  • Status Verification:
    • DescribeConnections API shows:
      • MACsec capability status
      • Port encryption status
      • Current CKN in use
      • Encryption mode
      • Associated MACsec keys
  • Troubleshooting Indicators:
    • portEncryptionStatus field shows current state
    • macSecKeys array shows all associated keys
    • state field indicates key association status

Migration Considerations

  • Traffic Migration Options:
    • AssociateVIF API to move existing VIFs (involves downtime)
    • Build new VIFs and update routing (minimizes disruption)
    • Consider using Direct Connect Resiliency Toolkit
  • Testing Recommendations:
    • Verify encryption status before migration
    • Test with non-production traffic first
    • Prepare rollback plan

Exam Tips

  • Technical Details:
    • MACsec is hop-by-hop encryption (not end-to-end)
    • Packet Number (PN) field prevents replay attacks
    • XPN variants address packet number exhaustion
    • Hardware encryption provides near line-rate performance
  • Implementation Focus:
    • Dedicated connections only (not hosted)
    • Available at select Direct Connect locations
    • 10 Gbps and 100 Gbps links only
    • Requires compatible on-premises equipment
  • Use Cases:
    • Financial services and healthcare industries
    • High-bandwidth workloads with security requirements
    • Media production and autonomous vehicle development
    • When IPsec throughput limitations are reached

AWS Direct Connect MACsec - Comprehensive Exam Summary

1. Core Concepts

  • Definition and Purpose:
    • IEEE 802.1AE standard for Layer 2 encryption
    • Point-to-point security for 10/100 Gbps links
    • Hardware-based encryption for near line-rate performance
    • Protects all Layer 2 traffic including control protocols
  • Key Security Benefits:
    • Confidentiality: Encrypts frame payload and ethertype
    • Integrity: Prevents data modification
    • Authentication: Ensures trusted peer communication
    • Replay Protection: Prevents out-of-order frame processing

2. Technical Specifications

  • AWS Implementation:
    • 100 Gbps: GCM-AES-XPN-256 cipher suite
    • 10 Gbps: GCM-AES-XPN-256 or GCM-AES-256
    • MKA Control: AES-CMAC-256
    • CKN/CAK: 32 Octets (64 Hex digits)
    • Confidentiality Offset: 0
  • Frame Structure:
    • 16-byte SecTAG + Encrypted payload + 16-byte ICV
    • Secure Channel Identifier (SCI) in SecTAG
    • Packet Number (PN) for replay protection

3. Configuration Requirements

  • Prerequisites:
    • Dedicated connections only (not hosted)
    • Direct Layer 2 adjacency with AWS device
    • MACsec-capable customer equipment
    • Transparent Layer 2 last-mile provider
  • Supported Topologies:
    • Router-Switch-Direct Connect Device
    • Router-Direct Connect Device
    • LAG configurations (all members must support MACsec)

4. Implementation Process

  • Configuration Steps:
    • Create connection with requestMACSec flag
    • Associate CKN/CAK using AssociateMacSecKey API
    • Verify status with DescribeConnections API
    • Set encryption mode (should_encrypt, must_encrypt, no_encrypt)
  • Key Management:
    • AWS Secrets Manager integration
    • Service-linked IAM role required
    • Up to 3 CKN/CAK pairs in keychain
    • Manual key rotation process

5. Critical Exam Points

  • Remember These Values:
    • Default encryption mode: should_encrypt
    • Maximum keychain capacity: 3 CKN/CAK pairs
    • CKN/CAK format: 32 Octets (64 Hex digits)
    • Available speeds: 10 Gbps and 100 Gbps only
  • Encryption Mode Behaviors:
    • should_encrypt: Attempts encryption, falls back to unencrypted
    • must_encrypt: Requires encryption, connection down if fails
    • no_encrypt: No encryption attempted, ignores MACsec frames

6. Advantages Over Other Security Methods

  • Compared to IPsec:
    • Higher throughput (hardware vs. crypto engine)
    • Lower latency (Layer 2 vs. Layer 3)
    • Protects all Layer 2 traffic including routing protocols
    • Less CPU-intensive (hardware offload)
  • Compared to TLS:
    • Secures all traffic, not just application data
    • Transparent to applications
    • No certificate management complexity

7. Ideal Use Cases

  • Industry Applications:
    • Financial services (regulatory requirements)
    • Healthcare (PHI protection)
    • Media production (large file transfers)
    • Autonomous vehicle development (high bandwidth)
    • Any scenario where IPsec throughput is insufficient
