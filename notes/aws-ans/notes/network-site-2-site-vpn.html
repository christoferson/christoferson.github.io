AWS Site-to-Site VPN - Core Concepts

Introduction

  • Purpose:
    • Enables secure communication between VPC and on-premises networks
    • Provides encrypted connection over the internet
    • Supports IPsec VPN connections
  • Default Behavior:
    • By default, EC2 instances in VPC cannot communicate with remote networks
    • VPN connection enables this communication
    • Requires proper routing configuration

Key Components

  • VPN Connection:
    • Secure connection between on-premises equipment and VPCs
    • Includes two VPN tunnels for redundancy
    • Both tunnels can be used simultaneously
  • Customer Gateway:
    • AWS resource representing your on-premises device
    • Contains information about your physical device or software
    • Requires static IP address (if using IP-based authentication)
  • Target Gateway:
    • AWS-side VPN endpoint
    • Can be either:
      • Virtual Private Gateway: Attached to a single VPC
      • Transit Gateway: Hub connecting multiple VPCs and networks

Tunnel Architecture

  • Dual Tunnel Design:
    • Each VPN connection provides two tunnels
    • Each tunnel terminates in different Availability Zones
    • Provides high availability and redundancy
  • Traffic Flow:
    • On-premises to AWS: Uses both tunnels
    • AWS to on-premises: Prefers one tunnel but fails over automatically
    • Automatic failover when a tunnel becomes unavailable

Exam Tips

  • Key Points:
    • Two tunnels per VPN connection
    • Customer gateway requires static IP address
    • Target gateway can be VGW or Transit Gateway
    • Tunnels terminate in different AZs
  • Remember:
    • Both tunnels should be configured for redundancy
    • Traffic automatically routes to available tunnel
    • VPN connection = AWS resource; Customer gateway device = your hardware

AWS Site-to-Site VPN - Features and Limitations

Supported Features

  • Protocol Support:
    • Internet Key Exchange version 2 (IKEv2)
    • NAT traversal
    • IPv6 traffic (transit gateway only)
  • Advanced Configurations:
    • 4-byte ASN for VGW (range: 1-2147483647)
    • 2-byte ASN for CGW (range: 1-65535)
    • Reusable IP addresses for customer gateways
    • Configurable tunnel options
  • Security Features:
    • AES 256-bit encryption
    • SHA-2 hashing
    • Additional Diffie-Hellman groups
    • Private Certificate authentication
  • Monitoring:
    • CloudWatch metrics integration
    • VPN logs
    • Tunnel status monitoring

Key Limitations

  • Technical Constraints:
    • IPv6 traffic not supported on virtual private gateways
    • No support for Path MTU Discovery
    • Cannot use overlapping CIDR blocks
    • Maximum 6 name servers per domain
  • VPN Connection Components:
    • Requires virtual private gateway or transit gateway
    • Requires customer gateway device
    • Requires customer gateway (AWS resource)
    • Always includes two VPN tunnels

IPv4 and IPv6 Support

  • IPv6 Constraints:
    • Only supported on transit gateway connections
    • Not supported on virtual private gateway connections
    • Cannot be enabled on existing VPN connections
    • A connection supports either IPv4 or IPv6, not both
  • IPv6 Implementation:
    • Only for inside tunnel IP addresses
    • Outside tunnel endpoints remain IPv4
    • Customer gateway public IP must be IPv4
    • Each tunnel gets both /30 IPv4 and /126 IPv6 CIDR blocks

Exam Tips

  • Feature Highlights:
    • IKEv2 and NAT traversal support
    • CloudWatch metrics integration
    • Advanced encryption options
    • Configurable tunnel options
  • Critical Limitations:
    • IPv6 only on transit gateway, not VGW
    • No Path MTU Discovery
    • Cannot enable IPv6 on existing connections
    • A connection supports either IPv4 or IPv6, not both
  • Common Scenarios:
    • IPv6 configuration requirements
    • ASN range limitations
    • Security feature implementation
    • Monitoring configuration

AWS Site-to-Site VPN - Gateway Types and Configurations

Virtual Private Gateway

  • Core Functionality:
    • VPN concentrator on the AWS side
    • Attaches to a single VPC
    • Enables communication between VPC and on-premises network
  • Configuration Options:
    • Customizable private ASN (Autonomous System Number)
    • Default ASN: 64512 if not specified
    • ASN cannot be changed after creation
    • IPv4 traffic only (no IPv6 support)
  • Limitations:
    • One VGW per VPC
    • Cannot share across multiple VPCs
    • No ECMP support (Equal Cost Multi-Path)
    • One primary tunnel across all VPN connections

Transit Gateway

  • Core Functionality:
    • Regional transit hub for network connectivity
    • Connects multiple VPCs and on-premises networks
    • Supports VPN attachments
    • Enables network consolidation
  • Advanced Features:
    • Supports both IPv4 and IPv6 traffic
    • Enables ECMP for multiple active tunnels
    • Supports multiple VPN connections
    • Can modify target gateway from VGW to TGW
  • Architecture Benefits:
    • Hub-and-spoke network design
    • Simplified network management
    • Reduced connection complexity
    • Enhanced routing capabilities

Customer Gateway Device

  • Definition:
    • Physical device or software application on your side
    • Located in your on-premises network
    • Initiates VPN tunnels by default
    • Must be configured to work with Site-to-Site VPN
  • Configuration Options:
    • Default: Customer gateway initiates tunnels
    • Optional: AWS can initiate IKE negotiation
    • Requires specific configuration based on device model
    • Must support IPsec protocol

Customer Gateway Resource

  • Required Information:
    • Static public IP address (if using IP-based authentication)
    • BGP ASN (for dynamic routing)
    • Optional device name
    • Optional private certificate (for certificate-based authentication)
  • BGP ASN Options:
    • Public ASN range: 1-4,294,967,295 (with exceptions)
    • Private ASN ranges: 64,512-65,534 or 4,200,000,000-4,294,967,294
    • Default ASN: 64512
    • Reserved ASNs: 7224 (all regions), 9059 (eu-west-1), etc.

Exam Tips

  • Key Differences:
    • VGW: Single VPC, IPv4 only, no ECMP
    • TGW: Multiple VPCs, IPv4/IPv6, supports ECMP

    • ASN ranges and defaults
  • Common Scenarios:
    • Choosing between VGW and TGW
    • BGP ASN configuration
    • IP-based vs certificate-based authentication
    • Tunnel initiation options

AWS Site-to-Site VPN - Tunnel Options and Configurations

Tunnel Configuration Overview

  • Basic Structure:
    • Each VPN connection provides two tunnels
    • Configuration file provided for each customer gateway device
    • Default values applied unless specified
    • Options configurable during creation or modification
  • Proposal Evaluation:
    • AWS endpoints evaluate proposals starting with lowest configured value
    • Customer gateway proposal order doesn't matter
    • Can restrict options using modify-vpn-connection-options command

Key Tunnel Options

  • Dead Peer Detection (DPD):
    • Timeout: 30+ seconds (Default: 40)
    • Actions: Clear, None, Restart (Default: Clear)
    • Determines behavior when peer becomes unresponsive
  • IKE Settings:
    • IKE versions: ikev1, ikev2 (Default: both)
    • Phase 1 DH groups: 2, 14-24 (Default: all)
    • Phase 2 DH groups: 2, 5, 14-24 (Default: all)
    • Phase 1 lifetime: 900-28,800 seconds (Default: 28,800)
    • Phase 2 lifetime: 900-3,600 seconds (Default: 3,600)
  • Encryption and Integrity:
    • Phase 1 encryption: AES128, AES256, AES128-GCM-16, AES256-GCM-16
    • Phase 2 encryption: Same options as Phase 1
    • Phase 1 integrity: SHA1, SHA2-256, SHA2-384, SHA2-512
    • Phase 2 integrity: Same options as Phase 1

IP Addressing Options

  • Inside Tunnel IPv4 CIDR:
    • Size /30 CIDR from 169.254.0.0/16 range
    • Must be unique across VPN connections on same gateway
    • Reserved ranges: 169.254.0.0/30 through 169.254.5.0/30 and 169.254.169.252/30
  • Inside Tunnel IPv6 CIDR:
    • Size /126 CIDR from local fd00::/8 range
    • Only for IPv6 VPN connections on transit gateway
    • Must be unique across VPN connections on same gateway
  • Network CIDRs:
    • Local/Remote IPv4 Network CIDR: Default 0.0.0.0/0
    • Local/Remote IPv6 Network CIDR: Default ::/0
    • Defines allowed traffic ranges

Advanced Options

  • Authentication:
    • Pre-shared key (PSK): 8-64 characters
    • Cannot start with zero (0)
    • Alphanumeric, periods, underscores allowed
    • Default: 32-character alphanumeric string
  • Rekey Settings:
    • Rekey fuzz: 0-100% (Default: 100)
    • Rekey margin time: 60 to half of phase 2 lifetime (Default: 270)
    • Replay window size: 64-2048 packets (Default: 1024)
  • Tunnel Control:
    • Startup action: Start or Add (Default: Add)
    • Tunnel endpoint lifecycle control: On or Off (Default: Off)
    • VPN logging options: json or text format

Exam Tips

  • Key Default Values:
    • DPD timeout: 40 seconds
    • DPD action: Clear
    • Phase 1 lifetime: 28,800 seconds (8 hours)
    • Phase 2 lifetime: 3,600 seconds (1 hour)
    • Startup action: Add (customer initiates)
  • Important Considerations:
    • AWS evaluates proposals from lowest value regardless of customer proposal order
    • Phase 2 lifetime must be less than Phase 1
    • Rekey timing can affect tunnel stability
    • Inside tunnel CIDRs must be unique
  • Common Scenarios:
    • Configuring encryption requirements
    • Setting up tunnel initiation
    • Managing DPD behavior
    • Handling inside tunnel addressing

AWS Site-to-Site VPN - Accelerated VPN Connections

Accelerated VPN Overview

  • Core Functionality:
    • Uses AWS Global Accelerator to optimize network path
    • Routes traffic to closest AWS edge location
    • Avoids public internet congestion
    • Improves connection reliability and performance
  • Implementation Details:
    • Two accelerators created automatically (one per tunnel)
    • Managed by AWS (not visible in Global Accelerator console)
    • Uses separate IP address pool for tunnel endpoints
    • Tunnel IPs selected from different network zones

Rules and Restrictions

  • Compatibility Requirements:
    • Only supported with transit gateway (not VGW)
    • Cannot be used with Direct Connect public virtual interface
    • Cannot enable/disable for existing VPN connection
    • NAT traversal (NAT-T) required and enabled by default
  • Configuration Requirements:
    • IKE negotiation must be initiated from customer gateway device
    • Affected by Startup Action and DPD Timeout Action settings
    • Certificate-based authentication may be incompatible due to packet fragmentation limitations
    • Customer gateway device must support IKE fragmentation if using certificates

Implementation Process

  • Enabling Acceleration:
    • Optional feature during VPN creation
    • Only available for transit gateway attachments
    • Disabled by default
    • Cannot be modified after creation
  • Migration Strategy:
    • Create new VPN connection with acceleration enabled
    • Configure customer gateway device to use new connection
    • Delete old VPN connection after migration
    • Verify NAT-T settings in configuration file

Performance Considerations

  • Benefits:
    • Reduced latency through optimized routing
    • Improved connection stability
    • AWS global network utilization
    • Congestion avoidance
  • Use Cases:
    • Globally distributed applications
    • Latency-sensitive workloads
    • Connections traversing congested internet paths
    • Mission-critical VPN connections

Exam Tips

  • Key Limitations:
    • Transit gateway only (not VGW)
    • Cannot modify existing connections
    • NAT-T required
    • Customer gateway must initiate IKE
  • Implementation Facts:
    • Uses AWS Global Accelerator behind the scenes
    • Two accelerators created (one per tunnel)
    • Separate IP address pool
    • Requires creation of new VPN connection to enable/disable
  • Common Scenarios:
    • Choosing between standard and accelerated VPN
    • Migration planning for acceleration
    • Certificate authentication compatibility
    • Global network optimization

AWS Site-to-Site VPN - Routing Options

Static vs Dynamic Routing

  • Selection Criteria:
    • Based on customer gateway device capabilities
    • Dynamic routing requires BGP support
    • Static routing used when BGP not supported
    • Cannot mix routing types on same connection
  • Dynamic Routing (BGP):
    • Uses BGP to advertise routes
    • No manual route specification needed
    • Provides robust liveness detection
    • Assists failover to second tunnel
    • Recommended when available
  • Static Routing:
    • Requires manual route specification
    • IP prefixes must be entered explicitly
    • Less automatic failover capability
    • May still perform health checks

Route Tables and Priority

  • VPC Route Table Requirements:
    • Must add route for remote network
    • Must specify virtual private gateway as target
    • Route propagation can be enabled
    • Longest prefix match determines routing
  • Route Priority Rules:
    • Local VPC routes always preferred over propagated routes
    • Static routes preferred over propagated routes
    • Static route targets prioritized: IGW, VGW, interface, instance, peering, NAT, TGW, endpoints
    • Only known IP prefixes can receive traffic

Path Selection Hierarchy

  • Selection Process:
    • Longest prefix match applied first
    • Tunnel health takes precedence over routing attributes
    • Routes prioritized from most to least preferred:
  • Priority Order:
    • 1. BGP routes from Direct Connect
    • 2. Static routes for Site-to-Site VPN
    • 3. BGP routes from Site-to-Site VPN
    • 4. For equal BGP routes, shortest AS PATH preferred
    • 5. For equal AS PATH length, lowest MED value preferred

Tunnel Selection and ECMP

  • Virtual Private Gateway:
    • One tunnel selected as primary egress path
    • Selection may change periodically
    • Only one tunnel used across all VPN connections
    • No ECMP support (no multiple active tunnels)
  • Transit Gateway:
    • Supports Equal Cost Multi-Path (ECMP)
    • Can use multiple tunnels simultaneously
    • Provides better throughput and redundancy
    • Recommended for higher bandwidth requirements

Routing Recommendations

  • Best Practices:
    • Use customer gateway devices supporting asymmetric routing
    • Avoid AS PATH prepending with asymmetric routing
    • Configure both tunnels for high availability
    • Advertise specific BGP routes to influence routing decisions
    • Use ECMP with transit gateway for higher throughput
  • Asymmetric Routing:
    • Strongly recommended for customer gateway devices
    • Allows different paths for request and response traffic
    • Enables MED values to determine tunnel priority
    • Provides more flexible routing options

Exam Tips

  • Key Concepts:
    • BGP recommended when available
    • Local routes always preferred over propagated routes
    • Static routes preferred over propagated routes
    • VGW uses one primary tunnel; TGW can use multiple (ECMP)
  • Route Priority Order:
    • Direct Connect BGP routes
    • Static VPN routes
    • BGP VPN routes
    • AS PATH length
    • MED value
  • Common Scenarios:
    • Choosing between static and dynamic routing
    • Route priority determination
    • Tunnel selection behavior
    • ECMP configuration with transit gateway

AWS Site-to-Site VPN - Comprehensive Summary

1. Core Architecture

  • Essential Components:
    • VPN Connection: Two encrypted tunnels between AWS and on-premises
    • Customer Gateway: AWS resource representing your on-premises device
    • Customer Gateway Device: Your physical or virtual appliance
    • Target Gateway: Either Virtual Private Gateway (VGW) or Transit Gateway (TGW)
  • Tunnel Design:
    • Two tunnels per connection for redundancy
    • Each tunnel terminates in different Availability Zones
    • Both tunnels should be configured for high availability

2. Gateway Options and Differences

  • Virtual Private Gateway:
    • Attaches to a single VPC
    • IPv4 traffic only
    • Uses one primary tunnel across all connections
    • No ECMP support
    • ASN range: 1-2147483647 (default: 64512)
  • Transit Gateway:
    • Connects multiple VPCs and on-premises networks
    • Supports both IPv4 and IPv6 traffic
    • Supports ECMP (multiple active tunnels)
    • Required for accelerated VPN connections

3. Routing Options

  • Dynamic Routing (BGP):
    • Recommended when customer gateway supports BGP
    • Automatic route advertisement
    • Better liveness detection and failover
    • Customer Gateway ASN range: 1-65535
  • Static Routing:
    • Used when BGP not supported
    • Requires manual route specification
    • Less automatic failover capability
  • Route Priority:
    • Local routes > Static routes > Propagated routes
    • Direct Connect BGP > Static VPN > BGP VPN
    • Shortest AS PATH > Lowest MED value

4. Advanced Features

  • Accelerated VPN:
    • Uses AWS Global Accelerator
    • Transit Gateway only
    • Requires NAT-T
    • Customer gateway must initiate IKE
  • IPv6 Support:
    • Transit Gateway only
    • Inside tunnel addresses only
    • Cannot mix IPv4 and IPv6 on same connection
    • Outside tunnel endpoints remain IPv4
  • Authentication Options:
    • Pre-shared key (PSK): 8-64 characters
    • Certificate-based: Using AWS Private CA
    • Certificate authentication may conflict with acceleration

5. Key Configuration Options

  • Tunnel Options:
    • IKE versions: ikev1, ikev2
    • Encryption: AES128, AES256, AES-GCM
    • DPD timeout: 30+ seconds (default: 40)
    • Phase 1/2 lifetimes: 28,800/3,600 seconds
  • Tunnel Initiation:
    • Default: Customer gateway initiates (Add)
    • Optional: AWS initiates (Start)
    • Start requires customer gateway IP address

6. Critical Limitations

  • Technical Constraints:
    • No Path MTU Discovery
    • IPv6 only on Transit Gateway
    • Cannot enable/disable acceleration on existing connections
    • Cannot mix IPv4 and IPv6 on same connection
  • Design Considerations:
    • Non-overlapping CIDR blocks recommended
    • Both tunnels should be configured
    • Customer gateway device should support asymmetric routing
    • Inside tunnel CIDRs must be unique

7. Exam Focus Areas

  • Architecture Understanding:
    • Component relationships and responsibilities
    • Differences between VGW and TGW
    • Tunnel redundancy design
  • Routing Knowledge:
    • Static vs dynamic routing
    • Route priority rules
    • Path selection process
    • ECMP capabilities
  • Feature Compatibility:
    • IPv6 support limitations
    • Accelerated VPN requirements
    • Authentication options
    • Tunnel option configurations

AWS Site-to-Site VPN - Getting Started Guide

Introduction

  • Overview:
    • Step-by-step process to set up a Site-to-Site VPN connection
    • Focuses on virtual private gateway implementation
    • Assumes existing VPC with one or more subnets
    • Alternative target options include transit gateway or "Not associated"

Prerequisites

  • Required Information:
    • Customer Gateway Device Details:
      • Vendor (e.g., Cisco)
      • Platform (e.g., ISR Series Routers)
      • Software version (e.g., IOS 12.4)
    • Customer Gateway Resource Information:
      • Internet-routable IP address
      • Routing type (static or dynamic)
      • BGP ASN (for dynamic routing)
      • Optional: Private certificate from AWS Private CA
    • VPN Connection Details:
      • IP prefixes for private network (static routing)
      • Optional: Tunnel options configuration
      • Optional: AWS-side ASN (if not using default)

Implementation Steps

  • Step 1: Create Customer Gateway
    • Provides AWS with information about your device
    • Requires either IP address or private certificate ARN
    • BGP ASN specification for dynamic routing
    • Optional device name for identification
  • Step 2: Create Target Gateway
    • Virtual Private Gateway:
      • Optional custom ASN (default or custom)
      • Must be attached to your VPC
      • ASN must differ from customer gateway ASN
    • Alternative: Transit Gateway (separate process)

Routing Configuration

  • Step 3: Configure Routing
    • Enable Route Propagation (recommended):
      • Automatically adds VPN routes to route table
      • Static routes propagated when VPN is UP
      • BGP routes propagated when VPN is UP
    • Alternative: Manual Route Addition:
      • Add static routes to route table
      • Specify destination CIDR and target gateway
    • Transit Gateway Routing:
      • Add route to subnet route table
      • Point to transit gateway as target

Security Configuration

  • Step 4: Update Security Groups
    • Allow access to VPC instances from on-premises network
    • Update inbound rules for:
      • SSH access
      • RDP access
      • ICMP access
    • Specify on-premises network CIDR as source

Exam Tips

  • Key Points:
    • Route propagation simplifies management
    • Security group updates essential for access
    • Customer gateway requires either IP or certificate
    • ASNs must be different between gateways
  • Common Scenarios:
    • VPC to on-premises connectivity
    • Static vs dynamic routing configuration
    • Security group configuration
    • Route table management

AWS Site-to-Site VPN - Architectural Scenarios

Common VPN Deployment Patterns

  • Multiple VPN Connections with Same Customer Gateway:
    • Connect multiple VPCs to same on-premises location
    • Reuse same customer gateway IP address
    • Each VPC has separate VPN connection
    • Simplifies on-premises device management
  • Multiple Customer Gateways to Single Virtual Private Gateway:
    • AWS VPN CloudHub architecture
    • Connect multiple locations to single VPC
    • Each location uses unique customer gateway
    • Enables site-to-site communication through AWS
    • Each location should advertise unique IP ranges
  • Redundant VPN Connections:
    • Protect against customer gateway device failure
    • Second VPN connection with second customer gateway device
    • Both devices advertise same IP ranges
    • Enables maintenance without connectivity loss

Single VPN Connection Architectures

  • Basic VPC with Virtual Private Gateway:
    • VPC with attached virtual private gateway
    • Single customer gateway device on-premises
    • Two tunnels for redundancy
    • VPC route tables direct traffic to virtual private gateway
  • VPC with Transit Gateway:
    • VPC with attached transit gateway
    • Single customer gateway device on-premises
    • Two tunnels for redundancy
    • VPC route tables direct traffic to transit gateway
    • Enables future expansion to multiple VPCs

Multiple VPN Connection Architectures

  • Multiple Sites with Virtual Private Gateway:
    • VPC with attached virtual private gateway
    • Multiple VPN connections to different locations
    • Each location has customer gateway device
    • Traffic routed through virtual private gateway
    • Can be used for redundancy or multi-site connectivity
  • Multiple Sites with Transit Gateway:
    • VPC with attached transit gateway
    • Multiple VPN connections to different locations
    • Each location has customer gateway device
    • Traffic routed through transit gateway
    • Simplified management for multiple connections
    • Easier scaling for additional sites

Hybrid Connectivity Architectures

  • Site-to-Site VPN with Direct Connect:
    • VPC with virtual private gateway
    • Direct Connect for primary connectivity
    • Site-to-Site VPN as backup or for encrypted traffic
    • Both connect through virtual private gateway
    • VPG may enter 'attaching' state during changes
  • Private IP VPN with Direct Connect:
    • Encrypts Direct Connect traffic
    • No public IP addresses required
    • Ensures both security and privacy
    • Helps meet regulatory requirements
    • Keeps traffic entirely on private network

Exam Tips

  • Key Architecture Decisions:
    • VPG vs Transit Gateway based on scale needs
    • Single vs redundant customer gateways based on availability requirements
    • CloudHub for multi-site connectivity
    • Direct Connect + VPN for secure hybrid connectivity
  • Common Scenarios:
    • Designing for high availability
    • Multi-site connectivity planning
    • Hybrid network architecture
    • Scaling VPN connections

AWS Site-to-Site VPN - CloudHub Architecture

CloudHub Overview

  • Core Concept:
    • Hub-and-spoke network model using AWS as the hub
    • Enables secure communication between multiple sites
    • Sites connect to AWS and communicate with each other
    • Can be used with or without a VPC
    • Cost-effective alternative to direct site-to-site connections
  • Use Cases:
    • Multiple branch offices needing interconnectivity
    • Primary or backup connectivity between sites
    • Leveraging existing internet connections
    • Simplified hub-and-spoke topology
    • Hybrid environments with Direct Connect and VPN

Architecture Components

  • Required Elements:
    • Single virtual private gateway
    • Multiple customer gateways (one per site)
    • Dynamically routed VPN connections
    • Non-overlapping IP ranges between sites
  • BGP Configuration:
    • Unique BGP ASN for each customer gateway
    • Site-specific prefix advertisements
    • Example prefixes: 10.0.0.0/24, 10.0.1.0/24, etc.
    • Routes received and re-advertised to each BGP peer
    • Enables site-to-site communication

Implementation Steps

  • Setup Process:
    • Create single virtual private gateway
    • Create multiple customer gateways with unique ASNs
    • Establish dynamically routed VPN connections
    • Configure customer gateway devices to advertise site-specific prefixes
    • Update VPC route tables (if connecting to VPC)
  • Routing Configuration:
    • Configure network statements in VPN configurations
    • Use aggregate routes in VPC route tables (e.g., 10.0.0.0/16)
    • Use more specific prefixes between customer gateways and VPG
    • Enable route propagation if using VPC

Hybrid CloudHub

  • Direct Connect Integration:
    • Corporate headquarters can use Direct Connect
    • Branch offices can use Site-to-Site VPN
    • All sites can communicate with each other
    • Example: New York HQ (Direct Connect) + LA and Miami branches (VPN)
    • Combines high-bandwidth Direct Connect with flexible VPN deployment

Pricing Model

  • Cost Structure:
    • Standard VPN connection hourly rates
    • Billed for each hour each VPN is connected
    • No charge for data sent from site to virtual private gateway
    • Standard AWS data transfer rates for traffic between sites
  • Example Scenario:
    • Los Angeles and New York sites with VPN connections
    • Two VPN connections at $0.05/hour each = $0.10/hour total
    • Data transfer from LA to NY charged at standard rates
    • Inbound to VPG free, outbound from VPG charged

Exam Tips

  • Key Requirements:
    • Dynamic routing (BGP) required
    • Unique ASN per customer gateway
    • Non-overlapping IP ranges between sites
    • Single virtual private gateway
  • Common Scenarios:
    • Multi-branch connectivity design
    • Hybrid Direct Connect and VPN architecture
    • Cost-effective site-to-site communication
    • BGP route advertisement configuration
  • Cost Considerations:
    • VPN connection hourly rates
    • Data transfer charges for inter-site communication
    • No charges for inbound to VPG
    • Standard rates for outbound from VPG

AWS Site-to-Site VPN - Redundant Connections for Failover

Redundancy Overview

  • Purpose:
    • Protect against customer gateway device failure
    • Enable maintenance without connectivity loss
    • Increase overall VPN connection availability
    • Provide business continuity during device outages
  • Basic Architecture:
    • Two VPN connections to same VPC/virtual private gateway
    • Two separate customer gateway devices
    • Each VPN connection has its own tunnels
    • Both devices advertise same IP ranges
    • Traffic automatically reroutes if one device fails

Implementation Requirements

  • AWS-Side Configuration:
    • Single virtual private gateway
    • Two customer gateway resources (one per device)
    • Two Site-to-Site VPN connections
    • Each customer gateway IP must be publicly accessible
  • On-Premises Configuration:
    • Two physical or software customer gateway devices
    • Both devices configured to advertise same IP ranges
    • Both devices properly configured for VPN tunnels
    • Network configured to use both devices

Routing Considerations

  • Dynamic Routing (Recommended):
    • Uses BGP to exchange routing information
    • Virtual private gateway can detect device failures
    • Automatic rerouting to working customer gateway
    • More efficient failover process
  • Static Routing:
    • Requires manual route entry for remote networks
    • Less automatic failover capability
    • Still works but requires more configuration
    • Network must be configured to select available path

Failover Mechanism

  • How Failover Works:
    • BGP or static routes provide path information
    • Virtual private gateway monitors tunnel health
    • If customer gateway device fails, VPG detects failure
    • Traffic automatically rerouted to working device
    • Network continues to function during failure
  • Network Configuration:
    • Should use routing information from BGP when available
    • Select available path based on routing protocol
    • Specific configuration depends on network architecture
    • May require additional on-premises routing configuration

Best Practices

  • Design Recommendations:
    • Use BGP-enabled devices when possible
    • Configure both customer gateway devices identically
    • Advertise same IP prefixes from both devices
    • Test failover scenarios regularly
    • Monitor both connections for health
  • Maintenance Considerations:
    • Schedule maintenance for one device at a time
    • Verify working connection before maintenance
    • Monitor traffic flow during device switchover
    • Validate connectivity after maintenance

Exam Tips

  • Key Concepts:
    • Two separate customer gateway devices required
    • Both devices advertise same IP ranges
    • BGP provides more efficient failover
    • Each connection has two tunnels (four tunnels total)
  • Common Scenarios:
    • High availability VPN design
    • Maintenance without downtime
    • Protecting against device failures
    • Comparing static vs dynamic routing for failover

    • Each VPN connection already has two tunnels

AWS Site-to-Site VPN Customer Gateway Devices - Study Guide

Introduction

  • What is a Customer Gateway Device:
    • Physical or software appliance in your on-premises network
    • Owned and managed by you (customer side of VPN connection)
    • Must be configured to work with AWS Site-to-Site VPN
    • Establishes encrypted tunnels to AWS virtual private gateway
  • High-Level Architecture:
    • Two tunnels per VPN connection for redundancy
    • Automatic failover between tunnels
    • AWS performs routine maintenance on tunnels
    • Only one tunnel is active at a time (unless using equal-cost multi-path routing)

Technical Requirements

  • Core Components:
    • Internet Key Exchange (IKE) security association
    • IPsec security association
    • Tunnel interface
    • Border Gateway Protocol (BGP) peering (optional)
  • Protocol Requirements:
    • IKE versions: IKEv1 or IKEv2 (IKEv2 recommended)
    • IKEv1 mode: Main mode only
    • Encryption: AES 128-bit or AES 256-bit
    • Hashing: SHA-1 or SHA-2 (256)
    • Perfect Forward Secrecy: Required
    • Diffie-Hellman groups:
      • Phase 1: Groups 2, 14-24
      • Phase 2: Groups 2, 5, 14-24

Additional Requirements

  • For Dynamic Routing (BGP):
    • Dead Peer Detection support
    • Tunnel binding to logical interface
    • BGP peering capability
    • Route-based VPN (not policy-based)
    • Maximum 1 security association pair per tunnel
  • Limitations:
    • No Path MTU Discovery support
    • Policy-based VPNs must limit to single security association
    • Maximum 1399 byte MTU

Exam Tips

  • Key Points to Remember:
    • Two tunnels per VPN connection for redundancy
    • IKEv2 is strongly recommended over IKEv1
    • Route-based VPN required for dynamic routing
    • Tunnel initiation can be from either side
    • AWS VPN is a route-based solution
  • Common Exam Scenarios:
    • Troubleshooting VPN connectivity issues
    • Identifying protocol requirements
    • Understanding tunnel redundancy
    • Differentiating static vs. dynamic routing requirements

AWS Site-to-Site VPN - Best Practices

Protocol Selection

  • IKE Version:
    • Use IKEv2 whenever possible
    • Only use IKEv1 if customer gateway device doesn't support IKEv2
    • IKEv2 advantages:
      • Simpler protocol
      • More robust
      • Enhanced security
      • Better performance

Packet Handling Optimization

  • Don't Fragment (DF) Flag:
    • Reset the DF flag on packets
    • Prevents ICMP Path MTU Exceeded messages
    • Helps applications without adequate ICMP processing
    • Configure customer gateway to override DF flag when possible
  • Fragmentation Approach:
    • Fragment IP packets before encryption
    • AWS Site-to-Site VPN reassembles before forwarding
    • Improves packet-per-second throughput
    • Enhances overall performance

MTU and MSS Optimization

  • Size Considerations:
    • Ensure packet size doesn't exceed destination network MTU
    • Consider forwarding paths (Direct Connect, etc.)
    • Account for protocol-specific requirements
  • Algorithm-Specific Settings:
    • Maximum supported MTU: 1446 bytes
    • Maximum supported MSS: 1406 bytes (IPv4)
    • Adjust based on encryption and authentication algorithms
    • Example optimal settings:
      • AES-GCM-16 without NAT-T: MTU 1446, MSS 1406
      • AES-GCM-16 with NAT-T: MTU 1438, MSS 1398
      • AES-CBC with SHA1/SHA2-256 without NAT-T: MTU 1438, MSS 1398

Additional Configuration Best Practices

  • IKE Unique IDs:
    • Disable IKE unique IDs setting if supported
    • Prevents inconsistent Phase 2 states between VPN peers
    • Ensures more reliable connection state

Exam Tips

  • Key Points to Remember:
    • IKEv2 is strongly preferred over IKEv1
    • MTU/MSS settings vary by encryption algorithm
    • Fragment before encryption for better performance
    • Reset DF flag when possible
    • Disable IKE unique IDs
  • Algorithm-Specific MTU/MSS Values:
    • Know that different algorithms require different MTU settings
    • NAT-Traversal reduces available MTU
    • IPv6-in-IPv4 requires smaller MSS values

AWS Site-to-Site VPN - Firewall Rules

Introduction

  • Firewall Requirements:
    • Static IP address required for IPsec tunnel endpoints
    • Specific firewall rules needed between AWS and customer gateway
    • Rules must be configured in both directions
    • Proper configuration essential for tunnel establishment

Inbound Rules (from Internet to Customer Gateway)

  • Rule I1:
    • Source IP: Tunnel1 Outside IP
    • Destination IP: Customer Gateway
    • Protocol: UDP
    • Source Port: 500
    • Destination Port: 500
    • Purpose: IKE negotiation for first tunnel
  • Rule I2:
    • Source IP: Tunnel2 Outside IP
    • Destination IP: Customer Gateway
    • Protocol: UDP
    • Source Port: 500
    • Destination Port: 500
    • Purpose: IKE negotiation for second tunnel
  • Rule I3:
    • Source IP: Tunnel1 Outside IP
    • Destination IP: Customer Gateway
    • Protocol: IP 50 (ESP)
    • Purpose: Encrypted IPsec traffic for first tunnel
  • Rule I4:
    • Source IP: Tunnel2 Outside IP
    • Destination IP: Customer Gateway
    • Protocol: IP 50 (ESP)
    • Purpose: Encrypted IPsec traffic for second tunnel

Outbound Rules (from Customer Gateway to Internet)

  • Rule O1:
    • Source IP: Customer Gateway
    • Destination IP: Tunnel1 Outside IP
    • Protocol: UDP
    • Source Port: 500
    • Destination Port: 500
    • Purpose: IKE negotiation for first tunnel
  • Rule O2:
    • Source IP: Customer Gateway
    • Destination IP: Tunnel2 Outside IP
    • Protocol: UDP
    • Source Port: 500
    • Destination Port: 500
    • Purpose: IKE negotiation for second tunnel
  • Rule O3:
    • Source IP: Customer Gateway
    • Destination IP: Tunnel1 Outside IP
    • Protocol: IP 50 (ESP)
    • Purpose: Encrypted IPsec traffic for first tunnel
  • Rule O4:
    • Source IP: Customer Gateway
    • Destination IP: Tunnel2 Outside IP
    • Protocol: IP 50 (ESP)
    • Purpose: Encrypted IPsec traffic for second tunnel

NAT Traversal Considerations

  • Additional Requirements:
    • If using NAT-T, allow UDP traffic on port 4500
    • Required in both directions
    • Check if device is advertising NAT-T capability
    • NAT-T encapsulates ESP packets in UDP for NAT compatibility

Exam Tips

  • Protocol Knowledge:
    • UDP 500: Internet Key Exchange (IKE)
    • IP Protocol 50: Encapsulating Security Payload (ESP)
    • UDP 4500: NAT Traversal (NAT-T)
  • Key Points:
    • Rules I1, I2, O1, O2: Enable IKE packet transmission
    • Rules I3, I4, O3, O4: Enable encrypted IPsec traffic
    • Both tunnels require separate rules
    • NAT-T requires additional UDP 4500 rules
  • Common Issues:
    • Missing firewall rules
    • Incorrect IP addresses
    • Blocked protocols
    • NAT-T requirements overlooked

AWS Site-to-Site VPN - Troubleshooting

Introduction

  • Troubleshooting Approach:
    • Structured methodology required
    • Different approaches for BGP vs. static routing
    • Device-specific troubleshooting may be needed
    • AWS VPN logs provide valuable diagnostic information

Troubleshooting with BGP

  • IKE Phase:
    • Verify IKE security association exists
    • Check encryption, authentication, PFS, and mode parameters
    • Confirm IKE configuration matches AWS requirements
    • Common issues: mismatched pre-shared keys, encryption parameters
  • IPsec Phase:
    • Verify IPsec security association is active
    • Check encryption, authentication, PFS, and mode parameters
    • Confirm IPsec configuration matches AWS requirements
    • Common issues: mismatched security parameters, policy conflicts
  • Tunnel Phase:
    • Verify firewall rules are properly configured
    • Test IP connectivity through tunnel with ping
    • Ping virtual private gateway address from customer gateway
    • Common issues: incorrect IP addressing, MTU problems
  • BGP Phase:
    • Verify BGP peering session is Active or Established
    • Check if customer gateway is advertising default route (0.0.0.0/0)
    • Allow ~30 seconds for BGP peering to become active
    • Common issues: BGP configuration errors, route advertisement problems

Troubleshooting without BGP (Static Routing)

  • IKE and IPsec Phases:
    • Same verification steps as with BGP
    • Check security associations and parameters
  • Tunnel Phase:
    • Same verification steps as with BGP
    • Verify firewall rules and IP connectivity
  • Static Routes Phase:
    • Verify static route to VPC CIDR with tunnels as next hop
    • Check static route on Amazon VPC console for return traffic
    • Ensure both tunnels are properly configured
    • Common issues: missing routes, incorrect CIDR blocks

Windows Server Specific Troubleshooting

  • Common Issues:
    • Security group rules not allowing IPsec traffic
    • Source/destination checking not disabled (for EC2 instances)
    • Incorrect route table configuration
    • ICMP not enabled on target instances
    • Dead gateway detection not enabled

General Troubleshooting Tips

  • Diagnostic Tools:
    • Enable debug features on customer gateway device
    • Use AWS VPN logs for connection issues
    • Check device-specific logs (Cisco, Juniper, etc.)
    • Use ping and traceroute for connectivity testing
  • Common Failure Points:
    • Firewall blocking required protocols
    • Mismatched security parameters
    • Incorrect routing configuration
    • MTU/fragmentation issues
    • NAT interference with IPsec

AWS Site-to-Site VPN Study Guide - Part 1: Core Concepts

Introduction to AWS Site-to-Site VPN

  • Definition:
    • Secure connection between on-premises network and AWS resources
    • Encrypted IPsec tunnels over the internet or Direct Connect
    • Supports both static and dynamic routing
  • Key Components:
    • VPN Connection: The encrypted link between AWS and on-premises
    • Customer Gateway: Resource representing your on-premises device
    • Target Gateway: AWS-side endpoint (Virtual Private Gateway, Transit Gateway, or Cloud WAN)
    • VPN Tunnels: Two encrypted tunnels for redundancy

VPN Attachment Types

  • Virtual Private Gateway Attachment:
    • Traditional VPN endpoint for a single VPC
    • Attached directly to a VPC
    • Limited to resources within that VPC
  • Transit Gateway Attachment:
    • Enables connectivity to multiple VPCs
    • Supports higher bandwidth and throughput
    • Supports IPv6 traffic (VPG does not)
    • Supports more BGP routes (5000 outbound, 1000 inbound)
  • Cloud WAN Attachment:
    • Integrates with AWS Cloud WAN global network
    • Managed through Cloud WAN core network
    • Enables global connectivity across regions

VPN Connection Options

  • Routing Options:
    • Dynamic Routing (BGP): Uses BGP to exchange routes
    • Static Routing: Manually configured routes
  • Tunnel Options:
    • Inside IP Version: IPv4 or IPv6 (IPv6 only with Transit Gateway)
    • Outside IP Address Type: Public IPv4 or Private IPv4
    • Tunnel Authentication: Pre-shared key or certificate-based
  • Advanced Features:
    • Accelerated VPN: Uses AWS Global Accelerator
    • Private IP VPN: Runs over Direct Connect without public IPs

Exam Tips

  • Key Concepts to Remember:
    • AWS always creates two tunnels for redundancy
    • Transit Gateway supports IPv6, Virtual Private Gateway does not
    • Dynamic routing requires BGP support on customer gateway
    • Accelerated VPN incurs additional charges
  • Common Scenarios:
    • Choosing between attachment types based on requirements
    • Understanding routing options and their implications
    • Identifying when to use Private IP VPN vs standard VPN
    • Troubleshooting connectivity issues

AWS Site-to-Site VPN Study Guide - Part 2: Creating VPN Connections

Creating a Transit Gateway VPN Attachment

  • Prerequisites:
    • Existing transit gateway
    • Customer gateway information (IP address, ASN)
    • Routing strategy (static or dynamic)
  • Key Configuration Steps:
    • Select transit gateway as target
    • Choose existing or create new customer gateway
    • Specify BGP ASN for customer gateway (default: 65000)
    • Select routing option (dynamic or static)
    • Choose tunnel IP version (IPv4 or IPv6)
    • Configure allowed CIDR ranges (default: 0.0.0.0/0 for IPv4, ::/0 for IPv6)

Creating a Cloud WAN VPN Attachment

  • Key Differences:
    • Target gateway type: "Not associated"
    • Attachment created for Cloud WAN core network
    • Similar configuration options as transit gateway
  • Important Considerations:
    • Cloud WAN must be configured separately
    • VPN attachment integrates with global network
    • Routing controlled through Cloud WAN policy

Tunnel Options Configuration

  • Customizable Parameters:
    • Inside tunnel IPv4 CIDR: /30 CIDR from 169.254.0.0/16
    • Inside tunnel IPv6 CIDR: /126 CIDR from fd00::/8
    • IKE versions: IKEv1 or IKEv2
    • Pre-shared key (PSK): Custom or auto-generated
  • Advanced Options:
    • Phase 1 and Phase 2 encryption algorithms
    • Phase 1 and Phase 2 integrity algorithms
    • DH group numbers
    • Lifetime values
    • Replay window size
    • Dead peer detection settings

Creating a Private IP VPN over Direct Connect

  • Prerequisites:
    • Existing transit gateway with CIDR block
    • Direct Connect gateway
    • Transit gateway association with Direct Connect gateway
    • Transit virtual interface (VIF)
  • Configuration Steps:
    • Create customer gateway with private IP address (RFC 1918)
    • Create VPN connection with transit gateway target
    • Select "PrivateIpv4" for outside IP address type
    • Specify transport attachment ID (Direct Connect attachment)
  • Key Limitations:
    • Acceleration not supported with Private IP VPN
    • Requires RFC 1918 addresses for tunnel endpoints
    • Cannot use point-to-point IPs for eBGP peering

Exam Tips

  • Key Configuration Details:
    • Default BGP ASN: 65000
    • Default CIDR ranges: 0.0.0.0/0 (IPv4), ::/0 (IPv6)
    • Inside tunnel IPv4 range: 169.254.0.0/16
    • Inside tunnel IPv6 range: fd00::/8
    • Maximum 6 name servers per domain
  • Common Scenarios:
    • Choosing between attachment types
    • Configuring tunnel options
    • Setting up Private IP VPN
    • Understanding prerequisites for each VPN type

AWS Site-to-Site VPN Study Guide - Part 3: Testing and Managing VPN Connections

Testing VPN Connectivity

  • Prerequisites:
    • Properly configured customer gateway device
    • Correct routing configuration in VPC route tables
    • Security groups/NACLs allowing ICMP traffic
    • Windows firewall allowing inbound ICMPv4 (for Windows instances)
  • Testing Process:
    • Launch EC2 instance in VPC with appropriate security group
    • Obtain private IP address of instance
    • Ping instance from on-premises network
    • Verify bidirectional connectivity
    • Test tunnel failover by disabling one tunnel

Modifying VPN Connection Options

  • Modifiable Parameters:
    • Local (customer gateway) CIDR ranges
    • Remote (AWS) CIDR ranges
    • IPv4 and IPv6 CIDR ranges
  • Modification Process:
    • Select VPN connection in console
    • Choose "Modify VPN connection options"
    • Update CIDR ranges as needed
    • Brief connectivity interruption during update
  • Important Notes:
    • VPN endpoint IP addresses do not change
    • Tunnel options remain unchanged
    • Temporary unavailability during update

Modifying VPN Tunnel Options

  • Modifiable Tunnel Parameters:
    • Inside tunnel IPv4/IPv6 CIDR blocks
    • IKE versions and pre-shared keys
    • Phase 1/2 encryption and integrity algorithms
    • DH groups and lifetime values
    • Dead peer detection settings
  • Modification Process:
    • Select VPN connection in console
    • Choose "Modify VPN tunnel options"
    • Select specific tunnel by outside IP address
    • Update tunnel options as needed
  • Important Warning:
    • Tunnel connectivity interrupted for several minutes
    • Plan for expected downtime
    • Only one tunnel can be modified at a time

Managing Static Routes

  • For Virtual Private Gateway:
    • Add/remove static routes from VPN configuration
    • Update route tables if route propagation not enabled
    • Manually update routes to reflect static IP prefixes
  • For Transit Gateway:
    • Add/modify/remove routes in transit gateway route table
    • No direct VPN connection route modification
    • Routes managed through transit gateway route tables

Changing Customer Gateway

  • Use Cases:
    • Customer gateway device replacement
    • IP address change of on-premises device
    • Migration to different on-premises location
  • Change Process:
    • Select VPN connection in console
    • Choose "Modify VPN connection"
    • Select "Customer gateway" as target type
    • Choose new customer gateway
    • Brief connectivity interruption during update

Exam Tips

  • Key Management Concepts:
    • One tunnel can be modified at a time
    • Tunnel modifications cause brief outages
    • VPN endpoint IPs don't change when modifying options
    • Static routes managed differently for VPG vs Transit Gateway
  • Testing Best Practices:
    • Test both directions of connectivity
    • Verify tunnel failover capability
    • Check security group and NACL configurations
    • Test from multiple on-premises locations if applicable

AWS Site-to-Site VPN Study Guide - Part 4: Security and Advanced Features

Replacing Compromised Credentials

  • Pre-shared Key Replacement:
    • Modify tunnel options for each tunnel individually
    • Specify new IKE pre-shared key
    • Update customer gateway device with new key
    • Brief connectivity interruption during update
  • Alternative Approach:
    • Delete and recreate VPN connection
    • Use same virtual private gateway
    • Specify new pre-shared keys or let AWS generate them
    • Note: Tunnel inside/outside addresses may change

Certificate-Based Authentication

  • AWS Side Certificate Rotation:
    • Use console to rotate tunnel endpoint certificates
    • Select specific tunnel endpoint
    • AWS automatically rotates certificates near expiration
    • Uses service-linked role for rotation
  • Customer Gateway Certificate:
    • Create new certificate in AWS Certificate Manager
    • Add certificate to customer gateway device
    • Update VPN configuration if needed

Private IP VPN with Direct Connect

  • Key Benefits:
    • End-to-end private connectivity with encryption
    • No public IP addresses required
    • Simplified network management
    • Improved security posture
    • Higher route scale (5000 outbound, 1000 inbound)
  • Architecture Components:
    • Transit gateway with private CIDR block
    • Direct Connect gateway
    • Transit virtual interface (VIF)
    • Customer gateway with private IP
  • Important Considerations:
    • CIDR blocks must not overlap with other attachments
    • Use RFC 1918 addresses for tunnel endpoints
    • Use loopback or LAN interface on customer gateway
    • Acceleration not supported

Modifying Target Gateway

  • Supported Migration Paths:
    • Virtual private gateway to transit gateway
    • Virtual private gateway to another virtual private gateway
    • Transit gateway to another transit gateway
    • Transit gateway to virtual private gateway
  • Migration Process:
    • Create new target gateway
    • Delete static routes if migrating from VPG to TGW
    • Modify VPN connection target
    • Update VPC route tables
    • Update target gateway routing if needed
    • Update customer gateway ASN if changed
  • Important Considerations:
    • Brief connectivity interruption during migration
    • MED values handled differently between VPG and TGW
    • TGW sets same MED across all tunnels
    • May require routing changes to avoid issues

Deleting VPN Resources

  • Deletion Order:
    • Delete VPN connection first
    • Then delete customer gateway if no longer needed
    • For virtual private gateway: detach from VPC, then delete
  • Important Warnings:
    • Deletion is permanent and non-recoverable
    • New configuration file needed if recreating connection
    • Customer gateway device needs reconfiguration
    • VPN connection remains visible briefly with "deleted" state

Exam Tips

  • Security Best Practices:
    • Rotate compromised credentials immediately
    • Use certificate-based authentication for higher security

AWS Site-to-Site VPN - Comprehensive Study Guide Summary

1. Core Architecture Components

  • Essential Elements:
    • VPN Connection: Encrypted IPsec link between AWS and on-premises
    • Customer Gateway: AWS resource representing on-premises device
    • Target Gateway: AWS-side endpoint (VPG, TGW, or Cloud WAN)
    • VPN Tunnels: Always two tunnels for redundancy
  • Target Gateway Types:
    • Virtual Private Gateway: Single VPC connectivity
    • Transit Gateway: Multi-VPC connectivity, higher bandwidth
    • Cloud WAN: Global network integration

2. Key Configuration Options

  • Routing Options:
    • Dynamic (BGP): ASN required, automatic route propagation
    • Static: Manually configured routes, no BGP required
  • Tunnel Options:
    • Inside IP Version: IPv4 or IPv6 (IPv6 only with TGW)
    • Authentication: Pre-shared key or certificate-based
    • Inside CIDR: IPv4 (/30 from 169.254.0.0/16) or IPv6 (/126 from fd00::/8)
    • Encryption/Integrity: Customizable algorithms
  • Advanced Features:
    • Accelerated VPN: Uses AWS Global Accelerator
    • Private IP VPN: Runs over Direct Connect with private IPs

3. Critical Limitations and Quotas

  • Transit Gateway vs Virtual Private Gateway:
    • TGW: 5000 routes outbound, 1000 routes inbound
    • VPG: 100 routes outbound, 100 routes inbound
    • TGW supports IPv6, VPG does not
    • TGW sets same MED across all tunnels
  • Private IP VPN Requirements:
    • Direct Connect transit VIF required
    • Transit gateway with CIDR block
    • RFC 1918 addresses for tunnel endpoints
    • No acceleration support

4. Management Operations

  • Modification Operations:
    • Connection Options: CIDR ranges (brief interruption)
    • Tunnel Options: One tunnel at a time (several minutes downtime)
    • Target Gateway: Supported migration paths with brief outage
    • Customer Gateway: Can be changed with brief interruption
  • Security Operations:
    • Credential Replacement: Modify tunnel options or recreate connection
    • Certificate Rotation: AWS automatic or manual process
    • Deletion: Connection first, then other resources

5. Testing and Troubleshooting

  • Connectivity Testing:
    • Verify security groups and NACLs allow ICMP
    • Test bidirectional connectivity
    • Test tunnel failover by disabling one tunnel
    • Verify route propagation or static routes
  • Common Issues:
    • Mismatched PSK or tunnel options
    • Incorrect routing configuration
    • Security group/NACL restrictions
    • Customer gateway device misconfiguration

6. Exam Focus Areas

  • Architecture Selection:
    • When to use each target gateway type
    • Benefits of Private IP VPN vs standard VPN
    • When to use accelerated VPN
    • Static vs dynamic routing considerations
  • Configuration Knowledge:
    • Tunnel IP addressing (169.254.0.0/16 for IPv4, fd00::/8 for IPv6)
    • Authentication options and security best practices
    • Route propagation and management
    • Migration processes between gateway types
  • Operational Understanding:
    • Impact of modifications on connectivity
    • Security credential management
    • Testing and validation procedures
    • Resource deletion order and implications

7. Key Differentiators

  • Virtual Private Gateway:
    • Single VPC connectivity
    • IPv4 only
    • Limited route scale (100 routes)
    • Different MED values per tunnel
  • Transit Gateway:
    • Multi-VPC connectivity
    • IPv4 and IPv6 support
    • Higher route scale (5000 routes)
    • Same MED value across tunnels
  • Private IP VPN:
    • Requires Direct Connect
    • Uses private IP addresses
    • Enhanced security posture
    • No acceleration support
