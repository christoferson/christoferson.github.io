AWS VPC Peering - Comprehensive Study Guide

Introduction to VPC Peering

  • Definition:
    • A networking connection between two VPCs
    • Enables direct, private communication between VPCs
    • Resources interact as if in the same network
    • Traffic never traverses the public internet
  • Core Characteristics:
    • Non-transitive relationship (one-to-one connection)
    • No single point of failure
    • No bandwidth bottlenecks
    • No gateway or hardware requirements

Key Capabilities

  • Connectivity Options:
    • Between VPCs in the same AWS account
    • Between VPCs in different AWS accounts
    • Between VPCs in different AWS Regions (inter-region)
  • Technical Benefits:
    • Low latency, high bandwidth connectivity
    • Secure communication within AWS network
    • Leverages existing VPC infrastructure
    • No additional hardware or physical connections

Implementation Requirements

  • Routing Configuration:
    • Route tables must be updated in both VPCs
    • Routes must point to the peering connection
    • Destination CIDR blocks must be specified
    • Security groups and NACLs still control access
  • CIDR Requirements:
    • VPC CIDR blocks cannot overlap
    • IPv4 and/or IPv6 address ranges must be unique
    • Secondary CIDR blocks also considered

Common Use Cases

  • Application Architecture:
    • Connecting application tiers across VPCs
    • Separating web servers from database servers
    • Microservices communication
  • Organizational Structure:
    • Resource sharing between teams
    • Cross-account access for business units
    • Development and production environment isolation
  • Geographic Distribution:
    • Multi-region application deployment
    • Disaster recovery configurations
    • Global resource access

Limitations and Considerations

  • Network Constraints:
    • Non-transitive connectivity (no A→B→C)
    • Maximum of 125 peering connections per VPC
    • Cannot create peering if CIDR blocks overlap
    • Cannot modify CIDR blocks after peering
  • Service Limitations:
    • VPC endpoint services not accessible across peering
    • Security group references limited to same-region peering
    • DNS resolution requires additional configuration
    • No support for IPv6 unicast or multicast

Exam Tips

  • Key Concepts:
    • Non-transitive nature of peering
    • CIDR block requirements
    • Route table configuration necessity
    • Cross-account and cross-region capabilities
  • Common Scenarios:
    • Troubleshooting connectivity issues
    • Designing multi-VPC architectures
    • Security considerations for peered VPCs
    • Comparing VPC peering with other connectivity options

AWS VPC Peering - Technical Deep Dive

What is VPC Peering?

  • Definition:
    • A networking connection between two VPCs
    • Enables communication using private IP addresses
    • Resources interact as if in the same network
    • Traffic never traverses the public internet
  • Technical Implementation:
    • Uses existing VPC infrastructure
    • Not a gateway or VPN connection
    • No separate physical hardware required
    • No single point of failure
    • No bandwidth bottlenecks

Connection Types

  • Supported Peering Scenarios:
    • Same account, same Region
    • Same account, different Regions
    • Different accounts, same Region
    • Different accounts, different Regions
  • Inter-Region Benefits:
    • All traffic encrypted automatically
    • Stays on global AWS backbone
    • Never traverses public internet
    • Reduced security threats (exploits, DDoS)
    • Geographic redundancy

Cost Structure

  • Pricing Model:
    • No charge to create peering connection
    • Data transfer within same AZ is free (even across accounts)
    • Standard data transfer charges apply for:
      • Cross-AZ traffic
      • Cross-Region traffic
  • Cost Optimization:
    • Keep related resources in same AZ when possible
    • Consider data transfer costs for cross-Region peering
    • Refer to EC2 pricing for specific rates

Use Cases

  • Common Applications:
    • File sharing networks across accounts
    • Resource access across organizational boundaries
    • Multi-account architectures
    • Cross-Region redundancy
    • Private communication between services

Exam Tips

  • Key Concepts:
    • Private IP communication without gateways/VPNs
    • No transitive peering (direct connections only)
    • Inter-Region traffic stays on AWS backbone
    • No single point of failure
  • Remember:
    • Free to create, pay for data transfer
    • Same-AZ transfer is free
    • Cross-AZ/Region incurs charges
    • All inter-Region traffic is encrypted

AWS VPC Peering - Connection Lifecycle and Process

VPC Peering Process

  • Step 1: Request Initiation
    • Requester VPC owner sends peering request
    • Target VPC can be owned by same or different AWS account
    • CIDR blocks cannot overlap between VPCs
  • Step 2: Request Acceptance
    • Accepter VPC owner accepts the connection request
    • Connection becomes active
    • Requests expire after 7 days if not accepted
  • Step 3: Route Table Configuration
    • Both VPC owners must update route tables
    • Routes must point to peer VPC CIDR blocks
    • Target must be the peering connection ID
  • Step 4: Security Group Configuration
    • Update security groups to allow traffic
    • Can reference peer VPC security groups (same Region only)
  • Step 5: DNS Resolution (Optional)
    • Enable DNS hostname resolution if needed
    • Allows public DNS hostnames to resolve to private IPs

Connection Lifecycle States

  • Initial States:
    • Initiating-request: Request being created
    • Pending-acceptance: Awaiting accepter approval (expires after 7 days)
    • Failed: Request creation failed
  • Response States:
    • Provisioning: Request accepted, connection being established
    • Active: Connection operational, traffic can flow
    • Rejected: Request declined by accepter
    • Expired: Request not accepted within 7 days
  • Termination States:
    • Deleting: Connection being removed
    • Deleted: Connection removed

Visibility Timeframes

  • Failed Requests:
    • Visible to requester for 2 hours
  • Expired Requests:
    • Visible to both parties for 2 days
  • Rejected Requests:
    • Visible to requester for 2 days
    • Visible to accepter for 2 hours
    • Same-account rejection: Visible for 2 hours
  • Deleted Connections:
    • Visible to deleting party for 2 hours
    • Visible to other party for 2 days
    • Same-account deletion: Visible for 2 hours

Multiple VPC Peering Relationships

  • One-to-One Relationship:
    • Each peering connection connects exactly two VPCs
    • One VPC can have multiple peering connections
    • Maximum of one peering connection between any two VPCs
  • Non-Transitive Nature:
    • No transitive peering relationships
    • If VPC A peers with VPC B and VPC C, VPC B cannot communicate with VPC C through VPC A
    • Direct peering required between all VPCs that need to communicate

Exam Tips

  • Key Process Points:
    • Request → Accept → Configure Routes → Configure Security
    • Both VPC owners must update route tables
    • 7-day expiration for pending requests
  • State Transitions:
    • Understand lifecycle states and visibility periods
    • Active state required for traffic flow
    • Region-specific acceptance for inter-Region peering
  • Critical Concepts:
    • Non-transitive nature of peering
    • One-to-one VPC relationships
    • Multiple peering connections per VPC

AWS VPC Peering - Limitations and Constraints

Connection Limitations

  • Quota Constraints:
    • Maximum 125 active/pending peering connections per VPC
    • Only one peering connection between any two VPCs
    • Cannot modify connection after creation
  • Tag Management:
    • Tags only applied in account/Region where created
    • Not synchronized across accounts or Regions
    • Must be managed separately in each context

DNS and IP Addressing

  • DNS Limitations:
    • Cannot connect to/query Amazon DNS server in peer VPC
    • DNS resolution support must be explicitly enabled
    • Private DNS hostnames require special handling for non-RFC 1918 ranges
  • IPv6 Considerations:
    • IPv6 communication requires explicit configuration
    • Both VPCs need IPv6 CIDR blocks
    • Instances must be enabled for IPv6
    • Route tables must include IPv6 routes

CIDR Block Restrictions

  • Overlap Constraints:
    • Cannot peer VPCs with matching/overlapping CIDR blocks
    • All CIDR blocks considered (primary and secondary)
    • Even non-overlapping secondary CIDRs blocked if primary CIDRs overlap
    • IPv4 and IPv6 overlap both considered

Transitive Peering Limitation

  • No Transitive Routing:
    • VPC peering does not support transitive relationships
    • If VPC A peers with VPC B and VPC C, VPC B cannot communicate with VPC C through VPC A
    • Direct peering connection required between all VPCs that need to communicate
    • Consider Transit Gateway for hub-and-spoke architecture

Edge-to-Edge Routing Limitations

  • Gateway Access Restrictions:
    • Internet Gateway: Peer VPC cannot use another VPC's internet gateway
    • NAT Gateway/Device: Peer VPC cannot use another VPC's NAT services
    • VPN Connection: Peer VPC cannot use another VPC's VPN to corporate network
    • Direct Connect: Peer VPC cannot use another VPC's Direct Connect connection
    • Gateway VPC Endpoints: Peer VPC cannot use another VPC's gateway endpoints

Inter-Region Specific Limitations

  • Technical Constraints:
    • MTU differences:
      • Same-Region: 9001 bytes (jumbo frames)
      • Inter-Region: 8500 bytes
    • DNS resolution support required for private DNS hostnames
    • Security group referencing only available in same-Region peering

Shared VPC Limitations

  • Ownership Restrictions:
    • Only VPC owners can manage peering connections
    • Participants cannot create, accept, reject, modify, or delete connections
    • VPC sharing does not extend to peering management

Exam Tips

  • Critical Limitations:
    • 125 peering connections per VPC maximum
    • No overlapping CIDR blocks
    • No transitive peering
    • No edge-to-edge routing through gateways
  • Alternative Solutions:
    • Transit Gateway for hub-and-spoke architecture
    • Direct Connect for corporate network access
    • VPC Endpoints for AWS service access
  • Common Exam Scenarios:
    • Identifying when peering is inappropriate
    • Recognizing transitive routing limitations
    • Understanding gateway access restrictions
    • Comparing VPC peering vs. Transit Gateway

AWS VPC Peering - Creation and Configuration

Creating a VPC Peering Connection

  • Supported Scenarios:
    • Same account, same Region
    • Same account, different Regions
    • Different accounts, same Region
    • Different accounts, different Regions
  • Request Location Requirements:
    • Inter-Region requests must be made from requester VPC's Region
    • Inter-Region acceptance must be done from accepter VPC's Region
    • Same-account peering allows self-creation and acceptance

Accepting or Rejecting Connections

  • Security Considerations:
    • Only accept requests from known/trusted AWS accounts
    • Reject unwanted requests to prevent "peer phishing"
    • Requests expire after 7 days if not accepted
  • Acceptance Process:
    • Accept from the Region of the accepter VPC
    • Connection moves to "Provisioning" then "Active" state
    • Route tables must be updated after acceptance
  • Rejection Process:
    • Rejected connections cannot be accepted later
    • New request must be created if needed
    • Rejected state visible to requester for 2 days

Route Table Configuration

  • Required Updates:
    • Both VPCs need route table entries
    • Destination: Peer VPC's CIDR block
    • Target: VPC peering connection ID (pcx-xxxxxxxx)
    • Can specify entire CIDR or specific ranges
  • Example Configuration:
    • VPC A route table:
      • Local route for VPC A CIDR
      • Route to VPC B CIDR via peering connection
    • VPC B route table:
      • Local route for VPC B CIDR
      • Route to VPC A CIDR via peering connection
  • IPv6 Configuration:
    • Similar process for IPv6 CIDR blocks
    • Specify IPv6 ranges as destinations
    • Target remains the peering connection ID

Security Group Configuration

  • Reference Options:
    • Reference peer security groups (same Region only)
    • Reference CIDR blocks (any peering scenario)
    • Reference specific IP addresses
  • Cross-Account References:
    • Format: account-id/security-group-id
    • Example: 123456789012/sg-1a2b3c4d
    • Only works in same Region
  • Security Group Management:
    • Identify referenced security groups
    • Monitor stale security group rules
    • Clean up stale references when connections deleted

DNS Resolution Configuration

  • Default Behavior:
    • Public DNS hostnames resolve to public IP addresses
    • DNS resolution disabled by default
  • With DNS Resolution Enabled:
    • Public DNS hostnames resolve to private IP addresses
    • Enables seamless private communication
  • Requirements:
    • Both VPCs must have DNS hostnames and resolution enabled
    • Peering connection must be in active state
    • Both requester and accepter options must be modified
    • Always required for inter-Region peering

Deleting Connections

  • Deletion Rights:
    • Either VPC owner can delete the connection
    • Can delete pending-acceptance connections
    • Cannot delete rejected connections (auto-deleted)
  • VPC Deletion Behavior:
    • Deleting a VPC also deletes its peering connections
    • Cannot delete VPC with pending-acceptance requests from other accounts
    • Must reject incoming requests first

Exam Tips

  • Creation Process:
    • Request → Accept → Configure Routes → Configure Security
    • Inter-Region requests must be made from requester Region
    • Inter-Region acceptance must be done in accepter Region
  • Configuration Requirements:
    • Both VPCs need route table updates
    • Security groups must allow traffic
    • DNS resolution requires explicit enabling
  • Security Best Practices:
    • Only accept requests from trusted accounts
    • Reject unwanted requests
    • Clean up stale security group references

AWS VPC Peering - Troubleshooting and Common Configurations

Troubleshooting VPC Peering Connections

  • Route Table Verification:
    • Confirm route tables contain routes to peer VPC CIDR
    • Verify target is the peering connection ID
    • Check subnet associations with route tables
    • Ensure routes are in both directions
  • Security Group Validation:
    • Verify inbound rules allow traffic from peer VPC
    • Confirm outbound rules permit traffic to peer VPC
    • Check security group references (same Region only)
    • Validate CIDR-based rules for cross-Region peering
  • Network ACL Inspection:
    • Examine subnet network ACLs
    • Verify both inbound and outbound rules
    • Confirm traffic allowed from/to peer VPC CIDR
    • Check rule numbers and processing order
  • Advanced Troubleshooting:
    • Use Reachability Analyzer to identify specific issues
    • Isolate problematic component (route table, security group, NACL)
    • Verify connection is in Active state
    • Check for overlapping CIDR blocks

Common VPC Peering Configurations

  • Full-Mesh Peering:
    • Every VPC directly connected to every other VPC
    • Maximum connectivity but complex to manage
    • n(n-1)/2 connections required (n = number of VPCs)
    • Becomes unwieldy with many VPCs
  • Hub-and-Spoke (Not with Peering):
    • Not possible with VPC peering alone (no transitive routing)
    • Requires Transit Gateway instead
    • Common misconception in exam scenarios
  • Isolated Multi-Account:
    • Peering between specific VPCs across accounts
    • Selective connectivity based on requirements
    • Common for organizational boundaries
  • Cross-Region Redundancy:
    • Peering between primary and DR Regions
    • Private communication for replication
    • Geographic distribution of workloads

VPC Peering vs. Transit Gateway

  • When to Use VPC Peering:
    • Simple connectivity between few VPCs
    • Direct one-to-one relationships
    • No need for transitive routing
    • Lower cost for simple topologies
  • When to Use Transit Gateway:
    • Hub-and-spoke architecture needed
    • Many VPCs to connect (simplified management)
    • Transitive routing required
    • Need for centralized network control
    • VPN and Direct Connect integration

Response Traffic Routing

  • Unicast Reverse Path Forwarding:
    • Not supported in VPC peering connections
    • Can cause issues with overlapping CIDRs
  • Overlapping CIDR Challenges:
    • Multiple peers with overlapping CIDRs cause routing ambiguity
    • Response traffic may route to incorrect VPC
    • Careful route table design required

Exam Tips

  • Troubleshooting Methodology:
    • Check route tables first
    • Verify security groups second
    • Examine network ACLs third
    • Use Reachability Analyzer for complex issues
  • Architecture Decision Points:
    • VPC Peering vs. Transit Gateway trade-offs
    • Full-mesh limitations with many VPCs
    • No transitive routing with peering
    • Response traffic routing considerations
  • Common Exam Scenarios:
    • Identifying connectivity issues
    • Selecting appropriate connection type
    • Recognizing when Transit Gateway is needed
    • Understanding troubleshooting steps
