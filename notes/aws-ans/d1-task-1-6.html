Domain 1: Network Design

Task Statement 1.6: Design a routing strategy and connectivity architecture that includes multiple AWS accounts, AWS Regions, and VPCs to support different connectivity patterns

Knowledge 1: Connectivity Patterns and Use Cases

  • VPC Peering (Direct VPC-to-VPC Connection):
    • Core Characteristics:
      • Direct Network Connectivity: One-to-one networking relationship between VPCs enabling direct communication using private IP addresses
        • Low latency, high bandwidth connection
        • No intermediate hops or gateway devices
        • Supports IPv4 and IPv6 traffic
      • Regional Flexibility: Supports both intra-region and inter-region peering
        • Same-region: No data transfer charges between AZs
        • Cross-region: Regional data transfer charges apply
        • Different AWS accounts supported
      • Non-Transitive Nature: Traffic cannot flow through intermediate VPCs
        • If VPC A peers with B and C, B cannot reach C through A
        • Requires direct peering for all required connections
        • Helps maintain network isolation and security
    • Implementation Scenarios:
      • Application Resource Sharing: Enable direct access between application tiers
        • Database sharing across environments
        • Microservices communication
        • Shared service access
      • Environment Separation: Isolate different stages while maintaining connectivity
        • Dev/Test/Prod isolation
        • Controlled access patterns
        • Resource optimization
  • Transit Gateway (Centralized Network Hub):
    • Architecture Components:
      • Hub-and-Spoke Topology: Central hub connecting multiple VPCs and networks
        • Single connection point for all networks
        • Simplified network architecture
        • Reduced connection complexity
      • Routing Control: Centralized management of route tables and propagation
        • Route table associations
        • Route propagation control
        • Blackhole routes for security
      • Multi-Account Support: Resource sharing across AWS accounts
        • RAM (Resource Access Manager) integration
        • Cross-account attachments
        • Organizational unit management
    • Advanced Features:
      • Multicast Support: Enable multicast communication between VPCs
        • IGMP protocol support
        • Multicast domains
        • Source/receiver management
      • Multi-Region Networking: Connect networks across AWS regions
        • Inter-region peering
        • Global network view
        • Regional redundancy
  • AWS PrivateLink (Secure Service Connectivity):
    • Security Features:
      • Private Service Access: Connect to services without public internet exposure
        • Interface endpoints (powered by ENI)
        • Private DNS resolution
        • Security group control
      • Service Provider Model: Expose services to other VPCs securely
        • Network load balancer integration
        • Service endpoint policies
        • Cross-account access control
    • Implementation Patterns:
      • SaaS Integration: Secure access to third-party services
        • AWS Marketplace integration
        • Service provider connectivity
        • Compliance requirements
      • Microservices Architecture: Secure service-to-service communication
        • Service mesh integration
        • API endpoint exposure
        • Service discovery

Knowledge 3: IP Address Management

  • IP Subnets (Network Segmentation Strategy):
    • CIDR Planning: Hierarchical IP address allocation framework
      • VPC CIDR Allocation:
        • Primary CIDR block (RFC 1918 ranges):
          • 10.0.0.0/8 (Large networks)
          • 172.16.0.0/12 (Medium networks)
          • 192.168.0.0/16 (Small networks)
        • Secondary CIDR blocks:
          • Non-overlapping ranges
          • Contiguous IP ranges
          • Maximum 5 CIDR blocks per VPC
        • CIDR restrictions:
          • Minimum /28 netmask
          • Maximum /16 netmask
          • No overlapping with existing VPC CIDRs
      • Subnet Design:
        • AZ distribution strategy:
          • Equal-sized subnets per AZ

Best Practices and Design Considerations

  • Network Design (Architecture Planning):
    • IP Address Management:
      • CIDR Block Planning (/16 to /28)
        • Objective: Ensure optimal IP address space allocation while maintaining flexibility for growth
        • Why:
          • Prevents IP exhaustion in growing environments
          • Enables clean subnet boundaries for different workloads
          • Facilitates easier troubleshooting and management
        • When to Use:
          • New VPC deployments
          • Network expansion planning
          • Multi-account environments
        • Best Practice Implementation:
          • Start with larger CIDR blocks (/16) for primary VPCs
          • Use smaller blocks (/24) for specialized workloads
          • Reserve /28 blocks for small, isolated components
        • Anti-patterns to Avoid:
          • Using too small CIDR blocks that limit growth
          • Overlapping CIDR ranges between VPCs
          • Inconsistent sizing across similar workloads
      • Subnet Allocation Strategy
        • Objective: Optimize resource distribution and isolation across Availability Zones
        • Why:
          • Ensures high availability of applications
          • Enables efficient resource utilization
          • Supports security isolation requirements
        • When to Use:
          • Initial VPC design
          • Application deployment planning
          • Compliance requirement implementation
        • Best Practice Implementation:
          • Allocate equal-sized subnets per AZ
          • Reserve larger blocks for compute-heavy workloads
          • Implement consistent naming conventions
        • Anti-patterns to Avoid:
          • Uneven subnet sizes across AZs
          • Single AZ subnet allocation
          • Over-segmentation leading to management complexity
