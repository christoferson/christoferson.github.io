Knowledge 3: Common IaC Template Issues

  • Hardcoded Values (Template Inflexibility):
    • IP Addresses (Network Configuration):
      • Common Issues:
        • CIDR Block Hardcoding:
          • Problem: Fixed VPC and subnet CIDR ranges
          • Impact:
            • Limited reusability across environments
            • IP overlap conflicts
            • Difficult network changes
          • Solution:
            • Use parameters for CIDR ranges
            • Implement CIDR validation
            • Define default values carefully
        • Security Group Rules:
          • Problem: Fixed IP addresses in security rules
          • Impact:
            • Inflexible security configurations
            • Manual updates required
            • Maintenance overhead
          • Solution:
            • Parameterize IP ranges
            • Use security group references
            • Implement dynamic rule generation
    • Resource Names (Identifier Management):
      • Naming Issues:
        • Static Resource Names:
          • Problem: Fixed resource identifiers
          • Impact:
            • Name conflicts in different environments
            • Limited stack reuse
            • Version control challenges
          • Solution:
            • Use dynamic naming patterns
            • Implement naming conventions
            • Utilize stack parameters
    • Account IDs (Cross-Account References):
      • Account Management:
        • Hardcoded Account Numbers:
          • Problem: Fixed AWS account references
          • Impact:
            • Limited account portability
            • Security risks
            • Maintenance complexity

              • Maintenance complexity

        • Stack Dependencies:

                • Document dependency chains
                • Implement change management
      • Circular Dependencies (Dependency Loop Prevention):
        • Detection and Resolution:
          • Common Patterns:
            • Direct Circular References:
              • Problem: Resource A depends on B, B depends on A
              • Impact:
                • Deployment failures
                • Stack creation errors
                • Update impossibility
              • Resolution:
                • Restructure dependencies
                • Use intermediate resources
                • Split into separate stacks
            • Indirect Circular References:
              • Problem: Complex dependency chains creating loops
              • Detection:
                • Dependency mapping
                • Resource graph analysis
                • Template validation
              • Prevention:
                • Modular design
                • Clear dependency hierarchy
                • Regular architecture review

Best Practices

  • Template Design (Infrastructure Definition Best Practices):
    • Parameter Usage:
      • Purpose: Create flexible and maintainable templates
      • Implementation:
        • Parameter Design:
          • Use descriptive names: Clear purpose indication
          • Implement constraints: Validate inputs early
          • Provide default values: When appropriate
        • Anti-patterns to Avoid:
          • Over-parameterization: Too many parameters create complexity
          • Weak constraints: Insufficient input validation
          • Missing descriptions: Unclear parameter purpose
    • Resource Tagging:
      • Purpose: Enable resource organization and management
      • Strategy:
        • Required Tags:
          • Environment: dev, test, prod
          • Owner: team or individual responsible
          • Cost Center: billing allocation
          • Application: system identification
        • Implementation:
          • Use parameter-based tags: Dynamic value assignment
          • Enforce tag standards: Through policies
          • Implement tag validation: Check compliance
  • Security Implementation (Security Best Practices):
    • Least Privilege Access:
      • Purpose: Minimize security risks through access control
      • Implementation:
        • IAM Role Design:
          • Specific permissions: Grant only required access
          • Resource-level constraints: Limit scope of access
          • Condition statements: Add contextual restrictions
        • Security Group Rules:
          • Specific source/destination: Avoid 0.0.0.0/0
          • Required ports only: Minimize attack surface
          • Rule documentation: Comment purpose of rules
