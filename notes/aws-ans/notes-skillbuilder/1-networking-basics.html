Design Considerations

CIDR Management
  • CIDR Overlap Prevention:
    • Critical Requirement: No overlapping CIDR blocks between VPCs
      • Prevents VPC peering
      • Blocks Direct Connect setup
      • Inhibits VPN connectivity
  • VPC Resizing Limitations:
    • Secondary CIDR Rules:
      • Must be from same RFC 1918 range as primary
      • Example: Primary 10.0.0.0/8
        • Can add: Other ranges within 10.0.0.0/8
        • Cannot add: 172.16.0.0/12 or 192.168.0.0/16
      • Alternative: Use non-RFC 1918 public ranges
Subnet Sizing Considerations
  • Load Balancer Requirements:
    • Minimum Subnet Size: /27 (32 IP addresses)
      • Purpose: Ensure proper scaling
      • Applies to: Each AZ where load balancer operates
      • Consideration: Account for AWS-reserved IPs
  • General Subnet Planning:
    • Minimum Size: /28 (16 IP addresses)
    • Reserved IPs:
      • First four IPs
      • Last IP address
      • Consider future growth

Route Tables and Traffic Control

Fundamental Concepts
  • Route Table Structure:
    • Components:
      • Destination: Target IP range
      • Target: Where traffic is directed
      • Local Route: Default VPC routing
    • Association Rules:

  • Database Tier:

      • Optional Routes:
        • VPC Endpoint routes

        • Transit Gateway routes

Security Controls in VPC

Network Access Control Lists (NACLs)
  • Core Characteristics: Subnet-level security filter
    • Scope of Control:
      • Subnet boundary protection
      • Inbound and outbound traffic filtering
      • No impact on intra-subnet communication
    • Stateless Nature:
      • Separate rules needed for:
        • Inbound traffic: Explicit allow/deny
        • Outbound traffic: Corresponding rules required
        • Return traffic: Must be explicitly allowed
  • Rule Processing:
    • Evaluation Order:
      • Lowest to highest rule number
      • First match applies
      • Processing stops at match
    • Rule Types:
      • Allow rules: Permit specified traffic
      • Deny rules: Block specified traffic
      • IP-based filtering: Explicit IP control
  • Default Configuration:
    • Default NACL:
      • Created with default VPC
      • Associated with all subnets
      • Default allow all behavior
Security Groups
  • Core Characteristics: Instance-level security filter
    • Attachment Point:
      • Elastic Network Interface (ENI)
      • Resource-level protection
      • Instance boundary security
    • Stateful Nature:
      • Single stream view of traffic
      • Return traffic automatically allowed
      • No separate outbound rules needed for responses
  • Rule Configuration:
    • Rule Types:
      • Allow Rules Only:
        • No explicit deny capability
        • Implicit deny by default
        • Everything not allowed is denied
      • Reference-Based Rules:
        • AWS resource IDs
        • Other security groups
        • Self-referential rules
  • Advanced Features:
    • Resource Recognition:
      • AWS resource identification
      • Dynamic rule updates
      • Service-based rules
    • Group References:
      • Cross-group rules
      • Group chaining
      • Simplified management
Security Implementation Best Practices
  • Defense in Depth:
    • Layer Security Controls:
      • Use both NACLs and Security Groups
      • NACL for subnet-wide policies
      • Security Groups for fine-grained control
  • Rule Management:
    • NACL Rules:
      • Number rules with gaps (10, 20, 30)
      • Allow easy rule insertion
      • Document rule purpose
    • Security Group Rules:
      • Use resource-based rules when possible
      • Minimize IP-based rules
      • Regular rule audit and cleanup

Key Differences: NACLs vs Security Groups

Feature NACLs Security Groups

Public IP Address

This is a large range of IP addresses. But a year after IPv4 was introduced, the Internet Architecture Board (IAB) and supporting community realized there were not enough addresses as more and more devices were connected to the internet. Every laptop, mobile phone, tablet, server, Internet of Things (IoT) device and so on needs at least one IP address to connect to the global network, the internet.

The IPv4 range is split using classful addressing into smaller ranges.

  • Class A: (Primary Range)
    • Range: 0.0.0.0 - 127.255.255.255
    • Capacity: Over 2.1 billion IP addresses
  • Class B: (Secondary Range)
    • Range: 128.0.0.0 - 191.255.255.255
    • Capacity: Over 1 billion IP addresses
  • Class C: (Tertiary Range)
    • Range: 192.0.0.0 - 223.255.255.255
    • Capacity: Over 2 million IP addresses
  • Additional Classes:
    • Class D and Class E ranges also exist

Note: All of these IP addresses are publicly routed to connect to these using public IP addresses.

Private IP Address

There were also ranges created for private networks. These IP addresses cannot communicate over the internet and they are private. The private IPv4 addresses were also broken down into ranges.

  • Private Class A: (Large Private Networks)
    • Range: 10.0.0.0 - 10.255.255.255
    • Capacity: One single Class A network
    • Size: 16,777,216 addresses
  • Private Class B: (Medium Private Networks)
    • Range: 172.16.0.0 - 172.31.255.255
    • Capacity: 16 Class B networks
    • Size: 1,048,576 addresses
  • Private Class C: (Small Private Networks)
    • Range: 192.168.0.0 - 192.168.255.255
    • Capacity: 256 Class C networks
    • Size: 65,536 addresses

AWS Note: The default Amazon VPC in AWS is configured using a Class B range.

Important: For private IP addresses to communicate with the internet, they have to use network address translation (NAT).
